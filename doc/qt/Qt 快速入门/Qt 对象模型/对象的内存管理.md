# 对象的内存管理

## 概述

Qt 采用了一套基于**对象树（Object Tree）和父子关系**的半自动内存管理机制。当一个 `QObject` 被销毁时，它会自动销毁其所有子对象。这意味着在大多数场景下，开发者只需管理顶层对象的生命周期，子对象由 Qt 自动回收——既避免了手动 `delete` 的繁琐，又降低了内存泄漏的风险。

理解 Qt 的内存管理规则是编写健壮 Qt 程序的基础，尤其在涉及动态创建控件、多线程、信号槽连接等场景时。

---

## 1. 核心机制：父子关系与自动销毁

### 1.1 设置父对象

每个 `QObject` 可以有一个父对象（parent）。设置父对象的方式有两种：

**方式一：构造函数传入**

```cpp
QObject *parent = new QObject;
QObject *child = new QObject(parent); // child 的 parent 被设为 parent
```

**方式二：调用 `setParent()`**

```cpp
QObject *parent = new QObject;
QObject *child = new QObject;
child->setParent(parent); // 运行时设置父对象
```

对于 Qt 控件，`QWidget` 的构造函数同样接受 `parent` 参数：

```cpp
QPushButton *btn = new QPushButton("Click Me", this); // this 是父控件
```

### 1.2 自动销毁原理

当父对象被销毁时，其析构函数会遍历子对象列表，逐一 `delete` 所有子对象。子对象析构时又会递归销毁它们各自的子对象，形成**级联销毁**。

```
Parent 析构
  ├── delete Child_1
  │     ├── delete Grandchild_1_1
  │     └── delete Grandchild_1_2
  └── delete Child_2
        └── delete Grandchild_2_1
```

核心源码逻辑（简化）：

```cpp
QObject::~QObject()
{
    // ...
    // 1. 先将自己从父对象的 children 列表中移除
    if (d->parent)
        d->parent->d_func()->children.removeAll(this);

    // 2. 递归删除所有子对象
    for (int i = 0; i < d->children.count(); ++i)
        delete d->children.at(i);
    d->children.clear();
    // ...
}
```

### 1.3 完整示例

```cpp
#include <QCoreApplication>
#include <QObject>
#include <QDebug>

class MyObject : public QObject {
    Q_OBJECT
public:
    MyObject(const QString &name, QObject *parent = nullptr)
        : QObject(parent) {
        setObjectName(name);
        qDebug() << "构造:" << objectName();
    }
    ~MyObject() override {
        qDebug() << "析构:" << objectName();
    }
};

int main(int argc, char *argv[])
{
    QCoreApplication app(argc, argv);

    MyObject *root = new MyObject("Root");
    MyObject *childA = new MyObject("ChildA", root);
    MyObject *childB = new MyObject("ChildB", root);
    new MyObject("GrandChild", childA);

    qDebug() << "--- 准备删除 root ---";
    delete root;
    // 输出：
    // 构造: "Root"
    // 构造: "ChildA"
    // 构造: "ChildB"
    // 构造: "GrandChild"
    // --- 准备删除 root ---
    // 析构: "Root"
    // 析构: "ChildA"
    // 析构: "GrandChild"
    // 析构: "ChildB"

    return 0;
}
```

> **注意**：子对象的析构顺序与添加顺序的**逆序**不完全一致——Qt 按内部 children 列表的正序依次 delete，每个 child 析构时递归深入。

## 2. 栈对象与堆对象的交互陷阱

### 2.1 栈上子对象 + 堆上父对象 → 双重释放

```cpp
void danger() {
    QObject *parent = new QObject;
    QObject child(parent); // child 在栈上，但注册为 parent 的子对象

    delete parent;
    // parent 析构时尝试 delete &child → 对栈对象调 delete → 未定义行为！
}
```

**规则**：如果子对象分配在栈上，必须确保子对象**先于**父对象销毁，或者不要设置父子关系。

### 2.2 正确的栈对象写法

```cpp
void safe() {
    QObject parent;
    QObject child(&parent);
    // child 先析构（C++ 栈对象按声明的逆序析构）
    // child 析构时从 parent 的 children 列表中移除自己
    // parent 再析构，children 列表已空，不会 double free
}
```

**关键原则**：当父子对象都在栈上时，必须保证**子对象声明在父对象之后**，以确保析构顺序正确（子先于父）。

### 2.3 错误的声明顺序

```cpp
void wrong_order() {
    QObject child;       // 先声明 → 后析构
    QObject parent;      // 后声明 → 先析构
    child.setParent(&parent);

    // parent 先析构 → 尝试 delete &child → 崩溃！
}
```

## 3. `deleteLater()` —— 延迟删除

### 3.1 为什么需要延迟删除

直接 `delete` 一个正在处理事件（如信号处理中）的对象可能导致悬空指针崩溃：

```cpp
connect(btn, &QPushButton::clicked, this, [btn]() {
    delete btn; // 危险！btn 的 clicked 信号处理还未完成，Qt 内部可能还在访问 btn
});
```

### 3.2 使用 `deleteLater()`

`deleteLater()` 不会立即删除对象，而是向事件队列投递一个 `QDeferredDeleteEvent`，等控制权返回事件循环后再安全地执行删除：

```cpp
connect(btn, &QPushButton::clicked, this, [btn]() {
    btn->deleteLater(); // 安全！等事件循环处理完当前事件后再删除
});
```

### 3.3 何时实际执行删除

```
当前事件处理中调用 deleteLater()
    │
    ▼
QDeferredDeleteEvent 被投递到事件队列
    │
    ▼
当前事件处理函数返回
    │
    ▼
事件循环取出 QDeferredDeleteEvent
    │
    ▼
执行 delete obj
```

### 3.4 典型应用场景

| 场景 | 说明 |
|---|---|
| 信号槽中删除发送者 | 避免在槽函数中 delete sender |
| 工作线程中删除对象 | 确保在正确的线程事件循环中执行删除 |
| 网络回复处理 | `QNetworkReply` 的 `finished` 信号中调用 `reply->deleteLater()` |
| 对话框自删除 | `setAttribute(Qt::WA_DeleteOnClose)` 内部使用类似机制 |

### 3.5 注意事项

- 如果没有运行事件循环（如 `QCoreApplication::exec()` 之前或之后），`deleteLater()` 投递的事件不会被处理，对象**不会被删除**。
- 多次调用 `deleteLater()` 是安全的，Qt 会确保只删除一次。

## 4. 智能指针在 Qt 中的应用

虽然 Qt 拥有父子关系机制，但在某些场景下仍需借助智能指针来管理非 QObject 对象或无父对象的 QObject。

### 4.1 `QScopedPointer`

类似 `std::unique_ptr`，离开作用域时自动释放：

```cpp
void processData() {
    QScopedPointer<QFile> file(new QFile("data.txt"));
    if (!file->open(QIODevice::ReadOnly))
        return; // file 自动释放，不泄漏
    // 处理文件...
} // file 自动释放
```

### 4.2 `QSharedPointer` / `QWeakPointer`

类似 `std::shared_ptr` / `std::weak_ptr`：

```cpp
QSharedPointer<MyData> data = QSharedPointer<MyData>::create();
QWeakPointer<MyData> weakRef = data;

// 在回调中安全地检查对象是否仍存活
if (auto locked = weakRef.lock()) {
    locked->doSomething();
}
```

### 4.3 `QPointer` —— 专为 QObject 设计的弱指针

`QPointer` 是 Qt 专属的智能指针，当所指的 `QObject` 被销毁时自动置为 `nullptr`：

```cpp
QPointer<QLabel> label = new QLabel("Hello");
label->show();

// 某处 label 被父对象销毁...
delete label->parentWidget();

if (label) {
    label->setText("World"); // 不会执行，label 已自动置 nullptr
} else {
    qDebug() << "label 已被销毁";
}
```

**典型用途**：缓存指向可能被外部（父对象、事件循环等）销毁的 QObject 指针。

### 4.4 与标准库智能指针的对比

| 特性 | `QPointer` | `QScopedPointer` | `QSharedPointer` | `std::unique_ptr` | `std::shared_ptr` |
|---|---|---|---|---|---|
| 所有权 | 无（观察者） | 独占 | 共享 | 独占 | 共享 |
| 自动置空 | ✅ 对象销毁时 | ❌ | ❌ | ❌ | ❌ |
| 引用计数 | ❌ | ❌ | ✅ | ❌ | ✅ |
| 适用类型 | 仅 QObject | 任意 | 任意 | 任意 | 任意 |
| 线程安全 | ✅ | ❌ | ✅（引用计数） | ❌ | ✅（引用计数） |

### 4.5 智能指针与父子关系的冲突

**不要**同时使用智能指针的所有权管理和 Qt 的父子关系管理同一个对象，否则会导致双重释放：

```cpp
// ❌ 错误示例
std::unique_ptr<QPushButton> btn(new QPushButton("Click", parentWidget));
// parentWidget 析构时 delete btn → unique_ptr 析构时再 delete → 崩溃！

// ✅ 正确做法一：有 parent，不用所有权智能指针
QPushButton *btn = new QPushButton("Click", parentWidget);

// ✅ 正确做法二：用智能指针，不设 parent
auto btn = std::make_unique<QPushButton>("Click");
layout->addWidget(btn.release()); // 转移所有权给布局/父控件
```

## 5. `QWidget` 与布局的内存管理

### 5.1 布局自动设置父子关系

当控件被添加到布局中，而布局又设置到了某个父控件上，Qt 会自动建立父子关系：

```cpp
QWidget *window = new QWidget;
QVBoxLayout *layout = new QVBoxLayout(window); // layout 的 parent 是 window

QPushButton *btn1 = new QPushButton("Button 1");
QPushButton *btn2 = new QPushButton("Button 2");
layout->addWidget(btn1); // btn1 的 parent 自动设置为 window
layout->addWidget(btn2); // btn2 的 parent 自动设置为 window
```

因此，`delete window` 会自动释放 `btn1` 和 `btn2`。

### 5.2 `Qt::WA_DeleteOnClose`

对于对话框等窗口，可以设置关闭时自动删除：

```cpp
QDialog *dialog = new QDialog;
dialog->setAttribute(Qt::WA_DeleteOnClose);
dialog->show();
// 用户关闭对话框时，Qt 自动 delete dialog
```

这在创建非模态对话框时非常有用，避免了手动追踪对话框的生命周期。

### 5.3 `QWidget::removeWidget()` / `takeAt()` 不会删除

从布局中移除控件**不会**删除该控件，只是解除布局关系。控件仍然是父控件的子对象：

```cpp
layout->removeWidget(btn);
// btn 仍然存在，仍然是 window 的子对象
// 如要彻底删除：
btn->setParent(nullptr); // 先解除父子关系
delete btn;              // 再手动删除
```

## 6. 多线程环境下的内存管理

### 6.1 跨线程删除对象

`QObject` 不能在非创建它的线程中直接 `delete`。应使用 `deleteLater()` 或通过信号槽在正确线程中执行删除：

```cpp
// Worker 在子线程中运行
class Worker : public QObject {
    Q_OBJECT
public slots:
    void doWork() {
        // ... 执行耗时操作
        emit finished();
    }
signals:
    void finished();
};

// 主线程中
QThread *thread = new QThread;
Worker *worker = new Worker;
worker->moveToThread(thread);

connect(thread, &QThread::started, worker, &Worker::doWork);
connect(worker, &Worker::finished, worker, &QObject::deleteLater); // 在子线程事件循环中安全删除
connect(worker, &Worker::finished, thread, &QThread::quit);
connect(thread, &QThread::finished, thread, &QObject::deleteLater); // 线程结束后安全删除

thread->start();
```

### 6.2 `moveToThread()` 与父子关系约束

- **有父对象的 QObject 不能被 moveToThread**。Qt 会输出警告并忽略操作。
- 将父对象 `moveToThread` 时，所有子对象一起迁移。

```cpp
QObject *parent = new QObject;
QObject *child = new QObject(parent);

QThread *thread = new QThread;
parent->moveToThread(thread); // child 也跟着移到 thread
// child->moveToThread(thread); // 单独移动 child 会失败，因为它有 parent
```

## 7. 常见内存问题与排查

### 7.1 典型内存泄漏场景

```cpp
// ❶ 忘了设 parent，也忘了 delete
void createWidgets() {
    QPushButton *btn = new QPushButton("Leak"); // 无 parent，无人管理 → 泄漏
}

// ❷ 容器内的裸指针
QList<MyObject*> list;
list.append(new MyObject);
list.clear(); // 只清空指针列表，不 delete 对象 → 泄漏
// 修复：qDeleteAll(list); list.clear();

// ❸ 移除子控件但忘记 delete
layout->removeWidget(widget);
widget->hide();
// widget 还在，只是不可见 → 如果不再需要应 delete
```

### 7.2 典型 Use-After-Free / 双重释放

```cpp
// ❶ 保存了已被父对象删除的子对象指针
QPushButton *btn = new QPushButton(parentWidget);
QPushButton *cached = btn;  // 缓存了裸指针
delete parentWidget;         // btn 已被级联删除
cached->setText("Crash!");   // Use-After-Free!
// 修复：使用 QPointer<QPushButton> cached = btn;

// ❷ 智能指针 + 父子关系同时管理
auto btn = std::make_unique<QPushButton>("Btn", parent);
// parent 析构 delete btn.get() → unique_ptr 析构再 delete → 双重释放
```

### 7.3 内存排查工具

| 工具 | 平台 | 用途 |
|---|---|---|
| Valgrind (memcheck) | Linux/macOS | 检测泄漏、越界、use-after-free |
| AddressSanitizer (ASan) | 全平台 | 编译时插桩检测内存错误 |
| Visual Studio 诊断工具 | Windows | CRT 内存泄漏检测 |
| `QObject::dumpObjectTree()` | 全平台 | 打印当前对象树，排查父子关系 |
| `QObject::dumpObjectInfo()` | 全平台 | 打印对象的信号槽连接信息 |

**使用 `dumpObjectTree()` 排查示例：**

```cpp
QWidget *window = new QWidget;
window->setObjectName("MainWindow");
QPushButton *btn = new QPushButton("Test", window);
btn->setObjectName("TestButton");

window->dumpObjectTree();
// 输出：
// QWidget::MainWindow
//   QPushButton::TestButton
```

## 8. 最佳实践总结

### 8.1 Golden Rules

1. **堆上分配的 QObject 尽量设置 parent**——让 Qt 的对象树自动管理生命周期。
2. **栈上的 QObject 不要设 parent**（除非严格保证析构顺序：子先于父）。
3. **在信号槽中删除对象用 `deleteLater()`**——永远不要在槽函数中直接 `delete sender()`。
4. **缓存 QObject 指针用 `QPointer`**——避免悬空指针访问。
5. **不要同时用智能指针所有权和 parent 管理同一个对象**。
6. **`qDeleteAll()` 清理指针容器**——`QList<T*>` 析构不会 delete 元素。
7. **对话框用 `WA_DeleteOnClose`**——避免非模态对话框泄漏。

### 8.2 决策流程图

```
新建 QObject 子类实例
    │
    ├─ 有明确的父对象？
    │     ├─ 是 → new Obj(parent)，由 parent 管理，不需额外操作
    │     └─ 否 → 继续判断
    │
    ├─ 生命周期局限于函数/作用域？
    │     ├─ 是 → 栈上分配 或 QScopedPointer/unique_ptr
    │     └─ 否 → 继续判断
    │
    ├─ 需要共享所有权？
    │     ├─ 是 → QSharedPointer / shared_ptr
    │     └─ 否 → 继续判断
    │
    └─ 稍后会被设 parent 或转移所有权？
          ├─ 是 → new Obj()，后续 setParent() 或交给容器管理
          └─ 否 → new Obj()，在合适时机手动 delete 或 deleteLater()
```

### 8.3 速查表

| 场景 | 推荐做法 |
|---|---|
| UI 控件 | `new Widget(parent)` |
| 非模态对话框 | `new QDialog` + `setAttribute(Qt::WA_DeleteOnClose)` |
| 模态对话框 | 栈上分配 `QDialog dlg(this); dlg.exec();` |
| 网络回复 | `reply->deleteLater()` 在 finished 信号中 |
| 工作线程 | `worker->deleteLater()` 在 finished 信号中 |
| 临时 QObject | `QScopedPointer` 或栈上分配 |
| 缓存的指针 | `QPointer<T>` 观察，不拥有 |
| 指针列表清理 | `qDeleteAll(list); list.clear();` |
