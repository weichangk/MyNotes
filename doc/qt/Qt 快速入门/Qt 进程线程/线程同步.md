# 线程同步

## 概述

当多个线程同时访问共享资源（变量、数据结构、文件等）时，如果没有适当的同步机制，就会产生**竞态条件（Race Condition）**——程序的行为取决于线程执行的随机时序，导致数据损坏、崩溃或逻辑错误。

Qt 提供了一套完整的线程同步原语：

| 工具 | 用途 |
|---|---|
| `QMutex` | 互斥锁，保护临界区 |
| `QMutexLocker` | RAII 风格的自动锁管理 |
| `QRecursiveMutex` | 可重入互斥锁 |
| `QReadWriteLock` | 读写锁，允许多读单写 |
| `QWaitCondition` | 条件变量，线程间等待/通知 |
| `QSemaphore` | 信号量，控制并发资源数量 |
| `QAtomicInt` / `QAtomicPointer` | 原子操作，无锁编程 |

---

## 1. 竞态条件——为什么需要同步

### 1.1 经典问题：计数器

```cpp
// ❌ 未同步的共享计数器
class Counter {
public:
    int value() const { return m_count; }
    void increment() { m_count++; } // 非原子操作！

private:
    int m_count = 0;
};
```

`m_count++` 在 CPU 层面实际上是三步操作：

```
1. 读取 m_count 到寄存器   (LOAD)
2. 寄存器 +1               (ADD)
3. 写回 m_count            (STORE)
```

当两个线程同时执行 `increment()`：

```
线程 A                  线程 B
────────                ────────
LOAD m_count (=0)
                        LOAD m_count (=0)
ADD → 1
                        ADD → 1
STORE m_count (=1)
                        STORE m_count (=1)  ← 丢失了一次递增！
```

预期 `m_count = 2`，实际为 `1`。这就是竞态条件。

### 1.2 更复杂的竞态

```cpp
// ❌ 非线程安全的容器操作
QList<int> sharedList;

// 线程 A
if (!sharedList.isEmpty()) {        // 检查
    int val = sharedList.takeFirst(); // 使用 ← 两步之间线程 B 可能已清空列表！
}

// 线程 B
sharedList.clear();
```

> **核心原则**：对共享数据的 **"检查 + 操作"** 必须作为一个原子步骤（临界区）执行。

## 2. QMutex —— 互斥锁

### 2.1 基本概念

`QMutex`（互斥量）是最基础的同步原语。同一时刻只允许一个线程持有锁，其他试图加锁的线程会阻塞等待。

```
线程 A           mutex           线程 B
────────         ─────           ────────
lock() ──────► [已锁定]
执行临界区                       lock() → 阻塞等待...
unlock() ────► [已解锁] ──────► 获得锁
                                 执行临界区
                                 unlock()
```

### 2.2 基本用法

```cpp
#include <QMutex>

class ThreadSafeCounter
{
public:
    void increment() {
        m_mutex.lock();
        m_count++;
        m_mutex.unlock();
    }

    void decrement() {
        m_mutex.lock();
        m_count--;
        m_mutex.unlock();
    }

    int value() const {
        m_mutex.lock();
        int val = m_count;
        m_mutex.unlock();
        return val;
    }

private:
    mutable QMutex m_mutex;
    int m_count = 0;
};
```

> **注意**：`value()` 是 `const` 方法，但需要加锁，因此 `m_mutex` 声明为 `mutable`。

### 2.3 tryLock() —— 非阻塞尝试

```cpp
if (m_mutex.tryLock()) {
    // 成功获得锁
    doWork();
    m_mutex.unlock();
} else {
    // 锁被其他线程持有，不等待，走其他逻辑
    qDebug() << "资源忙，稍后重试";
}

// 带超时的 tryLock
if (m_mutex.tryLock(1000)) { // 最多等 1 秒
    doWork();
    m_mutex.unlock();
} else {
    qWarning() << "获取锁超时";
}
```

### 2.4 死锁

```cpp
// ❌ 经典死锁：两个线程以相反顺序获取两把锁
QMutex mutexA, mutexB;

// 线程 1                     // 线程 2
mutexA.lock();                mutexB.lock();
mutexB.lock(); // 等待 B      mutexA.lock(); // 等待 A
// 永远等待...                 // 永远等待...
```

**避免死锁的方法**：

1. **固定加锁顺序**——所有线程按相同顺序获取多把锁
2. **使用 tryLock()**——获取不到就释放已持有的锁并重试
3. **减少锁粒度**——尽量不要同时持有多把锁
4. **使用 `std::scoped_lock`（C++17）**——一次性锁定多个互斥量

```cpp
// ✅ 固定顺序避免死锁
void transfer(Account &from, Account &to, int amount) {
    // 始终先锁地址较小的账户
    QMutex *first = &from.mutex, *second = &to.mutex;
    if (first > second) std::swap(first, second);

    first->lock();
    second->lock();
    from.balance -= amount;
    to.balance += amount;
    second->unlock();
    first->unlock();
}
```

## 3. QMutexLocker —— RAII 自动锁管理

### 3.1 问题：手动 lock/unlock 容易出错

```cpp
// ❌ 如果中间抛异常或 return，锁不会释放
void process() {
    m_mutex.lock();

    if (errorCondition) {
        return; // 忘记 unlock → 死锁！
    }

    if (anotherError) {
        throw std::runtime_error("oops"); // 异常 → 锁泄漏！
    }

    m_mutex.unlock();
}
```

### 3.2 使用 QMutexLocker

`QMutexLocker` 在构造时加锁，析构时自动解锁——利用 C++ 的 RAII 机制确保锁一定会被释放：

```cpp
#include <QMutexLocker>

void process() {
    QMutexLocker locker(&m_mutex); // 构造时自动 lock

    if (errorCondition) {
        return; // locker 析构 → 自动 unlock ✅
    }

    if (anotherError) {
        throw std::runtime_error("oops"); // 栈展开 → locker 析构 → 自动 unlock ✅
    }
} // 正常退出 → locker 析构 → 自动 unlock ✅
```

### 3.3 完整示例：线程安全的队列

```cpp
#include <QMutex>
#include <QMutexLocker>
#include <QQueue>

template<typename T>
class ThreadSafeQueue
{
public:
    void enqueue(const T &value) {
        QMutexLocker locker(&m_mutex);
        m_queue.enqueue(value);
    }

    bool tryDequeue(T &result) {
        QMutexLocker locker(&m_mutex);
        if (m_queue.isEmpty())
            return false;
        result = m_queue.dequeue();
        return true;
    }

    int size() const {
        QMutexLocker locker(&m_mutex);
        return m_queue.size();
    }

    bool isEmpty() const {
        QMutexLocker locker(&m_mutex);
        return m_queue.isEmpty();
    }

private:
    mutable QMutex m_mutex;
    QQueue<T> m_queue;
};
```

### 3.4 QMutexLocker 的额外控制

```cpp
QMutexLocker locker(&m_mutex);

// 手动临时释放锁
locker.unlock();
doSomethingWithoutLock(); // 不需要锁的操作
locker.relock(); // 重新加锁

// 作用域结束自动释放
```

## 4. QRecursiveMutex —— 可重入互斥锁

### 4.1 问题：普通 QMutex 不支持递归加锁

```cpp
QMutex mutex;

void funcA() {
    mutex.lock();
    funcB(); // funcB 也会加同一把锁
    mutex.unlock();
}

void funcB() {
    mutex.lock(); // ❌ 死锁！当前线程已持有此锁
    // ...
    mutex.unlock();
}
```

### 4.2 使用 QRecursiveMutex

```cpp
QRecursiveMutex mutex;

void funcA() {
    mutex.lock();    // 第 1 次加锁，计数 = 1
    funcB();
    mutex.unlock();  // 计数 = 0，真正释放
}

void funcB() {
    mutex.lock();    // ✅ 同一线程再次加锁，计数 = 2
    // ...
    mutex.unlock();  // 计数 = 1，未真正释放
}
```

> **原理**：`QRecursiveMutex` 内部维护一个计数器和持有者线程 ID。同一线程可以多次 `lock()`，每次 `unlock()` 减少计数，计数归零时才真正释放锁。

### 4.3 与 QMutexLocker 配合

```cpp
QRecursiveMutex mutex;

void funcA() {
    QMutexLocker locker(&mutex);
    funcB(); // 安全递归
}

void funcB() {
    QMutexLocker locker(&mutex); // 同一线程再次加锁，OK
    // ...
}
```

### 4.4 何时使用

| 场景 | 选择 |
|---|---|
| 同一锁不会被同一线程重入 | `QMutex`（性能更好） |
| 函数相互调用，可能递归获取同一锁 | `QRecursiveMutex` |
| 回调函数中可能再次进入加锁代码 | `QRecursiveMutex` |

> **建议**：优先使用 `QMutex`。如果确实需要递归锁，先考虑是否可以重构代码来避免递归加锁，因为递归锁有额外的性能开销且可能掩盖设计问题。

## 5. QReadWriteLock —— 读写锁

### 5.1 动机

在"读多写少"的场景下，`QMutex` 要求所有访问互斥——即使多个线程只是读取数据，也必须排队，浪费大量性能。

`QReadWriteLock` 区分了读操作和写操作：

- **多个读线程**可以同时持有读锁（共享）
- **写锁**是排他的——写入时不允许任何读或写
- 写线程等待时，新的读请求也会被阻塞（避免写饥饿）

```
读锁（共享）        写锁（排他）
─────────           ──────────
线程 A: 读 ✅       线程 A: 写 ✅
线程 B: 读 ✅       线程 B: 读 ❌（等待）
线程 C: 读 ✅       线程 C: 写 ❌（等待）
```

### 5.2 基本用法

```cpp
#include <QReadWriteLock>

class ConfigManager
{
public:
    QString getValue(const QString &key) const {
        m_lock.lockForRead(); // 获取读锁
        QString val = m_config.value(key);
        m_lock.unlock();
        return val;
    }

    void setValue(const QString &key, const QString &value) {
        m_lock.lockForWrite(); // 获取写锁（排他）
        m_config[key] = value;
        m_lock.unlock();
    }

    QMap<QString, QString> allValues() const {
        m_lock.lockForRead();
        auto copy = m_config; // 在读锁保护下拷贝
        m_lock.unlock();
        return copy;
    }

private:
    mutable QReadWriteLock m_lock;
    QMap<QString, QString> m_config;
};
```

### 5.3 RAII 辅助类

```cpp
#include <QReadLocker>
#include <QWriteLocker>

class ConfigManager
{
public:
    QString getValue(const QString &key) const {
        QReadLocker locker(&m_lock);
        return m_config.value(key);
    }

    void setValue(const QString &key, const QString &value) {
        QWriteLocker locker(&m_lock);
        m_config[key] = value;
    }

private:
    mutable QReadWriteLock m_lock;
    QMap<QString, QString> m_config;
};
```

### 5.4 递归模式

```cpp
// 默认：非递归
QReadWriteLock lock;

// 递归模式
QReadWriteLock recursiveLock(QReadWriteLock::Recursive);
```

### 5.5 性能对比

```
场景：8 线程，90% 读 + 10% 写

QMutex         ████████████████████████████  慢（所有操作互斥）
QReadWriteLock ████████                       快（读操作可并发）
```

> **适用场景**：读操作远多于写操作（>10:1）时，`QReadWriteLock` 有显著优势。如果读写比例接近，`QMutex` 反而更快（读写锁有更大的管理开销）。

## 6. QWaitCondition —— 条件变量

### 6.1 概念

条件变量允许线程在某个条件不满足时**挂起等待**，其他线程在条件满足时**唤醒**等待的线程。它总是与一个互斥锁配合使用。

```
生产者线程                              消费者线程
──────────                              ──────────
lock(mutex)                             lock(mutex)
生产数据                                while (队列空)
signal(condition) ────────────►           condition.wait(mutex) ← 释放锁并挂起
unlock(mutex)                           // 被唤醒后 mutex 已重新加锁
                                        取出数据
                                        unlock(mutex)
```

### 6.2 经典生产者-消费者模型

```cpp
#include <QMutex>
#include <QWaitCondition>
#include <QQueue>
#include <QThread>
#include <QDebug>

// 共享数据
const int BUFFER_SIZE = 10;
QQueue<int> buffer;
QMutex mutex;
QWaitCondition bufferNotEmpty;
QWaitCondition bufferNotFull;

// 生产者
class Producer : public QThread
{
protected:
    void run() override {
        for (int i = 0; i < 100; ++i) {
            mutex.lock();

            // 缓冲区满时等待
            while (buffer.size() >= BUFFER_SIZE) {
                bufferNotFull.wait(&mutex); // 释放 mutex 并挂起
            }

            buffer.enqueue(i);
            qDebug() << "生产:" << i << " 队列大小:" << buffer.size();

            bufferNotEmpty.wakeOne(); // 唤醒一个消费者
            mutex.unlock();

            QThread::msleep(50); // 模拟生产耗时
        }
    }
};

// 消费者
class Consumer : public QThread
{
protected:
    void run() override {
        while (true) {
            mutex.lock();

            // 缓冲区空时等待
            while (buffer.isEmpty()) {
                bufferNotEmpty.wait(&mutex);
            }

            int value = buffer.dequeue();
            qDebug() << "消费:" << value << " 队列大小:" << buffer.size();

            bufferNotFull.wakeOne(); // 唤醒一个生产者
            mutex.unlock();

            QThread::msleep(100); // 模拟消费耗时

            if (value == 99) break; // 结束标志
        }
    }
};
```

### 6.3 为什么用 while 而不是 if

```cpp
// ❌ 使用 if 可能导致虚假唤醒问题
if (buffer.isEmpty()) {
    bufferNotEmpty.wait(&mutex);
}
// 被唤醒后直接执行 → 但此时 buffer 可能又被其他消费者取空了！

// ✅ 使用 while 循环检查条件
while (buffer.isEmpty()) {
    bufferNotEmpty.wait(&mutex);
}
// 被唤醒后重新检查条件 → 即使虚假唤醒也安全
```

> **虚假唤醒（Spurious Wakeup）**：操作系统可能在没有对应 `wake` 的情况下唤醒等待线程。这是 POSIX 标准允许的行为，所有条件变量都需要用 `while` 循环防护。

### 6.4 wakeOne() vs wakeAll()

| 方法 | 行为 | 适用场景 |
|---|---|---|
| `wakeOne()` | 唤醒一个等待线程（随机选择） | 生产者-消费者，一个任务只需一个处理者 |
| `wakeAll()` | 唤醒所有等待线程 | 状态变更通知（如配置更新），所有线程都需要响应 |

```cpp
// 配置更新，通知所有工作线程
void ConfigManager::updateConfig(const Config &newConfig) {
    QMutexLocker locker(&m_mutex);
    m_config = newConfig;
    m_configChanged.wakeAll(); // 所有工作线程都需要重新加载配置
}
```

### 6.5 带超时的等待

```cpp
mutex.lock();
while (buffer.isEmpty()) {
    if (!bufferNotEmpty.wait(&mutex, 5000)) {
        // 超过 5 秒仍未被唤醒
        qWarning() << "等待超时";
        mutex.unlock();
        return;
    }
}
// 取出数据...
mutex.unlock();
```

### 6.6 配合 QReadWriteLock 使用

```cpp
QReadWriteLock rwLock;
QWaitCondition condition;

// 等待端
rwLock.lockForWrite();
while (!conditionMet) {
    condition.wait(&rwLock); // 也支持 QReadWriteLock
}
rwLock.unlock();
```

## 7. QSemaphore —— 信号量

### 7.1 概念

信号量本质是一个**计数器**——允许最多 N 个线程同时访问某个资源。`QMutex` 可以看作 N=1 的特殊信号量。

```
信号量(3)：允许最多 3 个线程同时进入

线程 A: acquire() → 计数 2 → 进入 ✅
线程 B: acquire() → 计数 1 → 进入 ✅
线程 C: acquire() → 计数 0 → 进入 ✅
线程 D: acquire() → 计数 0 → 阻塞等待 ❌
...
线程 A: release() → 计数 1 → 线程 D 被唤醒 → 进入 ✅
```

### 7.2 基本 API

```cpp
#include <QSemaphore>

QSemaphore sem(5); // 初始计数为 5

sem.acquire();     // 计数 -1，若为 0 则阻塞
sem.acquire(3);    // 计数 -3，若不足则阻塞

sem.release();     // 计数 +1
sem.release(2);    // 计数 +2

sem.available();   // 当前可用计数

sem.tryAcquire();        // 非阻塞尝试，成功返回 true
sem.tryAcquire(1, 1000); // 带超时（1 秒）
```

### 7.3 实战：连接池

```cpp
class ConnectionPool
{
public:
    explicit ConnectionPool(int maxConnections)
        : m_semaphore(maxConnections)
    {
        for (int i = 0; i < maxConnections; ++i) {
            m_connections.enqueue(createConnection(i));
        }
    }

    // 获取连接（可能阻塞等待可用连接）
    Connection *acquire() {
        m_semaphore.acquire(); // 有可用名额才继续

        QMutexLocker locker(&m_mutex);
        return m_connections.dequeue();
    }

    // 归还连接
    void release(Connection *conn) {
        {
            QMutexLocker locker(&m_mutex);
            m_connections.enqueue(conn);
        }
        m_semaphore.release(); // 释放一个名额
    }

    // 非阻塞获取
    Connection *tryAcquire(int timeoutMs = 0) {
        if (!m_semaphore.tryAcquire(1, timeoutMs))
            return nullptr;

        QMutexLocker locker(&m_mutex);
        return m_connections.dequeue();
    }

private:
    QSemaphore m_semaphore;
    QMutex m_mutex;
    QQueue<Connection*> m_connections;
};
```

### 7.4 实战：环形缓冲区（生产者-消费者）

使用两个信号量实现高效的生产者-消费者模式，比条件变量更简洁：

```cpp
const int BUFFER_SIZE = 1024;
char buffer[BUFFER_SIZE];

QSemaphore freeSlots(BUFFER_SIZE); // 空闲槽位数
QSemaphore usedSlots(0);           // 已用槽位数

// 生产者
class Producer : public QThread
{
protected:
    void run() override {
        for (int i = 0; i < 10000; ++i) {
            freeSlots.acquire();             // 等待空闲槽位
            buffer[i % BUFFER_SIZE] = "ABCDEFGHIJ"[i % 10];
            usedSlots.release();             // 通知有新数据
        }
    }
};

// 消费者
class Consumer : public QThread
{
protected:
    void run() override {
        for (int i = 0; i < 10000; ++i) {
            usedSlots.acquire();             // 等待可用数据
            char c = buffer[i % BUFFER_SIZE];
            freeSlots.release();             // 通知槽位已释放
            qDebug() << "消费:" << c;
        }
    }
};
```

> **关键点**：两个信号量的计数之和始终等于 `BUFFER_SIZE`——`freeSlots + usedSlots = BUFFER_SIZE`，天然保证了不会溢出也不会读空。

## 8. 原子操作 —— 无锁编程

### 8.1 概念

原子操作是 CPU 硬件级别保证的不可分割操作，不需要互斥锁，性能远高于锁。适用于简单的整数或指针操作。

### 8.2 QAtomicInt

```cpp
#include <QAtomicInt>

class AtomicCounter
{
public:
    void increment() {
        m_count.fetchAndAddOrdered(1); // 原子 +1
    }

    void decrement() {
        m_count.fetchAndSubOrdered(1); // 原子 -1
    }

    int value() const {
        return m_count.loadAcquire(); // 原子读
    }

private:
    QAtomicInt m_count{0};
};
```

### 8.3 常用原子操作

```cpp
QAtomicInt n(0);

// 基本读写
n.storeRelease(42);    // 原子写入（带 release 语义）
int val = n.loadAcquire(); // 原子读取（带 acquire 语义）

// 加减
n.fetchAndAddOrdered(5);  // 原子 += 5，返回旧值
n.fetchAndSubOrdered(3);  // 原子 -= 3，返回旧值

// CAS（Compare And Swap）—— 无锁编程核心
int expected = 10;
bool success = n.testAndSetOrdered(expected, 20);
// 如果 n == expected，则 n = 20 并返回 true
// 否则不修改，返回 false

// 引用计数
n.ref();   // 原子 +1，返回 true 如果结果非零
n.deref(); // 原子 -1，返回 true 如果结果非零
```

### 8.4 内存序（Memory Ordering）

| 后缀 | 含义 | 性能 |
|---|---|---|
| `Relaxed` | 无内存屏障，只保证操作本身原子性 | 最快 |
| `Acquire` | 读取后的操作不会重排到此操作之前 | 中等 |
| `Release` | 写入前的操作不会重排到此操作之后 | 中等 |
| `Ordered` | 完全的内存屏障（acquire + release） | 最慢（但最安全） |

```cpp
// 典型的"发布-获取"模式
QAtomicInt ready(0);
int data = 0;

// 线程 A（生产者）
data = 42;                   // 普通写入
ready.storeRelease(1);       // Release：保证 data=42 在 ready=1 之前完成

// 线程 B（消费者）
while (ready.loadAcquire() == 0) {} // Acquire：保证之后读到正确的 data
qDebug() << data;            // 保证看到 42
```

### 8.5 QAtomicPointer

```cpp
#include <QAtomicPointer>

QAtomicPointer<MyData> sharedPtr(nullptr);

// 原子替换
MyData *newData = new MyData;
MyData *oldData = sharedPtr.fetchAndStoreOrdered(newData);
delete oldData; // 安全释放旧数据

// CAS
MyData *expected = nullptr;
if (sharedPtr.testAndSetOrdered(expected, newData)) {
    // 设置成功（原来是 nullptr）
} else {
    // 其他线程已经设置了值
    delete newData;
}
```

### 8.6 何时使用原子操作 vs 互斥锁

| 场景 | 推荐 |
|---|---|
| 简单计数器（递增/递减） | `QAtomicInt` |
| 布尔标志 | `QAtomicInt` (0/1) |
| 指针的原子交换 | `QAtomicPointer` |
| 保护复杂数据结构 | `QMutex` / `QReadWriteLock` |
| 需要条件等待 | `QWaitCondition` |

### 9.2 线程安全的缓存

```cpp
class ImageCache
{
public:
    QImage get(const QString &key) const {
        QReadLocker locker(&m_lock);
        return m_cache.value(key);
    }

    void put(const QString &key, const QImage &image) {
        QWriteLocker locker(&m_lock);

        // 检查缓存大小限制
        if (m_cache.size() >= MAX_CACHE_SIZE) {
            // 移除最早插入的
            m_cache.erase(m_cache.begin());
        }

        m_cache[key] = image;
    }

    bool contains(const QString &key) const {
        QReadLocker locker(&m_lock);
        return m_cache.contains(key);
    }

    void clear() {
        QWriteLocker locker(&m_lock);
        m_cache.clear();
    }

private:
    static constexpr int MAX_CACHE_SIZE = 100;
    mutable QReadWriteLock m_lock;
    QMap<QString, QImage> m_cache;
};
```

### 9.3 多线程任务队列（完整实现）

```cpp
#include <QThread>
#include <QMutex>
#include <QMutexLocker>
#include <QWaitCondition>
#include <QQueue>
#include <QDebug>
#include <functional>

class TaskQueue : public QObject
{
    Q_OBJECT
public:
    using Task = std::function<void()>;

    void submit(Task task) {
        QMutexLocker locker(&m_mutex);
        m_tasks.enqueue(std::move(task));
        m_condition.wakeOne();
    }

    void shutdown() {
        QMutexLocker locker(&m_mutex);
        m_shutdown = true;
        m_condition.wakeAll();
    }

private:
    friend class WorkerThread;

    // 阻塞获取任务（供工作线程调用）
    bool waitForTask(Task &task) {
        QMutexLocker locker(&m_mutex);
        while (m_tasks.isEmpty() && !m_shutdown) {
            m_condition.wait(&m_mutex);
        }
        if (m_shutdown && m_tasks.isEmpty())
            return false;
        task = m_tasks.dequeue();
        return true;
    }

    QMutex m_mutex;
    QWaitCondition m_condition;
    QQueue<Task> m_tasks;
    bool m_shutdown = false;
};

class WorkerThread : public QThread
{
    Q_OBJECT
public:
    explicit WorkerThread(TaskQueue *queue, QObject *parent = nullptr)
        : QThread(parent), m_queue(queue) {}

protected:
    void run() override {
        TaskQueue::Task task;
        while (m_queue->waitForTask(task)) {
            try {
                task();
            } catch (const std::exception &e) {
                qWarning() << "任务异常:" << e.what();
            }
            emit taskCompleted();
        }
        qDebug() << "工作线程退出:" << QThread::currentThread();
    }

signals:
    void taskCompleted();

private:
    TaskQueue *m_queue;
};

// 使用示例
class ThreadPool : public QObject
{
    Q_OBJECT
public:
    explicit ThreadPool(int threadCount, QObject *parent = nullptr)
        : QObject(parent)
    {
        for (int i = 0; i < threadCount; ++i) {
            auto *worker = new WorkerThread(&m_queue, this);
            m_workers.append(worker);
            worker->start();
        }
    }

    ~ThreadPool() {
        m_queue.shutdown();
        for (auto *worker : m_workers)
            worker->wait();
    }

    void submit(TaskQueue::Task task) {
        m_queue.submit(std::move(task));
    }

private:
    TaskQueue m_queue;
    QList<WorkerThread*> m_workers;
};

// 调用
ThreadPool pool(4);
pool.submit([]() {
    qDebug() << "任务 1 执行于" << QThread::currentThread();
});
pool.submit([]() {
    qDebug() << "任务 2 执行于" << QThread::currentThread();
});
```

## 10. 同步原语选择决策

```
需要同步共享数据？
  │
  ├─ 只是简单的整数/布尔/指针？
  │     └─ YES → QAtomicInt / QAtomicPointer
  │
  ├─ 读多写少（>10:1）？
  │     └─ YES → QReadWriteLock + QReadLocker/QWriteLocker
  │
  ├─ 需要等待某个条件成立？
  │     └─ YES → QMutex + QWaitCondition
  │
  ├─ 需要限制并发数量？
  │     └─ YES → QSemaphore
  │
  ├─ 同一线程可能递归加锁？
  │     └─ YES → QRecursiveMutex
  │
  └─ 通用互斥保护？
        └─ QMutex + QMutexLocker（默认首选）
```

## 11. 总结

| 要点 | 说明 |
|---|---|
| **首选 QMutexLocker** | 总是使用 RAII 管理锁，避免手动 lock/unlock 导致的锁泄漏 |
| **最小化锁的范围** | 只在必要的代码段持有锁，尽快释放，减少竞争 |
| **避免死锁** | 固定加锁顺序；避免在持有锁时调用外部回调；优先使用 tryLock |
| **while 循环等待** | 使用 `QWaitCondition` 时始终用 `while` 检查条件（防虚假唤醒） |
| **读写分离** | 读多写少场景使用 `QReadWriteLock` 提升并发性能 |
| **原子操作** | 简单计数/标志使用原子操作，避免锁的开销 |
| **信号量限流** | 需要控制并发数量时使用 `QSemaphore` |
| **优先信号槽** | 能用跨线程信号槽解决的问题，优先用信号槽，比手动同步更安全 |
