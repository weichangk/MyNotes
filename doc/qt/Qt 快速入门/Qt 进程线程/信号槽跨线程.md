# 信号槽跨线程

## 概述

Qt 信号槽机制天然支持跨线程通信——当信号的发射者和接收者处于不同线程时，Qt 会自动通过**事件队列**将槽函数调用投递到接收者所在线程执行，实现线程间安全的数据传递。

这是 Qt 最强大也最容易被误解的特性之一。理解其底层原理对于编写正确的多线程 Qt 程序至关重要。

---

## 1. 连接类型（Connection Type）

`connect()` 的第五个参数指定连接类型，决定了槽函数在哪个线程、以何种方式被调用：

```cpp
connect(sender, &Sender::signal, receiver, &Receiver::slot, Qt::ConnectionType);
```

### 1.1 五种连接类型

| 类型 | 枚举值 | 行为 |
|---|---|---|
| **自动连接** | `Qt::AutoConnection`（默认） | 运行时判断：同线程用 Direct，跨线程用 Queued |
| **直接连接** | `Qt::DirectConnection` | 在**发射者所在线程**中直接调用槽函数（等同函数调用） |
| **队列连接** | `Qt::QueuedConnection` | 将调用投递到**接收者所在线程**的事件队列 |
| **阻塞队列连接** | `Qt::BlockingQueuedConnection` | 与 Queued 相同，但发射线程会**阻塞等待**槽函数执行完毕 |
| **唯一连接** | `Qt::UniqueConnection` | 修饰符，与上述类型组合，防止重复连接 |

### 1.2 图解对比

```
DirectConnection（同步）：
┌────────────┐
│  线程 A     │
│  emit sig()─┼─► slot() 立即执行（在线程 A 中）
│  下一行代码  │   ← slot() 返回后才继续
└────────────┘

QueuedConnection（异步）：
┌────────────┐     事件队列     ┌────────────┐
│  线程 A     │                 │  线程 B     │
│  emit sig()─┼──► [事件入队] ──┼─► slot()    │
│  下一行代码  │  ← 立即继续     │  在 B 的    │
│  ...        │                 │  事件循环中  │
└────────────┘                 └────────────┘

BlockingQueuedConnection（同步跨线程）：
┌────────────┐     事件队列     ┌────────────┐
│  线程 A     │                 │  线程 B     │
│  emit sig()─┼──► [事件入队] ──┼─► slot()    │
│  阻塞等待...│                 │  执行中     │
│  ...........│  ◄── 完成通知 ──┼── slot返回  │
│  下一行代码  │  ← 解除阻塞     │             │
└────────────┘                 └────────────┘
```

## 2. AutoConnection 的判断逻辑

### 2.1 运行时判断

`Qt::AutoConnection` **不是**在 `connect()` 时决定连接类型，而是在**每次信号发射时**动态判断：

```cpp
// Qt 内部伪代码（简化）
void QMetaObject::activate(QObject *sender, int signalIndex, void **argv)
{
    for (每个连接的槽) {
        if (connectionType == Qt::AutoConnection) {
            if (receiver->thread() == QThread::currentThread()) {
                // 同一线程 → 直接调用
                directCall(slot, argv);
            } else {
                // 不同线程 → 投递到接收者的事件队列
                postEvent(receiver, new QMetaCallEvent(slot, argv));
            }
        }
        // ...
    }
}
```

### 2.2 关键点

```cpp
// 场景：Worker 被 moveToThread 到子线程
QThread *thread = new QThread;
Worker *worker = new Worker;
worker->moveToThread(thread);

// 在主线程中连接（连接时 worker 还在主线程？不影响！）
connect(this, &Controller::startWork, worker, &Worker::doWork);
// ↑ AutoConnection：每次 emit 时判断
//   emit 在主线程 → worker->thread() 是子线程 → QueuedConnection

thread->start();
emit startWork(); // 此时 worker 在子线程 → Queued → 安全
```

### 2.3 AutoConnection 可能出现的意外

```cpp
// Worker 在子线程中
connect(worker, &Worker::resultReady, worker, &Worker::processResult);
// ↑ 如果 resultReady 是在子线程中 emit 的，
//   sender 线程 == receiver 线程 → DirectConnection
//   processResult 在子线程同步执行

// 如果 resultReady 是在主线程中 emit 的（比如测试代码直接调用），
//   sender 线程 ≠ receiver 线程 → QueuedConnection
//   processResult 被投递到子线程事件循环执行
```

> **经验法则**：如果你明确知道信号一定跨线程发射，显式指定 `Qt::QueuedConnection` 更清晰。

## 3. QueuedConnection 深入

### 3.1 工作原理

当使用队列连接时，Qt 在发射信号的瞬间做以下操作：

1. **拷贝所有参数**（通过元对象系统的 `QMetaType` 进行深拷贝）
2. **创建 `QMetaCallEvent`** 封装槽函数指针和参数副本
3. **投递事件**到接收者所在线程的事件队列（`QCoreApplication::postEvent`）
4. **发射线程立即返回**，不等待槽函数执行

接收者线程在其事件循环中：

5. **取出 `QMetaCallEvent`**
6. **调用槽函数**，传入参数副本
7. **释放参数副本**

### 3.2 参数必须注册到元类型系统

队列连接需要 Qt 能够在运行时拷贝和销毁参数，因此所有参数类型必须注册到 `QMetaType`：

```cpp
// Qt 内置类型（int、QString、QImage 等）自动注册，无需操作

// 自定义类型必须注册
struct SearchResult {
    QString filePath;
    int lineNumber;
    QString content;
};

// 方式一：在类型声明后注册（推荐放在头文件中）
Q_DECLARE_METATYPE(SearchResult)

// 方式二：在 main() 或使用前调用（运行时注册）
qRegisterMetaType<SearchResult>("SearchResult");
```

**未注册时的错误**：

```
QObject::connect: Cannot queue arguments of type 'SearchResult'
(Make sure 'SearchResult' is registered using qRegisterMetaType().)
```

### 3.3 哪些类型需要注册

| 类型 | 是否需要注册 |
|---|---|
| `int`、`double`、`bool` 等基本类型 | ❌ 已内置 |
| `QString`、`QByteArray`、`QVariant` | ❌ 已内置 |
| `QList<T>`、`QMap<K,V>`（T/K/V 已注册） | ❌ 已内置 |
| `QImage`、`QPixmap`、`QColor` 等 Qt 类型 | ❌ 已内置 |
| 自定义 struct/class | ✅ 需要 `Q_DECLARE_METATYPE` |
| 自定义枚举 | ✅ 需要 `Q_ENUM` 或 `Q_DECLARE_METATYPE` |
| 指针类型 `MyClass*` | ✅ 需要注册（但传递指针需注意生命周期） |
| `std::shared_ptr<T>` | ✅ 需要注册 |

### 3.4 参数的深拷贝

```cpp
// ✅ 安全：QString 被深拷贝
signals:
    void dataReady(const QString &data);
// 即使信号参数是 const 引用，队列连接也会拷贝一份副本

// ⚠️ 危险：传递原始指针
signals:
    void bufferReady(char *buffer, int size);
// 队列连接拷贝的是指针值，不是指针指向的数据！
// 如果 buffer 在槽执行前被释放 → 悬空指针

// ✅ 安全替代：使用 QByteArray
signals:
    void bufferReady(const QByteArray &buffer);
// QByteArray 隐式共享 + 深拷贝，跨线程安全
```

### 3.5 需要运行的事件循环

队列连接的槽函数只有在接收者线程的**事件循环运行时**才会被调用：

```cpp
// ❌ 槽永远不会执行
QThread *thread = new QThread;
Worker *worker = new Worker;
worker->moveToThread(thread);
connect(this, &Controller::doWork, worker, &Worker::process);

// 忘记 start() → 子线程没有事件循环 → 事件永远不被处理
emit doWork();

// ✅ 必须启动线程
thread->start(); // 启动后 QThread::run() 调用 exec() 进入事件循环
emit doWork();   // 事件被子线程事件循环取出并处理
```

## 4. DirectConnection 跨线程

### 4.1 行为

强制在发射信号的线程中直接调用槽函数，不经过事件队列：

```cpp
connect(worker, &Worker::progress, this, &MainWindow::updateProgress,
        Qt::DirectConnection);

// ⚠️ worker 在子线程中 emit progress()
// → updateProgress() 在子线程中执行（不是主线程！）
// → 如果 updateProgress() 操作了 GUI → 崩溃
```

### 4.2 什么时候用 DirectConnection 跨线程

极少。但有一个常见场景——**性能关键的回调**：

```cpp
class AudioEngine : public QObject {
    Q_OBJECT
signals:
    void audioBufferReady(const float *data, int frames);
};

class AudioMeter : public QObject {
    Q_OBJECT
public slots:
    void processBuffer(const float *data, int frames) {
        // 纯计算，不操作 GUI，需要低延迟
        float peak = 0;
        for (int i = 0; i < frames; ++i)
            peak = qMax(peak, qAbs(data[i]));
        m_peak.storeRelease(static_cast<int>(peak * 1000)); // 原子写入
    }
private:
    QAtomicInt m_peak{0};
};

// 直接连接：audioBufferReady 在音频线程中发射，
// processBuffer 也在音频线程中执行（低延迟，避免队列开销）
connect(engine, &AudioEngine::audioBufferReady,
        meter, &AudioMeter::processBuffer,
        Qt::DirectConnection);

// 主线程通过原子变量读取结果更新 UI
```

> **使用 DirectConnection 跨线程的前提**：槽函数必须是**线程安全**的（不操作 GUI、不访问未加锁的共享数据）。

## 5. BlockingQueuedConnection

### 5.1 行为

与 `QueuedConnection` 一样将调用投递到接收者线程，但发射线程会**阻塞等待**直到槽函数执行完毕。可以获取槽函数的返回值。

### 5.2 典型用途——跨线程调用主线程 GUI 操作

```cpp
// 子线程中需要弹出对话框并获取用户选择
class Worker : public QObject {
    Q_OBJECT
public slots:
    void doWork() {
        // ... 处理中 ...

        // 需要在主线程弹出确认对话框
        bool confirmed = false;
        QMetaObject::invokeMethod(m_mainWindow, [&confirmed, this]() {
            // 这个 lambda 在主线程执行
            auto result = QMessageBox::question(m_mainWindow,
                "确认", "是否继续处理？");
            confirmed = (result == QMessageBox::Yes);
        }, Qt::BlockingQueuedConnection);

        if (confirmed) {
            continueProcessing();
        }
    }
private:
    QWidget *m_mainWindow;
};
```

### 5.3 死锁风险

```cpp
// ❌ 致命死锁：发射者和接收者在同一线程
connect(obj, &MyClass::sig, obj, &MyClass::slot,
        Qt::BlockingQueuedConnection);
emit obj->sig();
// sig() 发射 → 等待 slot() 在 obj 的线程执行完
// 但 sig() 本身就在 obj 的线程中 → obj 的线程正在等待自己 → 死锁！
```

> **Qt 会在运行时检测并警告**：`Qt: Dead lock detected while activating a BlockingQueuedConnection`

```cpp
// ❌ 间接死锁
// 主线程 → BlockingQueued 调用子线程函数
// 子线程函数中 → BlockingQueued 调用主线程函数
// → 两个线程互相等待 → 死锁
```

### 5.4 安全使用原则

1. **绝不在同一线程中使用** `BlockingQueuedConnection`
2. **避免双向的** `BlockingQueuedConnection`
3. 尽量使用 `QueuedConnection` + 回调信号代替
4. 确保接收者线程的事件循环正在运行

## 6. QMetaObject::invokeMethod() —— 灵活的跨线程调用

### 6.1 基本语法

```cpp
// 在指定连接类型下调用对象的方法
QMetaObject::invokeMethod(object, "methodName", connectionType, args...);

// Qt 5.10+ 支持 Lambda
QMetaObject::invokeMethod(object, []() { /* ... */ }, connectionType);
```

### 6.2 异步投递到其他线程

```cpp
// 等效于 QueuedConnection 信号槽，但不需要 connect
QMetaObject::invokeMethod(worker, "processData",
                          Qt::QueuedConnection,
                          Q_ARG(QString, "input.dat"));

// Lambda 版本（推荐）
QMetaObject::invokeMethod(worker, [data]() {
    worker->processData(data);
}, Qt::QueuedConnection);
```

### 6.3 同步跨线程调用并获取返回值

```cpp
QString result;
QMetaObject::invokeMethod(worker, "getStatus",
                          Qt::BlockingQueuedConnection,
                          Q_RETURN_ARG(QString, result));

qDebug() << "Worker 状态:" << result;
```

### 6.4 在主线程执行 GUI 操作（最常见用法）

```cpp
// 子线程中安全地更新 UI
void Worker::onProgress(int percent) {
    QMetaObject::invokeMethod(m_progressBar, [this, percent]() {
        m_progressBar->setValue(percent); // 在主线程执行
    }, Qt::QueuedConnection);
}

// 或者使用更简洁的写法（Qt 5.10+）
void Worker::showMessage(const QString &msg) {
    QMetaObject::invokeMethod(qApp, [msg]() {
        QMessageBox::information(nullptr, "通知", msg);
    });
    // 不指定连接类型时默认 AutoConnection
}
```

### 6.5 invokeMethod vs 信号槽

| 特性 | `connect()` + 信号槽 | `QMetaObject::invokeMethod()` |
|---|---|---|
| 编译期类型检查 | ✅（新语法） | ❌（字符串版本）/ ✅（Lambda 版本） |
| 一对多 | ✅ 一个信号多个槽 | ❌ 一次调用一个方法 |
| 需要预先连接 | ✅ | ❌ 即时调用 |
| 适用场景 | 常规信号通知 | 动态调用、一次性跨线程调用 |

## 7. 跨线程信号槽的参数传递

### 7.1 值类型 vs 引用类型

```cpp
signals:
    void dataReady(QString data);          // 值传递：拷贝
    void dataReady(const QString &data);   // const 引用：队列连接时仍然拷贝
    void dataReady(QString &data);         // 非 const 引用：❌ 队列连接不支持
```

> **规则**：队列连接中，所有参数都按值拷贝。信号参数声明为 `const &` 只是优化同线程直接连接时的拷贝，跨线程时 Qt 仍然执行深拷贝。

### 7.2 传递大型数据

```cpp
// ❌ 低效：每次发射都深拷贝大 QImage
signals:
    void frameReady(QImage frame); // 1920x1080 RGBA ≈ 8MB 拷贝

// ✅ 利用隐式共享（Copy-on-Write）
// QImage 内部使用引用计数。队列连接拷贝时只增加引用计数（O(1)），
// 只有被修改时才真正拷贝数据。
// → 实际上传递 QImage 的开销很小（前提是接收端不修改它）

// ✅ 传递 QSharedPointer（零拷贝）
signals:
    void frameReady(QSharedPointer<QImage> frame);
// 注册类型：
qRegisterMetaType<QSharedPointer<QImage>>("QSharedPointer<QImage>");

// ✅ 对于自定义类型，使用移动语义
signals:
    void dataReady(QByteArray data);
// Qt 5.6+ 的队列连接内部会使用移动语义优化
```

### 7.3 传递指针的注意事项

```cpp
// ⚠️ 传递裸指针——必须确保对象在槽执行时仍然存活
signals:
    void taskDone(Result *result);

// 发射线程
Result *r = new Result;
emit taskDone(r);
delete r; // ❌ 槽可能还没执行！

// ✅ 方案一：让接收端负责 delete
emit taskDone(r); // 接收端 delete

// ✅ 方案二：传值或智能指针
signals:
    void taskDone(QSharedPointer<Result> result);
```

### 7.4 自定义类型注册完整示例

```cpp
// types.h
#pragma once
#include <QMetaType>

struct TaskResult {
```

## 8. 实战示例

### 8.1 完整的异步文件处理器

```cpp
// fileprocessor.h
#pragma once
#include <QObject>
#include <QThread>

class FileProcessor : public QObject
{
    Q_OBJECT
public:
    explicit FileProcessor(QObject *parent = nullptr);
    ~FileProcessor();

    void startProcessing(const QStringList &files);
    void cancel();

signals:
    // 这些信号在子线程中发射，通过队列连接到达主线程
    void fileProcessed(const QString &path, qint64 size);
    void progressChanged(int current, int total);
    void finished(int successCount, int failCount);
    void errorOccurred(const QString &path, const QString &error);

private:
    // 内部信号：用于从主线程向子线程投递任务
    void doProcess(const QStringList &files);

    QThread m_thread;
    QAtomicInt m_cancelled{0};
};

// fileprocessor.cpp
#include "fileprocessor.h"
#include <QFile>
#include <QFileInfo>
#include <QDebug>

FileProcessor::FileProcessor(QObject *parent)
    : QObject(parent)
{
    // 将自己移到子线程——注意不能有 parent
    // 所以这里不 moveToThread(this)，而是用 Worker 模式
    // 但为了展示另一种模式，我们用 lambda 在子线程中工作
    m_thread.start();
}

FileProcessor::~FileProcessor()
{
    m_cancelled.storeRelease(1);
    m_thread.quit();
    m_thread.wait();
}

void FileProcessor::startProcessing(const QStringList &files)
{
    m_cancelled.storeRelease(0);

    // 通过 invokeMethod 将任务投递到子线程
    // 使用 lambda 捕获 files 值拷贝
    QMetaObject::invokeMethod(this, [this, files]() {
        doProcess(files);
    }, Qt::QueuedConnection);
}

void FileProcessor::cancel()
{
    m_cancelled.storeRelease(1);
}

void FileProcessor::doProcess(const QStringList &files)
{
    int success = 0, fail = 0;

    for (int i = 0; i < files.size(); ++i) {
        if (m_cancelled.loadAcquire()) {
            emit finished(success, fail);
            return;
        }

        const QString &path = files[i];
        QFile file(path);

        if (file.open(QIODevice::ReadOnly)) {
            QByteArray content = file.readAll();
            // 模拟处理...
            QThread::msleep(100);
            emit fileProcessed(path, content.size());
            success++;
        } else {
            emit errorOccurred(path, file.errorString());
            fail++;
        }

        emit progressChanged(i + 1, files.size());
    }

    emit finished(success, fail);
}
```

```cpp
// mainwindow.cpp —— 主线程使用
void MainWindow::onStartClicked()
{
    auto *processor = new FileProcessor; // 无 parent，后面手动管理

    // 创建专用线程并将 processor 移入
    auto *thread = new QThread(this);
    processor->moveToThread(thread);

    // 所有信号自动通过 QueuedConnection 到达主线程
    connect(processor, &FileProcessor::progressChanged,
            this, [this](int cur, int total) {
        m_progressBar->setRange(0, total);
        m_progressBar->setValue(cur);
    });

    connect(processor, &FileProcessor::fileProcessed,
            this, [this](const QString &path, qint64 size) {
        m_logView->appendPlainText(
            QString("✅ %1 (%2 bytes)").arg(path).arg(size));
    });

    connect(processor, &FileProcessor::errorOccurred,
            this, [this](const QString &path, const QString &err) {
        m_logView->appendPlainText(
            QString("❌ %1: %2").arg(path, err));
    });

    connect(processor, &FileProcessor::finished,
            this, [this, processor, thread](int ok, int fail) {
        m_statusLabel->setText(
            QString("完成：%1 成功，%2 失败").arg(ok).arg(fail));
        thread->quit();
    });

    // 清理链
    connect(thread, &QThread::finished, processor, &QObject::deleteLater);
    connect(thread, &QThread::finished, thread, &QObject::deleteLater);

    // 启动
    thread->start();

    // 通过信号触发处理（QueuedConnection 投递到子线程）
    QStringList files = getSelectedFiles();
    QMetaObject::invokeMethod(processor, [processor, files]() {
        processor->startProcessing(files);
    }, Qt::QueuedConnection);

    // 取消按钮
    connect(m_cancelBtn, &QPushButton::clicked, processor, [processor]() {
        processor->cancel();
    });
}
```

### 8.2 线程安全的日志系统

利用跨线程信号槽将多个线程的日志统一发送给主线程显示：

```cpp
class LogCollector : public QObject
{
    Q_OBJECT
public:
    static LogCollector *instance() {
        static LogCollector inst;
        return &inst;
    }

    // 可以从任意线程安全调用
    void log(const QString &module, const QString &message) {
        // 直接发射信号——AutoConnection 自动判断
        // 如果在非主线程调用 → QueuedConnection → 事件队列
        // 如果在主线程调用 → DirectConnection → 直接执行
        emit newLog(QDateTime::currentDateTime(), module, message,
                    reinterpret_cast<quintptr>(QThread::currentThreadId()));
    }

signals:
    void newLog(const QDateTime &time, const QString &module,
                const QString &message, quintptr threadId);

private:
    LogCollector() = default;
};

// 主线程中连接到 UI
connect(LogCollector::instance(), &LogCollector::newLog,
        m_logWidget, [this](const QDateTime &time, const QString &module,
                            const QString &msg, quintptr tid) {
    QString line = QString("[%1][%2][T:%3] %4")
        .arg(time.toString("hh:mm:ss.zzz"), module)
        .arg(tid, 0, 16)
        .arg(msg);
    m_logWidget->appendPlainText(line);
});

// 在任何线程中使用
LogCollector::instance()->log("Network", "下载完成");
LogCollector::instance()->log("Database", "查询耗时 230ms");
```

### 8.3 跨线程的请求-响应模式

```cpp
// 封装一个线程安全的异步服务
class DataService : public QObject
{
    Q_OBJECT
public:
    // 在任意线程调用，结果通过回调返回到调用者线程
    void fetchData(const QString &key,
                   std::function<void(const QVariant &)> callback)
    {
        QObject *context = new QObject; // 临时上下文对象
        QThread *callerThread = QThread::currentThread();

        // 在工作线程执行查询
        QMetaObject::invokeMethod(this, [this, key, callback, context, callerThread]() {
            // 这里在工作线程执行
            QVariant result = queryDatabase(key); // 模拟耗时查询

            // 将结果投递回调用者线程
            QMetaObject::invokeMethod(context, [callback, result, context]() {
                callback(result);
                context->deleteLater(); // 清理临时对象
            }, Qt::QueuedConnection);

        }, Qt::QueuedConnection);

        // 将 context 移动到调用者线程（确保回调在调用者线程执行）
        context->moveToThread(callerThread);
    }

private:
    QVariant queryDatabase(const QString &key) {
        QThread::msleep(500);
        return QVariant("数据: " + key);
    }
};

// 在主线程中使用
m_service->fetchData("user/123", [this](const QVariant &result) {
    // 这个回调在主线程执行，可以安全更新 UI
    m_label->setText(result.toString());
});
```

## 9. 常见陷阱与最佳实践

### 9.1 信号发射后对象被销毁

```cpp
// ❌ 信号已入队但对象被销毁
Worker *worker = new Worker;
worker->moveToThread(thread);
connect(worker, &Worker::done, this, &MainWindow::onDone);

// 子线程中
void Worker::process() {
    // ...
    emit done(result);
    delete this; // ❌ 信号还在队列中，槽可能访问已销毁的 worker
}

// ✅ 使用 deleteLater
void Worker::process() {
    // ...
    emit done(result);
    deleteLater(); // 安全：等事件循环处理完所有待处理事件后再删除
}
```

### 9.2 Lambda 捕获 this 的生命周期

```cpp
// ❌ 对象可能在 lambda 执行前被销毁
connect(worker, &Worker::result, this, [this](const QString &data) {
    m_label->setText(data); // this 可能已销毁！
});
// 解决：connect 的第三个参数是 this，Qt 会在 this 销毁时自动断开连接 ✅

// ❌ 第三个参数不是 this
connect(worker, &Worker::result, [this](const QString &data) {
    m_label->setText(data); // this 销毁后仍会执行！
});
// ✅ 始终提供 context 对象作为第三个参数
connect(worker, &Worker::result, this, [this](const QString &data) {
    m_label->setText(data);
});
```

### 9.3 connect 的第三个参数决定槽的线程

```cpp
// worker 在子线程，this（MainWindow）在主线程

// 情况 A：第三个参数是 this → 槽在 this 的线程（主线程）执行
connect(worker, &Worker::done, this, [this]() {
    m_label->setText("完成"); // ✅ 主线程
});

// 情况 B：没有第三个参数（context） → 槽在发射者的线程执行
connect(worker, &Worker::done, [this]() {
    m_label->setText("完成"); // ❌ 子线程！
});

// 情况 C：第三个参数是 worker → 槽在 worker 的线程（子线程）执行
connect(worker, &Worker::done, worker, [this]() {
    m_label->setText("完成"); // ❌ 子线程！
});
```

> **黄金规则**：Lambda 版 `connect()` 中，**第三个参数（context 对象）的线程亲和性决定了 Lambda 在哪个线程执行**。始终为跨线程连接指定正确的 context 对象。

### 9.4 高频信号的队列堆积

```cpp
// ❌ 高频信号可能导致事件队列无限增长
// 传感器 1000Hz 发射 → 主线程处理不过来 → 队列堆积 → 内存飙升 + 延迟增大

// ✅ 方案一：降频（在发射端节流）
class Sensor : public QObject {
    QElapsedTimer m_throttle;
    void onDataReady(double value) {
        if (m_throttle.elapsed() < 16) return; // 限制 60fps
        m_throttle.restart();
        emit valueChanged(value);
    }
};

// ✅ 方案二：使用 DirectConnection + 原子变量（避免队列）
QAtomicInt latestValue;
connect(sensor, &Sensor::rawValue, this, [&latestValue](int v) {
    latestValue.storeRelease(v);
}, Qt::DirectConnection);

// 主线程用定时器定期读取
QTimer *timer = new QTimer(this);
connect(timer, &QTimer::timeout, this, [this, &latestValue]() {
    m_display->setValue(latestValue.loadAcquire());
});
timer->start(16); // 60fps 更新 UI
```

### 9.5 跨线程信号槽 vs 其他跨线程方案

| 方案 | 优势 | 劣势 | 适用场景 |
|---|---|---|---|
| **信号槽** | 松耦合、类型安全、自动线程调度 | 参数需注册、有拷贝开销 | 通用跨线程通信 |
| **QMetaObject::invokeMethod** | 不需要预先 connect | 字符串版本无编译检查 | 一次性跨线程调用 |
| **QMutex + 共享数据** | 低延迟、零拷贝 | 需手动管理锁、易死锁 | 高频数据交换 |
| **QAtomicInt** | 无锁、最低延迟 | 仅限简单类型 | 标志位、计数器 |
| **QtConcurrent::run** | 最简单的异步 | 无法取消、不适合长期任务 | 一次性计算 |

## 10. 底层原理

### 10.1 队列连接的内部流程

```
emit mySignal(arg1, arg2)
        │
        ▼
QMetaObject::activate()
        │
        ├─ 同线程 → 直接调用 slot(arg1, arg2)
        │
        └─ 跨线程 →
            │
            ▼
        为每个参数调用 QMetaType::create()（深拷贝）
            │
            ▼
        创建 QMetaCallEvent {
            slotIndex,
            copiedArg1,
            copiedArg2
        }
            │
            ▼
        QCoreApplication::postEvent(receiver, event)
            │
            ▼
        事件被放入 receiver->thread() 的事件队列
            │
            ▼
        receiver 线程的事件循环取出事件
            │
            ▼
        QObject::event() 处理 QMetaCallEvent
            │
            ▼
        调用 slot(copiedArg1, copiedArg2)
            │
            ▼
        释放参数副本
```

### 10.2 线程亲和性检查源码（简化）

```cpp
// Qt 内部 QMetaObject::activate 简化逻辑
void QMetaObject::activate(QObject *sender, int signalIndex, void **argv)
{
    // 遍历连接列表
    for (Connection *c = connections; c; c = c->next) {
        QObject *receiver = c->receiver;
        Qt::ConnectionType type = c->connectionType;

        if (type == Qt::AutoConnection) {
            type = (receiver->d_func()->threadData == currentThreadData())
                   ? Qt::DirectConnection
                   : Qt::QueuedConnection;
        }

        switch (type) {
        case Qt::DirectConnection:
            // 直接调用
            callSlot(receiver, c->method, argv);
            break;

        case Qt::QueuedConnection:
            // 拷贝参数并投递事件
            postMetaCallEvent(receiver, c->method, argv);
            break;

        case Qt::BlockingQueuedConnection:
            if (receiver->d_func()->threadData == currentThreadData()) {
                qWarning("Dead lock detected!");
                return;
            }
            // 拷贝参数、投递事件、等待完成
            QSemaphore semaphore;
            postMetaCallEvent(receiver, c->method, argv, &semaphore);
            semaphore.acquire(); // 阻塞直到槽执行完毕
            break;
        }
    }
}
```

## 11. 总结

| 要点 | 说明 |
|---|---|
| **AutoConnection** | 默认行为，运行时判断：同线程 Direct，跨线程 Queued |
| **QueuedConnection** | 参数深拷贝 + 事件队列投递，异步执行，需要事件循环 |
| **DirectConnection** | 在发射线程直接调用，跨线程使用需确保槽函数线程安全 |
| **BlockingQueuedConnection** | 跨线程同步调用，小心死锁 |
| **参数注册** | 自定义类型必须 `Q_DECLARE_METATYPE` + `qRegisterMetaType` |
| **context 对象** | Lambda 连接的第三个参数决定槽的执行线程和自动断开 |
| **生命周期** | 使用 `deleteLater()` 而非 `delete`，提供 context 防止悬空调用 |
| **高频信号** | 注意队列堆积，必要时降频或使用原子变量 + 定时轮询 |
| **首选方案** | 跨线程通信优先使用信号槽，比手动加锁更安全、更不易出错 |
