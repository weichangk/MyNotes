## 1. 整体设计哲学：以错误码为主，避免异常

VBL 的错误处理遵循以下原则：
- **所有可能失败的函数返回 VBL::Result**（enum 错误码）
- **不使用 C++ 异常**（跨 DLL 边界异常传播不安全；移动端性能敏感）
- **用宏简化重复的错误检查代码**
- **提供字符串化工具方便日志输出**

## 2. Result 枚举 — X-Macro 技术

**文件**：Include/Common/VblResultDef.h

X-Macro 是一种用宏消除枚举和字符串映射重复代码的技巧：

    // Include/Common/VblResultDef.h
    // 第一步：定义枚举值列表宏（X-Macro 核心）
    #define RESULT_CONFIG_DEF \
        RESULT_ENUM_REPLACE(rOk,                    0)   \
        RESULT_ENUM_REPLACE(rFailed,             1001)   \
        RESULT_ENUM_REPLACE(rPathUnavailable,    1002)   \
        RESULT_ENUM_REPLACE(rInvalidState,       1003)   \
        RESULT_ENUM_REPLACE(rNetworkError,       1013)   \
        RESULT_ENUM_REPLACE(rPermissionDenied,   1018)   \
        RESULT_ENUM_REPLACE(rOutOfMemory,        1032)   \
        /* ... 共100+ 个错误码 ... */
    
    // 第二步：展开为枚举
    #define RESULT_ENUM_REPLACE(name, value) name = value,
    enum class Result {
        RESULT_CONFIG_DEF
    };
    #undef RESULT_ENUM_REPLACE
    
    // 第三步：展开为字符串映射（同一份宏，两种展开）
    #define RESULT_ENUM_REPLACE(name, value) case Result::name: return #name;
    inline const char* RESULT_TOSTRING(Result r) {
        switch (r) {
            RESULT_CONFIG_DEF
            default: return "Unknown";
        }
    }
    #undef RESULT_ENUM_REPLACE
    
    // 第四步：反向映射（字符串 -> 枚举）
    #define RESULT_ENUM_REPLACE(name, value) if (strcmp(str, #name) == 0) return Result::name;
    inline Result STRING_TORESULT(const char* str) {
        RESULT_CONFIG_DEF
        return Result::rUnrecognized;
    }
    #undef RESULT_ENUM_REPLACE

**X-Macro 的威力**：
- 新增错误码只需在 RESULT_CONFIG_DEF 中加一行
- 枚举、ToString、FromString 三个地方同步更新，不可能出现"加了枚举忘加字符串映射"的错误
- 完全消除了手动维护映射表的工作量

## 3. 常用检查宏（防御性编程）

**文件**：Include/Common/VblPredefined.h

    // 检查空指针：为 nullptr 则返回指定错误码
    #define VBL_CHECK_NULL_RETURN(ptr, retVal) \
        do { if (!(ptr)) { \
            VBL_LOG_ERROR("[%s:%d] null pointer: " #ptr, __FILE__, __LINE__); \
            return retVal; \
        }} while(0)
    
    // 检查条件：条件为 false 则返回错误码
    #define VBL_CHECK_RETURN(cond, retVal) \
        do { if (!(cond)) { \
            VBL_LOG_ERROR("[%s:%d] check failed: " #cond, __FILE__, __LINE__); \
            return retVal; \
        }} while(0)
    
    // 断言（仅 Debug 编译）
    #define VBL_ASSERT(cond, msg) \
        do { if (!(cond)) { \
            VBL_LOG_ERROR("ASSERT: %s at [%s:%d]", msg, __FILE__, __LINE__); \
            assert(cond); \
        }} while(0)
    
    // Result 检查：失败则直接返回该 Result
    #define VBL_CHECK_RESULT(result) \
        do { Result _r = (result); \
             if (_r != Result::rOk) return _r; \
        } while(0)

**使用示例**：

    // 文件: modules/BusinessLayer/ProjectEditor/VbProjectEditor.cpp（示意）
    Result VbProjectEditor::ImportMedia(const std::string& filePath) {
        VBL_CHECK_NULL_RETURN(m_timeline, Result::rInvalidState);
        VBL_CHECK_RETURN(!filePath.empty(), Result::rPathUnavailable);
        
        auto adapter = m_adapterFactory->CreateMediaInfoAdapter();
        VBL_CHECK_NULL_RETURN(adapter, Result::rFailed);
        
        DmMediaInfo info;
        VBL_CHECK_RESULT(adapter->GetMediaInfo(filePath, info));  // 失败直接返回
        
        VBL_CHECK_RESULT(m_sourceManager->AddMedia(info));
        return Result::rOk;
    }

## 4. 错误码分区设计

    // 从 VblResultDef.h 可以看出错误码的分区设计
    
    0              -- rOk（成功）
    
    400, 500       -- HTTP 状态码镜像（rBadRequest, rServerInternalError）
                      与网络层错误码直接对应，便于透传
    
    1000-1099      -- 通用错误（rFailed, rPathUnavailable, rInvalidState...）
    
    1100-1199      -- 网络/云服务错误（rNetworkError, rLoginError...）
    
    1200+          -- 模块特有错误（各模块可以在自己的范围内定义）
    
    负数区间       -- 某些模块用负数表示可恢复的特殊状态

## 5. 与 std::error_code 的对比

| 方面 | VBL::Result | std::error_code |
|------|-------------|-----------------|
| 类型 | enum class | 类对象（分类+值）|
| 字符串化 | RESULT_TOSTRING 宏 | error_code::message() |
| 扩展性 | 集中枚举 | 分散定义 error_category |
| 跨 ABI | enum 整数稳定 | 类对象跨 ABI 不稳定 |
| 与异常 | 不结合 | 可与 std::error_condition 结合 |

VBL 选择简单 enum 而非 std::error_code 的原因：
- 跨 DLL 边界传递 int（enum 底层）最安全
- 不需要 error_category 的分类查询功能
- 历史原因：项目早于 C++11 中 error_code 的成熟

## 6. 面试要点

1. **X-Macro 是消除"枚举-字符串双维护"问题的经典技巧**：一份 RESULT_CONFIG_DEF 宏
   通过三次不同的 RESULT_ENUM_REPLACE 展开，同时生成枚举定义、ToString、FromString，
   新增一个错误码只需改一个地方。

2. **不用异常的理由是 ABI 和性能**：C++ 异常跨 DLL 传播需要统一的运行时（MSVC 不同版本
   之间异常 ABI 不兼容）；移动端和 WASM 平台异常处理有额外开销；
   错误码更明确（函数签名告诉你它会失败，而异常是隐式的）。

3. **VBL_CHECK_NULL_RETURN 宏统一了错误检查的样板代码**：避免大量的
   if (!ptr) { log; return err; } 重复，用宏压缩为一行，且自动记录文件名和行号。

4. **错误码的 HTTP 镜像设计**：rBadRequest=400 等与 HTTP 状态码对齐，
   网络层收到 HTTP 错误码后可以直接映射为 VBL::Result 返回，减少转换代码。

5. **Result::rOk 的显式返回**：VBL 的约定是所有成功路径显式 return Result::rOk，
   不允许函数"隐式成功"。这让代码审查时很容易确认所有路径都有明确的返回值。

## 7. 可能被追问的问题

**Q1：X-Macro 技术有什么缺点？**
①可读性稍差，不熟悉的开发者第一次看到可能困惑（为什么同一段宏被展开三次）；
②调试困难，宏展开后的代码在 IDE 中难以设置断点；
③宏没有作用域，容易与其他宏冲突（需要在每次展开后 #undef）。
VBL 通过代码注释和文档缓解了可读性问题。

**Q2：如果函数深层嵌套调用，错误码如何传递？**
VBL_CHECK_RESULT 宏：每层调用都检查子函数返回值，失败时直接向上返回，
形成自动的错误传播链。类似 Rust 的 ? 运算符，C++23 的 std::expected 也有类似功能。

**Q3：调用方如何区分"有 bug 的错误"和"正常业务错误"？**
区分策略：rFailed（通用失败，可能是 bug）vs 具体错误码（业务逻辑的预期错误）。
VBL_ASSERT 用于不应该发生的错误（调试期崩溃，发现 bug）；
返回 Result 用于可以优雅处理的错误（告知用户操作失败）。

**Q4：Result 和 bool 相比有什么优势？什么时候用 bool？**
Result 优势：携带失败原因，便于日志诊断和错误处理分支。
bool 适用场景：简单的是/否查询（IsValid()、HasMedia()），不需要失败原因。
VBL 约定：修改状态的函数返回 Result，查询状态的函数返回 bool 或具体值。
