# 02 · 设计模式 — 工厂模式（Factory）

> **面试价值**：⭐⭐⭐⭐  **优先级**：P0  
> **所属专题**：设计模式  
> **相关文件**：
> - `modules/DataModel/` — `IDmClipFactory` / `DmClipFactory`
> - `modules/Adapter/wes/` — `DmWesAdapterFactory` / `IAdapterFactory`
> - `Interface/` — 接口定义

---

## 1. 概念介绍

工厂模式将对象的**创建**与**使用**解耦，调用方只依赖抽象接口，不关心具体实现类。

| 模式变体 | 适用场景 | VBL 对应 |
|----------|----------|---------|
| **工厂方法**（Factory Method）| 一个产品族，多个具体实现 | `IDmClipFactory` |
| **抽象工厂**（Abstract Factory）| 多个产品族，整体切换 | `IAdapterFactory`（WES/NLE两套）|
| **静态工厂**（Static Factory）| 简单枚举决定实现 | 内部 `Create()` 辅助函数 |

## 2. 项目中的实现

### 2.1 `IDmClipFactory` — DataModel 层 Clip 工厂

VBL 的数据模型层用工厂接口创建各类 Clip 对象（视频 Clip、音频 Clip、特效 Clip 等）。调用方只持有 `IDmClip*` 接口，工厂负责根据参数创建正确的具体类型。

**文件**：`modules/DataModel/` + `Interface/DataModel/`

```cpp
// Interface/DataModel/IDmClipFactory.h（接口定义）
class IDmClipFactory {
public:
    virtual ~IDmClipFactory() = default;
    
    // 工厂方法：根据类型创建 Clip
    virtual SafePtr<IDmClip> CreateClip(DmClipType type) = 0;
    virtual SafePtr<IDmClip> CreateClipFromJson(const std::string& json) = 0;
    virtual SafePtr<IDmVideoClip> CreateVideoClip(const DmMediaInfo& info) = 0;
    virtual SafePtr<IDmAudioClip> CreateAudioClip(const DmMediaInfo& info) = 0;
    virtual SafePtr<IDmEffectClip> CreateEffectClip(const std::string& effectId) = 0;
};

// modules/DataModel/DmClipFactory.h（具体工厂）
class DmClipFactory : public IDmClipFactory {
public:
    SafePtr<IDmClip> CreateClip(DmClipType type) override;
    SafePtr<IDmClip> CreateClipFromJson(const std::string& json) override;
    SafePtr<IDmVideoClip> CreateVideoClip(const DmMediaInfo& info) override;
    // ...
};
```

**关键设计**：  
- 返回值是 `SafePtr<IDmClip>`（接口的智能指针），调用方无需知道具体类型  
- `CreateClipFromJson` 实现了从序列化数据还原对象的反序列化工厂  
- 配合 `DmClipType` 枚举，可在运行时动态选择构造路径

### 2.2 `IAdapterFactory` / `DmWesAdapterFactory` — 多引擎适配器工厂

VBL 支持 WES（Wondershare Engine Studio）和 NLE（Non-Linear Editing）两套底层渲染引擎。通过抽象工厂，上层代码无感切换：

**文件**：`modules/Adapter/wes/DmWesAdapterFactory.h`

```cpp
// Interface/Adapter/IAdapterFactory.h（抽象工厂接口）
class IAdapterFactory {
public:
    virtual ~IAdapterFactory() = default;
    
    // 每个工厂方法创建对应的 Adapter 对象
    virtual SafePtr<IMediaInfoAdapter>   CreateMediaInfoAdapter() = 0;
    virtual SafePtr<ITimelineAdapter>    CreateTimelineAdapter() = 0;
    virtual SafePtr<IClipAdapter>        CreateClipAdapter() = 0;
    virtual SafePtr<IRenderAdapter>      CreateRenderAdapter() = 0;
};

// modules/Adapter/wes/DmWesAdapterFactory.h（WES 引擎工厂）
class DmWesAdapterFactory : public IAdapterFactory {
public:
    SafePtr<IMediaInfoAdapter>   CreateMediaInfoAdapter() override;  // 返回 WES 实现
    SafePtr<ITimelineAdapter>    CreateTimelineAdapter() override;   // 返回 WES 实现
    // ...
};

// modules/Adapter/nle/DmNleAdapterFactory.h（NLE 引擎工厂）
class DmNleAdapterFactory : public IAdapterFactory {
public:
    SafePtr<IMediaInfoAdapter>   CreateMediaInfoAdapter() override;  // 返回 NLE 实现
    // ...
};
```

**引擎切换的关键代码**：
```cpp
// VBL 初始化时根据配置选择引擎
// 文件路径: modules/BusinessLayer/ProjectEditor/VbProjectEditor.cpp（示意）
std::unique_ptr<IAdapterFactory> CreateAdapterFactory(EngineType type) {
    switch (type) {
        case EngineType::WES:
            return std::make_unique<DmWesAdapterFactory>();
        case EngineType::NLE:
            return std::make_unique<DmNleAdapterFactory>();
        default:
            VBL_ASSERT(false, "Unknown engine type");
            return nullptr;
    }
}

// 使用方只依赖接口，完全不知道底层是 WES 还是 NLE
IAdapterFactory* factory = GetAdapterFactory();
auto mediaAdapter = factory->CreateMediaInfoAdapter();
auto info = mediaAdapter->GetMediaInfo(filePath);
```

### 2.3 BsPluginLoader 中的动态工厂

插件系统通过动态库加载实现运行时工厂，是工厂模式的延伸：

```cpp
// modules/BaseService/BsPluginManager/BsPluginLoader.cpp（示意）
// 通过 dlopen/LoadLibrary 加载插件动态库，获取工厂函数指针
using CreatePluginFn = IPlugin*(*)();

IPlugin* LoadPlugin(const std::string& libPath) {
    void* handle = dlopen(libPath.c_str(), RTLD_LAZY);  // macOS/Linux
    if (!handle) return nullptr;
    
    auto createFn = (CreatePluginFn)dlsym(handle, "CreatePlugin");
    return createFn ? createFn() : nullptr;
}
```

## 4. 面试要点

1. **工厂模式的核心价值是解耦**：`IDmClipFactory` 让 DataModel 层的调用方不依赖具体 Clip 实现类，当底层引擎从 WES 切换到 NLE 时，只需切换工厂，上层代码不变。

2. **抽象工厂解决"产品族"切换问题**：`IAdapterFactory` 一次性封装了 MediaInfoAdapter、TimelineAdapter、ClipAdapter 等一整套底层对象，保证同一时刻只会用同一引擎的全套实现，不会出现"WES 的 Timeline + NLE 的 Clip"这种混搭。

3. **工厂返回接口的智能指针**：VBL 返回 `SafePtr<IDmClip>` 而非裸指针，调用方无需手动管理内存，且类型信息完全封装在接口后面，符合依赖倒置原则（DIP）。

4. **插件系统 = 动态工厂**：`BsPluginLoader` 通过 `dlopen` + 函数指针实现运行时加载工厂，是工厂模式的动态扩展，适合需要热插拔的组件（滤镜插件、AI模块等）。

5. **与依赖注入（DI）结合**：VBL 在模块初始化时将工厂注入业务对象（构造函数注入），避免业务代码直接调用 `CreateXxx()`，提升可测试性。

## 5. 可能被追问的问题

**Q1：工厂方法和抽象工厂有什么区别？VBL 中分别在哪里用？**  
> 工厂方法：单一产品等级，一个接口一种创建方法（`IDmClipFactory` — 只创建各种 Clip）。  
> 抽象工厂：产品族，多个相关接口的集合（`IAdapterFactory` — 创建一整套 WES 或 NLE 适配器）。  
> 区别：抽象工厂保证同族产品的一致性；工厂方法更灵活，可单独替换。

**Q2：工厂模式和 `new` 直接创建对象相比有什么缺点？**  
> 缺点：增加了类的数量，代码复杂度上升；小项目过度设计反而增加维护成本。  
> VBL 的权衡：底层引擎切换是刚需，多引擎适配带来的复杂度是必要的，工厂模式在此有充分理由。

**Q3：`DmWesAdapterFactory` 创建的对象生命周期如何管理？**  
> 工厂方法返回 `SafePtr<IXxxAdapter>`，由 VBL 的 intrusive 引用计数体系管理。持有 `SafePtr` 的对象析构时会自动 `Release()`，最后一个持有者释放时对象被销毁。

**Q4：如何对使用了工厂的业务代码进行单元测试？**  
> 用 Mock 工厂替换真实工厂（依赖注入）：
> ```cpp
> class MockAdapterFactory : public IAdapterFactory {
>     SafePtr<IMediaInfoAdapter> CreateMediaInfoAdapter() override {
>         return SafePtr<IMediaInfoAdapter>(new MockMediaInfoAdapter());
>     }
>     // ...
> };
> // 测试时注入 mock
> VbProjectEditor editor(new MockAdapterFactory());
> ```

**Q5：动态库插件的工厂函数有什么约定？为什么用 extern "C"？**  
> 因为 C++ 有 name mangling（名字改编），不同编译器/版本生成的符号名不同，跨动态库调用会找不到符号。`extern "C"` 告诉编译器使用 C 的名字规则（不改编），确保 `dlsym("CreatePlugin")` 能找到函数。这也是为什么 VBL 的 wrapper 层也大量使用 `extern "C"` 的原因。
