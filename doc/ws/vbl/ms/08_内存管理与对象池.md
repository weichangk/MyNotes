## 1. 三层内存管理策略

VBL 的内存管理分三个层次：

    层次1：对象所有权 — SafePtr/RefCnt intrusive 引用计数（见文档05）
    层次2：对象复用   — 对象池（ObjectPool），避免频繁 new/delete
    层次3：资源 RAII  — 线程/文件/句柄等资源的自动管理

## 2. 对象池 — MediaInfoAdapterObjectPool

**文件**：modules/Adapter/wes/（MediaInfoAdapterObjectPool 相关）

对象池用于复用创建代价高的对象（如 MediaInfoAdapter 需要初始化底层引擎），
避免频繁的内存分配和对象初始化开销：

    // 对象池（单例 + shared_mutex + map）
    class MediaInfoAdapterObjectPool {
    public:
        // 单例访问
        static MediaInfoAdapterObjectPool& Instance() {
            static MediaInfoAdapterObjectPool s_instance;
            return s_instance;
        }
        
        // 借出对象（池中有则取，没有则创建）
        std::shared_ptr<IMediaInfoAdapter> Acquire() {
            std::unique_lock<std::shared_mutex> lock(m_mutex);
            if (!m_pool.empty()) {
                auto obj = m_pool.back();
                m_pool.pop_back();
                return obj;
            }
            lock.unlock();
            return CreateAdapter();  // 池为空时创建新对象
        }
        
        // 归还对象（重置状态后放回池中）
        void Release(std::shared_ptr<IMediaInfoAdapter> adapter) {
            adapter->Reset();  // 重要：归还前清理状态
            std::unique_lock<std::shared_mutex> lock(m_mutex);
            if (m_pool.size() < m_maxSize) {
                m_pool.push_back(adapter);
            }
            // 超出最大池容量则直接销毁（shared_ptr 引用计数归零）
        }
        
        // 预热：提前创建 N 个对象
        void Warmup(int count) {
            for (int i = 0; i < count; ++i) {
                m_pool.push_back(CreateAdapter());
            }
        }
        
    private:
        std::vector<std::shared_ptr<IMediaInfoAdapter>> m_pool;
        mutable std::shared_mutex                       m_mutex;  // 读写锁
        size_t                                          m_maxSize{16};
    };

**RAII 风格的对象池使用**：

    // 用 shared_ptr 的自定义 deleter 实现自动归还
    std::shared_ptr<IMediaInfoAdapter> AcquireScoped() {
        auto adapter = Acquire();
        // 自定义 deleter：析构时自动归还池
        return std::shared_ptr<IMediaInfoAdapter>(
            adapter.get(),
            [this, adapter](IMediaInfoAdapter*) {
                Release(adapter);  // 析构时自动归还
            }
        );
    }
    
    // 调用方使用（出作用域自动归还，无需手动 Release）
    {
        auto adapter = pool.AcquireScoped();
        auto info = adapter->GetMediaInfo(filePath);
    }  // 出作用域 --> 自动归还到池中

## 3. RAII 在线程和任务管理中的应用

**文件**：modules/BaseService/BackgroundTaskManager/BsBackgroundTask.hpp

RAII（Resource Acquisition Is Initialization）原则在 VBL 中广泛应用：

    // shared_ptr<thread> 作为线程的 RAII 管理
    class BsBackgroundTask {
    private:
        std::shared_ptr<std::thread> m_thread;  // RAII：thread 作为资源
    
    public:
        // Start() 时获取资源
        void Start() {
            m_thread = std::make_shared<std::thread>([this]() {
                m_func(this);
            });
        }
        
        // 析构时安全 join（而非暴力 detach 或 terminate）
        ~BsBackgroundTask() {
            Cancel();
            if (m_thread && m_thread->joinable()) {
                m_thread->join();  // 等待任务完成后再析构
            }
        }
    };

## 4. 引用计数内存管理的注意事项

### 4.1 循环引用问题

    // 错误示例：A 持有 B，B 持有 A -> 循环引用，内存泄漏
    struct A { SafePtr<B> b; };
    struct B { SafePtr<A> a; };  // 两者引用计数永远不为 0
    
    // 正确示例：弱方向用 WeakPtr
    struct A { SafePtr<B> b; };      // A 强持有 B
    struct B { WeakPtr<A> a; };      // B 弱持有 A（打破循环）

VBL 中的循环引用场景：
- DataModel 层：IDmTimeline 强持有 IDmClip，IDmClip 弱持有 IDmTimeline（反向引用）
- BusinessLayer：VbProjectEditor 强持有各 Service，Service 弱持有 ProjectEditor 的回调

### 4.2 SafePtr 和原始指针的使用边界

    // 规则：所有权传递用 SafePtr，临时使用/内部用裸指针
    
    // 接口返回值：SafePtr（转移所有权）
    SafePtr<IDmClip> GetClip(int index) const;
    
    // 内部循环、热路径：裸指针（不改变所有权，避免原子操作开销）
    void ProcessClips() {
        const auto& clips = m_timeline->GetClips();
        for (IDmClip* clip : clips) {  // 遍历时用裸指针，生命周期由 timeline 保证
            clip->Process();
        }
    }

## 5. 内存池 vs 对象池 vs std::allocator

| 方式 | 解决的问题 | VBL 中的应用 |
|------|------------|-------------|
| 对象池 | 复用创建代价高的对象 | MediaInfoAdapterObjectPool |
| 内存池 | 减少 heap 碎片化 | 未见显式使用（依赖系统 allocator）|
| RAII | 资源自动释放 | SafePtr、shared_ptr<thread> |
| 引用计数 | 共享对象所有权 | SafePtr/RefCnt 体系 |

## 6. 面试要点

1. **对象池的核心价值是避免频繁初始化**：MediaInfoAdapter 需要初始化底层 WES/NLE 引擎，
   代价远超 malloc。对象池预先创建并复用，将初始化代价均摊到预热阶段。

2. **shared_ptr 的自定义 deleter 实现 RAII 风格对象池**：对象归还逻辑封装在 deleter 中，
   调用方出作用域自动归还，不需要手动调用 Release()，完全 RAII 化。

3. **归还前必须 Reset**：对象池中的对象会被复用，如果不清理上一次使用的状态，
   下一个借用者会看到脏数据。Reset() 是对象池使用的不变量。

4. **循环引用用 WeakPtr 打破**：SafePtr 强引用形成环时计数永远不为 0。
   在设计数据结构时，父子关系中的"子->父"方向用 WeakPtr。

5. **热路径裸指针，边界 SafePtr**：在渲染帧等热路径中，用裸指针避免原子操作开销；
   在所有权传递（函数返回值、存储到容器）时用 SafePtr 保证生命周期安全。

## 7. 可能被追问的问题

**Q1：对象池的最大容量如何确定？超出怎么处理？**
根据业务峰值并发数确定（如最大 16 个并发 MediaInfo 查询则池容量设 16~32）。
超出容量的归还对象直接销毁（shared_ptr 引用计数归零触发析构），
不会导致内存泄漏，只是下次 Acquire 时需要重新创建。

**Q2：对象池如何处理线程安全？**
MediaInfoAdapterObjectPool 用 std::shared_mutex：
- Acquire/Release 涉及写操作，用 unique_lock 独占；
- 如果只有 pool.size() 查询，可以用 shared_lock 共享读。

**Q3：为什么 BsBackgroundTask 析构时要先 Cancel() 再 join()？**
如果任务是死循环（如监听线程），不先 Cancel() 就 join() 会永久阻塞。
先 Cancel() 设置停止标志，任务内部检测到标志后退出循环，join() 才能正常返回。

**Q4：RAII 和 C++ 异常处理的关系？**
RAII 是 C++ 异常安全的基础。即使函数因异常提前退出，局部变量（SafePtr、lock_guard 等）
的析构函数也会被调用，自动释放资源，防止泄漏。
VBL 以错误码为主不用异常，但 RAII 仍然有价值（防止逻辑错误导致的资源泄漏）。
