## 1. 概念介绍

单例模式确保一个类只有一个实例，并提供全局访问点。C++ 中有多种实现方式，各有优劣：

| 实现方式 | 线程安全 | 懒加载 | 内存管理 |
|----------|----------|--------|----------|
| Meyers 单例（局部静态变量）| ✅ C++11 保证 | ✅ | 程序退出自动析构 |
| 原始指针 + AddRef/Release | 手动控制 | ✅ | 手动引用计数 |
| `std::call_once` + static | ✅ | ✅ | 手动 |
| 双检锁（DCL，旧方案）| ❌ 需 volatile | ✅ | 手动 |

---

## 2. 项目中的三种实现

### 2.1 Meyers 单例 — `BsProxyFileName`

**文件**：`modules/BaseService/BsProxy/BsProxyFileName.h`

这是最推荐的 C++11 写法。利用局部静态变量的初始化只执行一次且线程安全的特性：

```cpp
// BsProxyFileName.h
class BsProxyFileName {
public:
    static BsProxyFileName& Instance() {
        static BsProxyFileName instance;   // C++11: 线程安全，仅初始化一次
        return instance;
    }

    BsProxyFileName(const BsProxyFileName&) = delete;
    BsProxyFileName& operator=(const BsProxyFileName&) = delete;

private:
    BsProxyFileName() = default;
    ~BsProxyFileName() = default;
};
```

**关键点**：  
- `static` 局部变量在 C++11 中保证初始化的线程安全性（§6.7 [stmt.dcl]）  
- 显式 `delete` 拷贝构造和赋值运算符，防止意外复制  
- 返回引用而非指针，避免调用方误用 `delete`  

---

### 2.2 原始指针 + AddRef 风格 — `DmGlobalBlockEvent`

**文件**：`modules/DataModel/`（DataModel 层全局事件对象）

这是项目中 COM 风格对象的单例写法，配合 VBL 自定义引用计数体系使用：

```cpp
// 伪代码示意（DmGlobalBlockEvent 风格）
class DmGlobalBlockEvent : public RefCnt<DmGlobalBlockEvent> {
public:
    static DmGlobalBlockEvent* GetInstance() {
        if (!s_instance) {
            s_instance = new DmGlobalBlockEvent();
            s_instance->AddRef();   // 持有一个引用，防止被释放
        }
        return s_instance;
    }

    static void DestroyInstance() {
        if (s_instance) {
            s_instance->Release();  // 释放持有的引用
            s_instance = nullptr;
        }
    }

private:
    static DmGlobalBlockEvent* s_instance;
};
```

**关键点**：  
- 使用 `AddRef/Release` 而不是 `delete`，与 VBL 的 `SafePtr<T>` 体系对齐  
- 需要显式调用 `DestroyInstance()` 管理生命周期，生命周期由宿主模块控制  
- 这是 COM 组件模型（如 Windows COM）中常见的单例风格  

---

### 2.3 模块级单例 — `VbPreferenceManager`

**文件**：`modules/BusinessLayer/PreferenceManager/VbPreferenceManager.h`

业务层偏好设置管理器，生命周期绑定到整个 VBL 会话：

```cpp
// VbPreferenceManager.h（典型模式）
class VbPreferenceManager {
public:
    static VbPreferenceManager* Instance();       // 获取单例
    static void Initialize(/* config params */);  // 显式初始化
    static void Destroy();                        // 显式销毁

    // 业务方法
    void SetPreference(const std::string& key, const std::string& value);
    std::string GetPreference(const std::string& key) const;

private:
    VbPreferenceManager() = default;
    static VbPreferenceManager* s_instance;
    std::unordered_map<std::string, std::string> m_prefs;
};
```

**关键点**：  
- 与 Meyers 单例不同，提供显式 `Initialize/Destroy`，便于测试时重置状态  
- 在 VBL 初始化流程（`IVbLModule::Init()`）中统一管理各模块单例的生命周期

```cpp
BsProxyFileName& fileNameMgr = BsProxyFileName::Instance();
```

```cpp
// 文件路径: modules/DataModel/Xxx/DmGlobalBlockEvent.cpp
DmGlobalBlockEvent* DmGlobalBlockEvent::s_instance = nullptr;

DmGlobalBlockEvent* DmGlobalBlockEvent::GetInstance() {
    if (!s_instance) {
        s_instance = new DmGlobalBlockEvent();
        s_instance->AddRef();  // ← 持有引用，防止外部 Release 导致意外销毁

```

## 4. 面试要点（直接可说的核心结论）

1. **三种实现各有场景**：Meyers 单例用于无外部依赖的工具类（`BsProxyFileName`）；COM 风格用于需要与 `SafePtr<T>` 体系集成的对象（`DmGlobalBlockEvent`）；显式 Init/Destroy 用于有初始化参数的业务模块（`VbPreferenceManager`）。

2. **Meyers 单例的线程安全保证来自 C++11 标准**：标准规定局部静态变量的初始化是线程安全的，编译器生成的代码等价于一次 `call_once`，不需要自己加锁。

3. **COM 风格单例与 `SafePtr` 体系配合**：通过 `AddRef()` 持有自身引用，防止被外部意外释放；这在多模块协作时比 `std::shared_ptr<T>` 更适合跨 ABI 边界传递。

4. **单例的析构顺序问题**：多个全局/静态单例之间若有依赖，必须显式控制销毁顺序（VBL 在 `IVbLModule::Release()` 中按依赖顺序逆序销毁）；Meyers 单例的析构顺序取决于初始化顺序，复杂依赖时需小心"静态初始化顺序问题（SIOF）"。

5. **测试友好性**：带 `Initialize/Destroy` 接口的单例可以在单元测试 `TearDown()` 中重置状态，纯 Meyers 单例则很难重置，测试时需注意用 mock 或依赖注入替代。

## 5. 可能被追问的问题

**Q1：Meyers 单例为什么是线程安全的？底层如何实现？**  
> C++11 标准（§6.7）规定：如果多个线程并发到达局部静态变量的初始化点，只有一个线程会执行初始化，其他线程会阻塞等待完成。编译器通常用一个 guard variable（本质是 `pthread_once` 或 `call_once` 风格的标志位）实现。

**Q2：单例模式的缺点有哪些？项目中如何规避？**  
> 缺点：①全局状态难以测试；②隐式依赖（不看代码不知道用了哪些单例）；③多线程下访问共享数据需要加锁。  
> VBL 的规避方式：业务模块通过接口注入（`IVbPreferenceManager*`）而非直接调用 `Instance()`，便于测试时替换 mock；显式 Init/Destroy 而非全局生命周期。

**Q3：`static DmGlobalBlockEvent* s_instance` 会有线程安全问题吗？**  
> 有，如果多线程同时调用 `GetInstance()`，在 `if (!s_instance)` 处可能产生竞争。完整的线程安全写法需要加 `std::mutex` + 双检锁，或改用 Meyers 写法。VBL 中此类单例通常在主线程统一初始化，由框架保证调用时序，而非依赖 `GetInstance()` 的线程安全。

**Q4：如何对单例进行单元测试？**  
> 策略1：对外暴露 `Reset()`（仅 `#ifdef VBL_TESTING` 时编译），允许测试 teardown 时重置。  
> 策略2：将单例封装为接口，业务代码依赖接口而非具体类，测试时注入 mock。  
> VBL 的 `AppendModulesBasic()` 会自动生成测试 target，测试时通过独立进程隔离全局状态。

**Q5：double-checked locking（双检锁）为什么在 C++11 之前是不安全的？**  
> 因为编译器和 CPU 都可能对指令重排序。`s_instance = new Singleton()` 实际包含三步：①分配内存②调用构造③赋值给指针，步骤②③可能被重排，导致另一个线程看到非空但未初始化的指针。C++11 引入了内存模型和 `std::atomic`，用 `atomic<Singleton*>` + `memory_order_acquire/release` 可以正确实现 DCL。
