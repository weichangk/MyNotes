### 0.2 类继承树

```
FFTaskRunner  （抽象基类）
├── FFParallelTaskRunner  — 并行运行器（每个任务独立 TaskQueuePage）
├── FFQueueTaskRunner     — 队列运行器（多任务共用同一 TaskQueuePage，FIFO/LIFO）
└── FFThreadTaskRunner    — 独立线程运行器（专属 WorkerThread，任务串行）

FFThreadGroup             — 管理一组 FFWorkerThread + TaskPagePriorityQueue
FFWorkerThread            — 单条工作线程（继承自 FFThread/QThread）
FFTaskScheduler           — 全局调度 + 退出同步（原子 stateFlag）
FFTaskQueuePageImpl       — 任务队列页（持有任务列表 + AtomicOperator）
FFTaskQueuePageProxy      — 轻量代理，线程间传递 TaskQueuePage 的引用
```

```
  调用 postTask()
```

### 0.4 三条 TaskRunner 的投递差异

```
并行（Parallel）：
  postTask(t1)  → 新建 TaskQueuePage₁ → Foreground 线程（并发）
  postTask(t2)  → 新建 TaskQueuePage₂ → Foreground 线程（并发）

队列（Queue / FIFO）：
  postTask(t1) ┐
  postTask(t2) ├→ 共用同一 TaskQueuePage → 顺序执行 t1 → t2 → t3
  postTask(t3) ┘

独立线程（Thread）：
  专属 FFWorkerThread → 与其他线程组隔离
  postTask(t1) → 本线程串行执行
```

### 0.5 退出状态机（stateFlag 原子位图）

```
stateFlag（quint32）布局：
  bit 0       = ExitFlag（1 = 已触发退出）
  bit 1..31   = BlockingTaskCount（需等待完成的任务计数）

正常运行：  0b000...000_0
触发退出：  0b000...000_1
有阻塞任务：0b000...010_1（blockCount=1，已退出）
全部完成：  0b000...000_1 → wakeOne() → waitExitComplete() 返回
```

## 第 1 章 设计动机与定位

### 1.1 为什么不直接用 QtConcurrent / QThreadPool

| 对比项 | QThreadPool + QtConcurrent | FFAsync |
|--------|---------------------------|---------|
| 优先级调度 | 无优先级区分 | 4 级 TaskPriority + 2 级 ThreadPriority |
| 回调线程保证 | 需要手动 QMetaObject::invokeMethod | QtPromise::then() 自动回主线程 |
| 序列化任务 | 需要自行加锁排队 | createQueueTaskRunner() 直接提供 |
| 独立串行线程 | 需要自建 QThread + 信号槽 | createThreadTaskRunner() 直接提供 |
| 优雅退出 | QThreadPool::waitForDone() 粒度粗 | kMustExecTaskOnExit 细粒度控制 |
| 退出时任务策略 | 全部等待 / 全部丢弃二选一 | 三种行为 Ignore/IgnoreUnRun/MustExec |
| Promise 链 | 不支持 | 原生 .then().fail().finally() |

### 1.2 FFAsync 在 Filmora 中的定位

FFAsync 是整个应用的**唯一**异步调度框架（`3rdparty/FilmoraFrameworkPlatform`），取代了早期分散的：
- `QtConcurrent::run()`（无优先级，无序列化）
- `QThreadPool::globalInstance()`（全局竞争，无退出控制）
- `QFutureWatcher + QTimer`（回调写法繁琐，见 `FFThumbnailCache` 遗留注释）

### 2.1 类声明

```cpp
// Include/FFAsync/FFAsync.h
namespace FFAsync {

class FFAsync final          // final：禁止继承
{
public:
    FFAsync() = delete;      // 纯静态类，不可实例化
    ~FFAsync() = delete;

    // ① 投递无返回值任务（fire-and-forget）
    static void postTask(TaskClosure task, FFTasksFeature = {});

    // ② 投递有返回值任务，返回 QtPromise::QPromise<T>
    template<class F>

    postReplyableResultTask(F&& task, FFTasksFeature = {});

    // ③ 投递有返回值任务，T=void 的特化版本
    template<class F>
    static std::enable_if_t<std::is_same_v<void, TaskResult_t<F>>,
                             QtPromise::QPromise<void>>
    postReplyableResultTask(F&& task, FFTasksFeature = {});

    // ④ 创建并行任务运行器
    static std::shared_ptr<FFTaskRunner>        createParallelTaskRunner(FFTasksFeature = {});
    // ⑤ 创建队列（串行）任务运行器
    static std::shared_ptr<FFQueueTaskRunner>   createQueueTaskRunner(FFTasksFeature = {});
    // ⑥ 创建独立线程串行任务运行器

};

} // namespace FFAsync
```

## 第 3 章 FFTasksFeature — 任务特征描述器

### 3.1 四个维度的任务特征

```cpp
// Include/FFAsync/FFTasksFeature.h

// ① 任务优先级（影响线程组调度频率）
enum TaskPriority {
    kLowestPriority  = 0,   // 同 kTryBestPriority：埋点上报、懒加载、配置同步
    kTryBestPriority = 0,
    kNormalPriority,         // 进度更新、文件下载、图片加载
    kTimeCriticalPriority,   // 用户操作后立即响应，如缩略图加载（默认值）
    kHighestPriority = kTimeCriticalPriority,
};

// ② 线程优先级（影响 OS 调度 CPU 时间片）
enum TaskThreadPriority {
    kBackground,   // 低线程优先级
    kNormal,       // 正常线程优先级（默认）
};

// ③ 退出时任务行为
enum TaskBehaviorOnExit {
    kIgnoreTaskOnExit,       // 退出时立即丢弃（即使未运行）
    kIgnoreUnRunTaskOnExit,  // 运行中的任务不中断，未运行的丢弃（默认）
    kMustExecTaskOnExit,     // 无论如何都必须运行完（例如：日志刷盘、数据保存）
};

// ④ 出队方式
enum TaskDequeueMethod {
    kFIFO,   // 先进先出（默认，用于队列 TaskRunner）
    kLIFO,   // 后进先出（用于优先展示最新请求的场景）
};
```

### 3.2 构造方式 — 类型安全的可变参数构造

```cpp
// 任意顺序传入 0~4 个特征枚举值，编译期校验类型合法性
FFTasksFeature feat(
    FFAsync::kTimeCriticalPriority,   // 高优先级
    FFAsync::kMustExecTaskOnExit,     // 退出时必须执行
    FFAsync::kLIFO                    // 后进先出出队
);

// 如果某个维度不传，使用默认值：
// TaskPriority        = kNormalPriority
// TaskThreadPriority  = kNormal
// TaskBehaviorOnExit  = kIgnoreUnRunTaskOnExit
// TaskDequeueMethod   = kFIFO
FFTasksFeature defaultFeat{};  // 全默认
```

**实现原理**：`getEnumValueFromArgs<EnumType, defaultValue>(args...)` 用 `if constexpr` +
`QtPrivate::List` 在编译期从参数包中找到对应枚举类型的值，找不到就用默认值。
**编译期静态断言**：传入非 `ValidTasksFeatureTypes` 中的类型会导致编译失败。

## 第 4 章 TaskRunner 体系 — 三种运行器

### 4.1 抽象基类 FFTaskRunner

```cpp
// Include/FFAsync/FFTaskRunner.h
class FFTaskRunner
{
public:
    virtual bool postTask(TaskClosure task) = 0;
    virtual void updateTaskRunnerPropirty(TaskPriority) = 0;

    // 模板方法：将带返回值的 F 包装为带 Promise 回复的任务
    template<class F>
    QtPromise::QPromise<typename TaskType<std::decay_t<F>>::ResultType>
    postReplyableResultTask(F&& task);

    // 便捷方法：投递无返回值的带回复任务（返回 QPromise<void>）
    QtPromise::QPromise<void> postReplyableTask(TaskClosure);
};
```

### 4.2 三种具体 TaskRunner 对比

| 运行器 | 内部机制 | 任务执行方式 | 典型使用场景 |
|--------|---------|------------|-------------|
| `FFParallelTaskRunner` | 每个任务新建一个 `TaskQueuePage` | **并发**，多任务同时执行 | 批量缩略图加载 |
| `FFQueueTaskRunner` | 所有任务共用同一个 `TaskQueuePage` | **串行**，严格按投递顺序执行 | 轮播资源加载、AI 结果处理 |
| `FFThreadTaskRunner` | 持有专属的 `FFWorkerThread` | 独立线程上**串行**执行 | 播放器多摄像头数据同步 |

### 4.3 FFQueueTaskRunner — 核心串行运行器

```cpp
// Src/FFAsync/FFQueueTaskRunner.cpp
bool FFQueueTaskRunner::postTask(TaskClosure task)
{
    if (!FFTaskRunnerDelegate::checkRunnerDelegateValid(m_delegate))
        return false;

    FFTask taskObject(std::move(task));
    // 将任务追加到【同一个】 m_taskQueuePage 上
    return m_delegate->postTaskOnTaskQueuePage(
        std::move(taskObject), m_taskQueuePage);
}
```

**关键设计**：并行 TaskRunner 每次 `postTask` 都新建临时 `TaskQueuePage`，
而队列 TaskRunner 的 `m_taskQueuePage` 是持久化的（在构造函数中分配，整个 Runner 生命周期共用）。
`FFTaskQueuePageImpl` 的原子操作保证多线程安全地追加任务。

## 第 5 章 FFThreadPool — 线程池核心

### 5.1 三层线程组结构

```cpp
// Src/FFAsync/FFThreadPool.h
class FFThreadPool : public QObject, public FFTaskRunnerDelegate
{
private:
    std::unique_ptr<FFThreadGroup> m_backgroundThreadGroup;  // LowestPriority
    std::unique_ptr<FFThreadGroup> m_commonThreadGroup;      // LowPriority
    std::unique_ptr<FFThreadGroup> m_foregroundThreadGroup;  // NormalPriority
    const std::unique_ptr<FFTaskScheduler> m_taskScheduler;
};
```

### 5.2 线程组选择策略

```cpp
// Src/FFAsync/FFThreadPool.cpp
FFThreadGroup* FFThreadPool::getThreadGroupFromTaskFeature(const FFTasksFeature& feature) const
{
    // 背景线程（最低优先级）：既是低任务优先级又是低线程优先级
    if (feature.taskPriority() == kTryBestPriority
        && feature.taskThreadPriority() == kBackground)
        return m_backgroundThreadGroup.get();

    // 普通线程（次低优先级）：低任务优先级但线程优先级不要求最低
    if (feature.taskPriority() <= kNormalPriority
        && feature.taskThreadPriority() == kBackground)
        return m_commonThreadGroup.get();

    // 前台线程（高优先级）：其余所有情况（包括默认）
    return m_foregroundThreadGroup.get();
}
```

**优先级路由表**：

| TaskPriority | TaskThreadPriority | 目标线程组 |
|-------------|-------------------|-----------|
| kTryBestPriority | kBackground | BackgroundThreadGroup |
| kNormalPriority | kBackground | CommonThreadGroup |
| kNormalPriority | kNormal | ForegroundThreadGroup |
| kTimeCriticalPriority | * | ForegroundThreadGroup |
| （默认）| （默认）| ForegroundThreadGroup |

### 5.3 线程数配置（初始化时）

```cpp
// Src/FFAsync/FFAsyncInitializer.cpp
FFAsyncInitializer::FFAsyncInitializer()
{
    init();  // 创建全局 FFThreadPool 单例
    const size_t maxNumForegroundThreads =
        static_cast<size_t>(std::max(3, QThread::idealThreadCount() - 1));
    // idealThreadCount() = CPU 逻辑核数
    // 例如 8 核 CPU → 最大 7 条前台线程
    FFThreadPool::getThreadPool()->startThreadPool({maxNumForegroundThreads}, nullptr);
}
```

启动参数 `ThreadPoolInitParams`：

| 字段 | 说明 | 默认值 |
|------|------|--------|
| `maxForegroundThreadNumber` | 前台线程组最大并发数 | `max(3, idealThreadCount - 1)` |
| `maxUtilityThreadNumber` | utility（common）线程组最大并发数 | 同上 |
| `expiryTimeout` | 空闲线程超时回收时间（ms） | 30000ms（30s） |

### 5.4 postTask 的完整路径

```
FFAsync::postTask(closure, feature)
    → FFThreadPool::postTask(closure, feature)
        → new FFTaskQueuePageImpl(kParallel, feature, nullptr)  [临时页]
        → FFThreadPool::postTaskOnTaskQueuePage(task, page)
            → FFTaskScheduler::willPostTask()         [退出检查]
            → page.startAtomicOperator()
            → atomicOp.beforeAppendTask()             [原子: 是否允许加入页]
            → FFTaskScheduler::getTaskQueuePageProxy() [分配代理]
            → atomicOp.appendTask(task)               [原子: 追加任务]
            → getThreadGroupFromTaskFeature(feature)  [选线程组]
            → threadGroup.startPostTaskQueuePage({proxy, atomicOp})
                → 唤醒/创建 FFWorkerThread
                    → FFTaskScheduler::runTaskAndGetNextTask()
                        → task.run()                  [执行 closure]
                        → 检查页上是否还有下一任务
```

## 第 6 章 FFThreadGroup — 线程组调度

### 6.1 核心职责

```cpp
// Src/FFAsync/FFThreadGroup.h
class FFThreadGroup : public QObject
{
public:
    // 启动线程组（maxThreadCount 条 WorkerThread）
    void startupThreadGroup(size_t maxThreadCount,
                            size_t maxTryBestThreadCount,
                            size_t expiryTimeout,
                            QSharedPointer<FFThreadTaskRunner> threadTaskRunner,
                            FFWorkerThread::Observer* observer);

    // 投递 TaskQueuePage，唤醒/创建 WorkerThread 处理
    void startPostTaskQueuePage(TaskQueuePageProxyAtomicPair);

    // 关联/解关联当前线程到此组（用于 ThreadTaskRunner 专属线程）
    void associateCurrentThreadToGroup();
    void disAssociateCurrentThreadToGroup();
};
```

### 6.2 动态线程管理策略

- **懒创建**：不预先创建所有线程，而是按需创建（最多 `maxThreadCount` 条）
- **超时回收**：`expiryTimeout`（默认 30s）空闲后 WorkerThread 自动退出
- **`maxTryBestThreadCount`**：限制低优先级任务并发数，防止占满线程池

### 6.3 线程组与 TaskQueuePage 的关系

```
同一时刻的线程组状态示例（maxThreadCount=4）：

WorkerThread-0  → 执行 QueuePage-A（任务 t1, t2, t3...）
WorkerThread-1  → 执行 QueuePage-B（单任务）
WorkerThread-2  → 执行 QueuePage-C（单任务）
WorkerThread-3  → 空闲，等待唤醒（30s 后超时回收）

内部优先队列（TaskPagePriorityQueue）：
  [kTimeCritical] QueuePage-D
  [kTimeCritical] QueuePage-E
  [kNormal]       QueuePage-F
```

### 7.2 执行循环逻辑

WorkerThread 的核心执行逻辑（伪代码）：

```
run() 循环：

        等待 wakeUpToRunTaskQueuePage() 触发

        while (taskQueuePageProxy 有效) {
            // 委托调度器运行任务，并取下一任务

            ① 如果有新 TaskQueuePage 在等待 → 继续处理
            ② 如果空闲超过 expiryTimeout → 线程退出
    }
```

## 第 8 章 TaskQueuePage — 任务队列页

### 8.1 核心概念

`TaskQueuePage`（`FFTaskQueuePageImpl`）是 FFAsync **最核心的内部概念**，它把"任务投递"和"线程调度"解耦：

- **并行任务**：每个 `postTask` 调用产生一个新的单任务 `TaskQueuePage`，可被任意 WorkerThread 并发执行
- **串行任务**：同一个 `FFQueueTaskRunner` 持有的持久 `TaskQueuePage`，所有任务追加到同一页上，保证顺序执行

```
┌─────────────────────────────────────────────────────┐
│              FFTaskQueuePageImpl                     │
│                                                     │
│  taskBehaviorOnExit: kIgnoreUnRunTaskOnExit         │
│  taskQueuePageExecMode: kParallel / kQueue          │
│  FFTasksFeature feature                             │
│                                                     │
│  任务列表（内部队列）:                                │
│    Task₁ { closure, index, behaviorOnExit }         │
│    Task₂ { closure, index, behaviorOnExit }   ←串行 │
│    Task₃ { closure, index, behaviorOnExit }         │
│                                                     │
│  AtomicOperator: 保证多线程安全地追加/取出任务        │
│  isInProcess: 是否正在被某条 WorkerThread 处理        │
└─────────────────────────────────────────────────────┘
```

### 8.2 原子操作保证线程安全

`startAtomicOperator()` 返回 `FFTaskQueuePageAtomicOperator`，使用 CAS（Compare-And-Swap）保证：
- `beforeAppendTask()`：原子地检查"页是否已在被处理"，返回 `isAllowedPostQueuePage`
- `appendTask(task)`：原子地将任务追加到队列
- `takeTask()`：原子地取出下一个待执行任务

这套原子机制使得**生产者（任何线程 postTask）和消费者（WorkerThread）无需互斥锁**即可安全操作同一个 TaskQueuePage。

## 第 9 章 FFTaskScheduler — 任务调度器

### 9.1 职责

`FFTaskScheduler` 是 FFAsync 中**退出控制的核心**，承担四大职责：
1. 确认任务是否还能被投递（`willPostTask`）
2. 确认任务是否还能被运行（`beforeRunTask`）
3. 实际运行任务并推进队列页（`runTaskAndGetNextTask`）
4. 管理优雅退出的阻塞任务计数（`stateFlag` 原子变量）

### 9.2 stateFlag — 原子位图设计

```cpp
// Src/FFAsync/FFTaskScheduler.cpp

// bit 0     = ExitFlag：1 表示 startExit() 已被调用
// bit 1..31 = BlockingTaskCount：当前正在执行或已投递的"阻塞退出"任务数量

constexpr quint32 kExitFlagOffset = 1;
constexpr quint32 kExitFlagValue  = 1;                   // bit 0 = 1
constexpr quint32 kAppendBlockTaskNumStepValue = 1 << 1; // bit 1 = 1（每个阻塞任务计数步长）
```

**原子操作组合**：

| 操作 | stateFlag 变化 | 语义 |
|------|---------------|------|
| `startExit()` | fetch_add(kExitFlagValue) | 置位 bit0（标记退出已开始）|
| `appendBlockingTaskNum()` | fetch_add(kAppendBlockTaskNumStep) | 阻塞任务数 +1 |
| `reduceBlockingTaskNum()` | fetch_sub(kAppendBlockTaskNumStep) | 阻塞任务数 -1 |
| `hasStartedExit()` | load() & kExitFlagValue | 读 bit0 |
| `hasBlockingTaskNum()` | load() >> kExitFlagOffset != 0 | 读高 31 位 |

### 9.3 退出时三种任务行为的运行逻辑

```cpp
// Src/FFAsync/FFTaskScheduler.cpp
bool FFTaskScheduler::beforeRunTask(TaskBehaviorOnExit behavior)
{
    switch (behavior) {
    case kIgnoreTaskOnExit:
        return !hasStartedExit();    // 退出后一律不运行

    case kIgnoreUnRunTaskOnExit:
        // 运行中的不中断（appendBlockingTaskNum 计数），
        // 但如果退出已开始此时才尝试运行的任务会被丢弃
        if (appendBlockingTaskNum() /*已退出*/) {
            reduceBlockingTaskNum();
            return false;
        }
        return true;

    case kMustExecTaskOnExit:
        // 无论如何都运行（计数已在 getTaskQueuePageProxy 时预加）
        return true;
    }
}
```

## 第 10 章 postReplyableResultTask — 带回复的异步任务

### 10.1 完整实现（FFAsync.inl）

```cpp
template<class F>
std::enable_if_t<!std::is_same_v<void, TaskResult_t<F>>, TaskPromiseResult_t<F>>
FFAsync::postReplyableResultTask(F&& task, FFTasksFeature taskFeature)
{
    // 警告：调用方必须在有 event loop 的线程（主线程），否则 then() 无法触发
    if (!checkPostTaskThreadIsValid())
        qWarning() << "post task thread must have an event loop...";

    using TaskResultType    = TaskResult_t<F>;       // 任务实际返回类型
    using TaskRawResultType = TaskRawResult_t<F>;    // 去掉引用后的类型

    // ① 在堆上分配结果槽（生命周期由 AutoReleaseWrapper 管理）
    auto result = new std::unique_ptr<TaskRawResultType>();

    // ② 包装原始任务：后台线程执行时，把结果写入 result 槽
    TaskClosure newTask = std::bind([](auto t, std::unique_ptr<TaskRawResultType>* res) {
        res->reset(new TaskRawResultType(t()));   // 执行任务，保存结果
    }, TaskRunType(std::forward<F>(task)), result);

    // ③ 构造 QPromise<T>，内部立即 postTask 到线程池
    return QtPromise::QPromise<TaskResultType>(
        [newTask, result, tf = std::move(taskFeature)]
        (const QPromiseResolve<TaskResultType>& resolve,
         const QPromiseReject<TaskResultType>&  reject)
        {
            FFAsync::postTask([newTask, result, resolve, reject,
                               currentThread = QPointer(QThread::currentThread())]() {
                newTask();   // 后台执行
                // 使用 AutoReleaseWrapper 确保 result 在任意路径下都被释放
                TaskInvoker<TaskRawResultType>::invoke(
                    buildAutoRelease(result).get(), resolve, reject, currentThread);
                // invoke 内部调用 resolve(result) → 触发主线程 then() 回调
            }, std::move(tf));
        });
}
```

### 10.2 TaskInvoker — 处理 Promise 嵌套

```cpp
// FFAsync.inl
template<class T>
struct TaskInvoker {
    // 普通类型：直接 resolve
    static void invoke(auto* result, auto resolve, auto reject, auto thread) {
        resolve(std::move(**result));
    }
};

// 特化：任务返回值本身是 QPromise<T>（任务内部又做了异步）
template<class T>
struct TaskInvoker<QtPromise::QPromise<T>> {
    static void invoke(auto* result, auto resolve, auto reject, auto thread) {
        const auto& innerPromise = std::move(**result);
        if (innerPromise.isFulfilled()) {
            resolve(innerPromise.data()->value());
        } else if (innerPromise.isRejected()) {
            reject(innerPromise.data()->error());
        } else {
            // Promise 还未完成，回到调用线程等待
            qtpromise_defer_f([=]() {
                innerPromise.then(
                    [=]() { resolve(innerPromise.data()->value()); },
                    [=]() { reject(innerPromise.data()->error()); });
            }, thread);
        }
    }
};
```

### 10.3 线程安全回调：QPointer + event loop

```cpp
// 关键：capture currentThread = QPointer(QThread::currentThread())
```

## 第 13 章 优雅退出机制

### 13.1 shutDown 调用链

```
~FFAsyncInitializer()
    → FFThreadPool::shutDown()
        → FFTaskScheduler::startExit()      ← 置位 ExitFlag（bit0=1）
        → FFThreadGroup[FG]::shutDown()     ← 通知 WorkerThread 停止等待
        → FFThreadGroup[Common]::shutDown()
        → FFThreadGroup[BG]::shutDown()
        → FFTaskScheduler::waitExitComplete() ← 阻塞，直到 BlockingTaskCount=0
```

### 13.2 阻塞任务计数的精确控制

```
场景：应用退出时有 2 个 kMustExecTaskOnExit 任务正在执行

退出流程：
  startExit() 调用后：
    stateFlag = 0b0000_0101   （bit0=1 退出，bit1..31 = 2 个阻塞任务）

  任务 A 完成 → afterRunTask() → reduceBlockingTaskNum()
    stateFlag = 0b0000_0011   （blockCount=1）

  任务 B 完成 → afterRunTask() → reduceBlockingTaskNum()
    stateFlag = 0b0000_0001   （blockCount=0，ExitFlag=1）
    → exitCondition->wakeOne()   ← 解除 waitExitComplete() 阻塞

  主线程：waitExitComplete() 返回 → 安全退出
```

### 13.3 三种行为在退出时的表现

| 行为 | willPostTask 返回 | beforeRunTask 返回 | 计入阻塞数 |
|------|------------------|-------------------|-----------|
| kIgnoreTaskOnExit | 退出后返回 false | 退出后返回 false | 否 |
| kIgnoreUnRunTaskOnExit | 退出后返回 false | 运行中不中断，新任务 false | 是（运行时加）|
| kMustExecTaskOnExit | 退出后依然返回 true | 永远返回 true | 是（投递时预加）|

## 第 16 章 面试题精选

### Q1：FFAsync 的 postTask 和 Qt 的 QtConcurrent::run 有什么本质区别？

**A：**
1. **优先级调度**：`QtConcurrent::run` 使用全局 `QThreadPool`，无优先级区分；`postTask` 支持 `kTimeCriticalPriority / kNormalPriority / kTryBestPriority` 三档，通过三个 `FFThreadGroup` 物理隔离不同优先级的任务，高优先级任务不会被低优先级任务"饿死"。

2. **序列化能力**：`QtConcurrent::run` 无法保证多个 `run` 调用的执行顺序；`createQueueTaskRunner()` 让多个任务严格按投递顺序串行执行，且无需外部加锁。

3. **回调线程**：`QtConcurrent::run` 返回 `QFuture`，回调需要手动用 `QFutureWatcher` + 信号槽；`postReplyableResultTask` 返回 `QPromise<T>`，`.then()` 自动在调用线程（主线程）执行，无需手动分发。

4. **优雅退出**：`QThreadPool::waitForDone()` 要等所有任务完成；FFAsync 提供三种退出行为（Ignore/IgnoreUnRun/MustExec），可精细控制哪些任务在关闭时必须完成。

---

### Q2：postReplyableResultTask 中 `buildAutoRelease(result)` 的作用是什么？

**A：**
`result` 是在堆上分配的 `new std::unique_ptr<TaskRawResultType>()`，在 `postTask` 的 closure 中被写入任务结果。
整个异步过程有两个路径都需要能释放这个堆内存：
- **成功路径**：`TaskInvoker::invoke` 消费结果后，`buildAutoRelease(result)` 创建 `AutoReleaseWrapper<unique_ptr>`，析构时自动 `delete result`
- **失败/拒绝路径**：同样走到 `AutoReleaseWrapper` 的析构

不用 `AutoReleaseWrapper` 直接用 `unique_ptr` 的原因：closure 中的 lambda 需要 `std::function`，而 `std::function` 要求内部存储的类型可复制；`unique_ptr` 不可复制，所以用裸指针 + `AutoReleaseWrapper` 在执行后统一回收。

---

### Q3：FFQueueTaskRunner 如何保证多线程安全地往同一 TaskQueuePage 追加任务？

**A：**
核心是 `FFTaskQueuePageImpl::startAtomicOperator()` + `beforeAppendTask()` 组合：

```
线程 A（postTask t₁）:
    atomicOp = page.startAtomicOperator()
    isAllowedPost = atomicOp.beforeAppendTask()  // CAS 检查+设置 isInProcess
    if (isAllowedPost):
        proxy = scheduler.getTaskQueuePageProxy(page)  // 标记页将被处理
    atomicOp.appendTask(t₁)  // 原子追加

线程 B（postTask t₂，并发）：
    atomicOp = page.startAtomicOperator()
    isAllowedPost = atomicOp.beforeAppendTask()  // 此时页已 isInProcess=true → 返回 false
    atomicOp.appendTask(t₂)  // 仍然原子追加到队列
    // proxy 为空，不再通知线程组（因为 WorkerThread 正在/即将处理该页）
    // WorkerThread 在完成 t₁ 后会检查页上是否还有下一个任务（t₂），继续处理

WorkerThread:
    after run t₁:
    taskProcessed() → true（t₂ 还在页上）→ 继续处理 t₂
    taskProcessed() → false（页为空）→ 返回空 proxy，线程回到空闲
```

无互斥锁，全程使用 CAS（原子操作），避免了锁竞争和线程阻塞。

### Q4：FFAsync 如何保证 then() 回调一定在主线程执行？

**A：**
在 `postReplyableResultTask` 的 closure（运行在后台线程）中，捕获了：
```cpp
currentThread = QPointer(QThread::currentThread())
// currentThread 指向调用 postReplyableResultTask 的那条线程（通常是主线程）
```

后台线程执行完任务后调用 `resolve(result)`。`QtPromise` 的 `resolve` 实现内部使用 `QtPromisePrivate::qtpromise_defer`，通过 `QMetaObject::invokeMethod(..., Qt::QueuedConnection)` 将 then 回调投递到 `currentThread` 的事件队列。因此只要调用方线程有 Qt event loop（主线程默认有），then() 就一定在那条线程执行。

如果调用方线程没有 event loop（例如在 WorkerThread 中调用），则会触发：
```cpp
qWarning() << "post task thread must have an event loop, otherwise the task reply will not be received";
```

### Q5：TaskBehaviorOnExit::kMustExecTaskOnExit 在 stateFlag 中的计数为什么要"预加"（在投递时而非执行时加）？

**A：**
考虑如下竞态场景：
```
线程 A：FFTaskScheduler::willPostTask() 返回 true（退出未开始）
主线程：startExit() 置位 ExitFlag，检查 hasBlockingTaskNum() = false，
        立即唤醒 exitCondition（认为所有阻塞任务已完成！）
线程 A：appendBlockingTaskNum()（为时已晚，exitCondition 已被唤醒）
```

如果在**执行时**才加计数，上述竞态会导致程序在任务未完成时就退出。
**预加**的做法：
- `getTaskQueuePageProxy()` 时（任务投递成功确认后、WorkerThread 处理前）就调用 `appendBlockingTaskNum()`
- `startExit()` 内部加锁后检查 `hasBlockingTaskNum()`，只有计数为 0 才唤醒条件变量
- 两者都有锁保护，消除了竞态

### Q6：项目中 FFQueueTaskRunner 和 FFThreadTaskRunner 的使用场景有什么区别？为什么 AI 结果处理用 Queue 而不用 Thread？

**A：**

| | FFQueueTaskRunner | FFThreadTaskRunner |
|-|------------------|--------------------|
| 线程来源 | 从共享线程池借用 | 专属独立线程（不共享）|
| 空闲时 | 不占用任何线程 | 线程一直存在（直到 Runner 销毁）|
| 适合场景 | 任务较稀疏、不需要线程亲和性 | 任务密集且有线程本地状态（TLS）|

AI 结果处理（`FAILongToShortResultListItem`）选择 `createQueueTaskRunner` 的原因：
1. AI 结果不是连续高频的，用完即释放，不需要常驻线程
2. 只需要"串行"保证，不需要"同一条线程"的线程亲和性
3. `QueueTaskRunner` 空闲时不占用线程资源，更节约

`createThreadTaskRunner` 适合**播放同步**等场景：需要固定线程处理连续帧数据，有线程本地的解码状态，且任务密集到几乎从不空闲。
