# 时间线性能优化详解

> 模块：Filmora Windows 桌面端 — `FTimelineView`
> 技术栈：C++17 + Qt 5.15 + QGraphicsView Framework
> 关键源文件：`Src/FTimelineView/FGraphicsClipPainter.h`、`FGraphicsType.h`、`FGraphicsTrackItem.h`

---

## 优化总览

| # | 优化项 | 问题根源 | 方案 | 关键类/宏 | 效果 |
|---|--------|---------|------|---------|------|
| 1 | PaintCache（绘制缓存） | 每帧重复计算字体宽度/构造 QFont | 缓存帧状态，`isSame()` 判断是否复用 | `FTimelinePaintCache`、`IFFPaintCache` | 单帧绘制 -30% |
| 2 | FreezeMode（冻结模式） | 拖拽时所有 Clip 全量重绘 | 拖拽期间非活跃 Clip 直接输出 QImage 缓存 | `bNeedForceUpdate`、`sigSubstituteStart/End` | 拖拽帧率 +50% |
| 3 | 轨道高度缓存 | 展开/折叠时所有轨道反复重算高度 | 懒计算 + 显式 `invalidate()` 失效 | `FGraphicsTrackHeightCache`、`FGraphicsClipItemLayoutCache` | 滚动流畅 |
| 4 | 批量更新合并 | N 个 Clip 操作触发 N 次 Observer 回调 | RAII 宏暂停事件，作用域结束发一次 BATCH | `FF_DECLARE_PAUSE_EVENT_COMMANDER` | 批量操作 -95% 回调 |
| 5 | 缩略图异步加载 | 主线程同步解码视频帧阻塞 UI | 线程池异步解码，`QueuedConnection` 回传主线程 | `IFFClipVideoThumbnailService`、`FThumbnailInfo` | 打开工程无卡顿 |
| 6 | Substitute 拖拽 | 拖拽时实时修改数据触发全链路回调 | 拖拽期间只移动视觉副本，松手后一次性提交 | `FGraphicsClipItemSubstitute` | 拖拽完全流畅 |
| 7 | Painter 选项掩码 | 小尺寸 Clip 仍尝试绘制缩略图/波形 | 位标志按需裁剪绘制项 | `FClipPainterOptions`、`cpoSmall`/`cpoDEFAULT` | 细小 Clip 绘制 -60% |

## 一、PaintCache — Clip 绘制状态缓存

### 1.1 背景

`FGraphicsClipItem::paint()` 每次被 Qt 调用时，需要绘制：背景色、边框、缩略图、文字名称、图标、关键帧菱形等元素。其中开销最大的几项：

- `QFontMetrics::horizontalAdvance(name)` — 计算文字显示宽度（每次调用约 5~15μs）
- `QFont` 对象构造 — 在高 DPI 下会触发字体渲染匹配
- `QString` clip 名称拼接与处理
- 图标状态判断（涉及多个 VBL 接口调用）

**典型问题场景**：工程中有 50 个 Clip，每次时间线缩放/滚动都触发所有可见 Clip 重绘。若每次都全量计算，单帧绘制耗时约 8~15ms，在 60fps 的预算（16.6ms/帧）内几乎没有余量。

### 1.2 优化方案

引入 `FTimelinePaintCache`（实现 `IFFPaintCache` 接口），将 Clip 的绘制状态序列化为可比较的结构体。每帧绘制时先检查状态是否变化，未变化则直接复用上次绘制的 `QImage`。

```cpp
// 定义于 Src/FTimelineView/FGraphicsType.h
struct FTimelinePaintCache : IFFPaintCache {
    QRectF    region;           // 缓存 QImage 的绘制区域
    QImage    cacheImage;       // 上一帧的像素图（直接 drawImage 输出）
    QRectF    boundingRect;     // Clip 的完整 bounding rect

    // ── 以下字段用于 isSame() 比较，任意字段变化则缓存失效 ──
    bool      bNeedForceUpdate = false; // 强制更新标志（缩略图加载完成后置 true）
    int       nTrackHeight;            // 轨道高度（影响缩略图行数）
    qreal     dPixPerFrame;            // 当前缩放比（影响缩略图密度）
    qlonglong nStartFrame;             // Clip 在时间线上的可见起始帧
    qlonglong nEndFrame;               // 可见结束帧
    qlonglong nTrimIn;                 // 裁剪入点
    qlonglong nTrimOut;                // 裁剪出点
    bool      bChecked;                // 是否选中
    bool      bSelected;
    bool      bIsMute;                 // 是否静音
    int       nVideoIconState;         // 视频特效图标状态（枚举值）
    int       nAudioIconState;
    int       nKeyFrameIconState;
    QString   strClipName;             // 片段名称（多语言/重命名后变化）
    FFSkin::SkinType skinType;         // 皮肤类型（深色/浅色模式）
    FFLocale::FFLanguageType oLanguageType; // 界面语言（影响文字渲染）

    bool isSame(const IFFPaintCache* ohs) const override; // 逐字段比较
    void build(const FTimelinePaintOption& option,
               IFGraphicsItemPainter* painter,
               FGraphicsTimelineScene* scene) override;   // 从当前状态构建
};
```

### 1.3 技术实现

**整体流程**：

```
FGraphicsBaseClipPainter::paint(option)
    │
    ├── 1. buildCache(option)
    │       从 ClipItem 当前状态（帧位置、缩放比、名称、图标状态...）
    │       构造一个新的 FTimelinePaintCache 对象
    │
    ├── 2. newCache->isSame(m_paintCache) ?
    │
    │   ┌─ YES（状态未变化）────────────────────────────────────────┐
    │   │  option.painter->drawImage(m_paintCache->region,          │
    │   │                            m_paintCache->cacheImage)      │ ← 快路径
    │   │  delete newCache;  return;   // 跳过所有绘制计算           │
    │   └────────────────────────────────────────────────────────────┘
    │
    └── NO（状态变化）
            paintWithCache(option, newCache)
                │  离屏绘制所有元素到 newCache->cacheImage：
                │  drawBackground() → drawBorder() → drawThumbnail()
                │  → drawText() → drawIcons() → drawKeyFrames()
                │
            option.painter->drawImage(newCache->region, newCache->cacheImage)
            delete m_paintCache;
            m_paintCache = newCache;   // 更新缓存
```

**缓存的持有位置**：`FGraphicsAbstractClipItem` 持有 `IFFPaintCache* m_pPaintChache`，供各 Painter 子类读写：

```cpp
// Src/FTimelineView/FGraphicsAbstractClipItem.h
class FGraphicsAbstractClipItem : public QGraphicsRectItem {
    IFFPaintCache* paintCache() const;
    void updatePaintCache(IFFPaintCache* paintCache);
    void clearPaintCache();   // 数据变化时调用，强制下次全量重绘
};
```

**缓存失效时机**：

| 触发事件 | 失效方式 |
|---------|---------|
| Clip 名称被修改 | `clearPaintCache()` → 下次 `isSame()` 返回 false |
| 视频特效开关变化 | 同上 |
| 缩略图异步加载完成 | `cache->bNeedForceUpdate = true`（不清空缓存，仅标记强制更新） |
| 时间线缩放级别变化 | `dPixPerFrame` 字段变化 → `isSame()` 自动检测 |
| 皮肤/语言切换 | `skinType`/`oLanguageType` 字段变化 → `isSame()` 自动检测 |

### 1.4 未来优化空间

1. **分区域缓存**：当前 `cacheImage` 覆盖整个 Clip。若只有右上角图标状态变化（如特效开关），可只局部更新图标区域，缩略图区域完全复用，避免缩略图重绘。
2. **GPU 纹理缓存**：将 `cacheImage`（CPU QImage）改为 `QOpenGLTexture`，在 GPU 侧缓存，减少每帧 CPU→GPU 数据传输带宽。
3. **多级 LOD 缓存**：在缩放过渡动画期间，预生成 2~3 个不同分辨率的 `cacheImage`，切换缩放级别时直接使用对应分辨率，避免中间帧重绘。

## 二、FreezeMode — 拖拽冻结绘制模式

### 2.1 背景

用户拖拽 Clip 时，`FGraphicsClipItemSubstitute`（替身 Item）每帧都在移动位置，Qt 的脏区域传播机制会将整个 Scene 可视区域标脏，导致所有可见 Clip 都需要调用 `paint()`。

**量化问题**：
- 30 个可见 Clip × 每帧 `paint()` = 30 次绘制调用
- 即使开启了 PaintCache，`buildCache()` + `isSame()` 本身也有约 0.5ms/个 的开销
- 30 × 0.5ms = 15ms，已接近 16.6ms 的帧预算上限
- 低配机（i5 + 核显）上实测帧率跌至 20~30fps，用户感知明显卡顿

### 2.2 优化方案

引入"冻结模式（FreezeMode）"：拖拽开始时，所有**非活跃** Clip（即不是正在被拖拽的那个）进入冻结状态，每次 `paint()` 直接输出上一帧的 `cacheImage`，**完全跳过** `buildCache()` 和 `isSame()` 的计算。

实现方式：利用 `FTimelinePaintCache::bNeedForceUpdate` 字段进行控制。拖拽期间将其保持为 `false`，使得 `isSame()` 比较时**永远返回 true**，从而走缓存快路径。

```cpp
struct FTimelinePaintCache : IFFPaintCache {
    bool bNeedForceUpdate = false;
    // bNeedForceUpdate = false + 状态字段未变化 → isSame() = true → 走缓存
    // bNeedForceUpdate = true                  → 强制重绘（缩略图就绪时使用）
};
```

### 2.3 技术实现

**拖拽生命周期中的冻结控制**：

```
拖拽开始
    │
    ├── FGraphicsItemSubstituteManager::startClipMove()
    │       创建 SubstituteItem，启动替身 Action
    │
    ├── FGraphicsTimelineScene 发出 sigSubstituteStart()
    │
    └── 所有 ClipItem 的 PaintCache->bNeedForceUpdate 保持 false
        （新建的 Cache 也默认 false，确保拖拽期间不触发强制重绘）

拖拽中（每帧 ~16ms）
    │
    ├── SubstituteItem::setPos(newPos) → 仅替身区域标脏
    │
    ├── 其他 ClipItem::paint() 被调用（Qt 整帧刷新）
    │       buildCache(option) → newCache（bNeedForceUpdate=false）
    │       newCache->isSame(m_paintCache)?
    │           ✓ 帧位置/名称/图标未变 → isSame() = true
    │           → drawImage(m_paintCache->cacheImage)  ← 0.1ms
    │           delete newCache;  return;
    │
    └── 实测：30 个冻结 Clip 总绘制耗时 < 3ms

拖拽结束（鼠标释放）
    │
    ├── Handler::commit() → IFFTimeline::moveClip()
    │       → onClipChanged(MOVE) → 被移动 ClipItem::clearPaintCache()
    │           （清空缓存，强制下次全量重绘新位置）
    │
    └── FGraphicsTimelineScene 发出 sigSubstituteEnd()
        → 销毁 SubstituteItem
        → 场景正常刷新，所有 Clip 重新绘制一帧
```

**核心绘制方法** `paintWithCacheOpt()`（简化逻辑）：

```cpp
void FGraphicsBaseClipPainter::paintWithCacheOpt(
    const FTimelinePaintOption& option, FTimelinePaintCache* cache)
{
    FTimelinePaintCache* newCache = buildCache(option);

    bool forceUpdate   = newCache->bNeedForceUpdate;
    bool stateChanged  = !newCache->isSame(m_paintCache);

    if (!forceUpdate && !stateChanged) {
        // ── 快路径：直接复用 cacheImage ──
        option.painter->drawImage(m_paintCache->region, m_paintCache->cacheImage);
        delete newCache;
        return;
    }

    // ── 慢路径：重新全量绘制 ──
    paintWithCache(option, newCache);  // 离屏绘制到 newCache->cacheImage
    option.painter->drawImage(newCache->region, newCache->cacheImage);

    delete m_paintCache;
    m_paintCache = newCache;
}
```

### 2.4 未来优化空间

1. **增量脏区追踪**：目前 SubstituteItem 移动导致整个视口标脏。可改为只将替身所在区域标脏，其他区域完全不触发 `paint()`（需要自定义 `QGraphicsScene::drawBackground` 和脏区合并逻辑）。
2. **帧预算分级**：设置每帧最多重绘 N 个 Clip，超出的排队到下一帧。结合视口可见性排序，优先重绘用户视线中心的 Clip。
3. **替身独立图层**：将所有 SubstituteItem 统一放在一个独立的 `QGraphicsItemGroup`（最高 Z 值），拖拽时只让该图层参与重绘，主轨道层整帧冻结，不参与 `drawItems()` 调用。

## 三、FGraphicsTrackHeightCache — 轨道高度懒计算

### 3.1 背景

轨道高度并非固定值，以下场景都会导致高度变化：

- 用户点击 ▼ 展开关键帧曲线面板（`FGraphicsKeyFramePanelItem` 展开 → 轨道变高约 120px）
- 用户上下拖动轨道分隔线手动调整高度
- AI 任务完成（如运动追踪分析完成），Clip 上出现新的关键帧行
- 工程加载时批量初始化所有轨道

**问题根源**：时间线轨道是**从下到上堆叠**排列（`bottomToTop=true`），任意一条轨道高度变化都会导致其**上方所有轨道**的 Y 坐标需要重新计算。若有 20 条轨道，展开最下面的关键帧面板会触发 20 次高度查询 + 20 次 `setPos()` 更新。

若每次高度变化都立即同步重算，在批量操作（如加载工程时同时初始化 20 条轨道）下，复杂度为 O(N²)。

### 3.2 优化方案

采用**懒计算 + 显式失效（Lazy Evaluation + Cache Invalidation）**模式：

```cpp
// Src/FTimelineView/FGraphicsTrackItem.h
class FGraphicsTrackHeightCache : public QObject {
public:
    int  height() const;   // 惰性计算：首次访问时才真正计算，结果写入缓存
    void invalidate();     // 显式失效：仅标记为脏，不立即计算

private:
    QScopedPointer<FGraphicsTrackHeightCachePrivate> d;  // Pimpl 隐藏实现细节
};
```

`FGraphicsClipItemLayoutCache` 负责管理每个 ClipItem 的布局脏状态：

```cpp
// Src/FTimelineView/FGraphicsType.h
struct FGraphicsClipItemLayoutCache {
    bool           needReLayout;  // 脏标记：true = 需要重新计算 setPos
    QRectF         cacheRect;     // 上次计算的 boundingRect 缓存
    QGraphicsItem* clipItem;

    void setCacheRect(QRectF rect) {
        cacheRect = rect;
        if (clipItem && clipItem->isVisible())
            needReLayout = false;  // 完成布局后清除脏标记
    }
};
```

### 3.3 技术实现

**展开关键帧面板的完整流程**：

```
用户点击 ▼ 展开关键帧面板
    │
    ├── FGraphicsKeyFramePanelItem::setVisible(true)
    │       KeyFramePanel 尺寸变化
    │
    ├── FGraphicsTimelineTrackItem::adjustHeight(clip)
    │       m_pHeightCache->invalidate()   ← 只标记失效，O(1)，不计算
    │
    ├── FGraphicsTimelineTrackItem::setClipsNeedLayout()
    │       遍历所有 ClipItem，设置 layoutCache->needReLayout = true
    │
    └── FGraphicsTimelineScene::updateTrackHeight()   ← 一次性重排入口
              │
              ├── 遍历所有 TrackItem（从下到上）
              │       trackY = startY
              │       for each trackItem:
              │           int h = trackItem->height()
              │                       │
              │                       └── FGraphicsTrackHeightCache::height()
              │                               若有效 → 直接返回缓存值   ← O(1)
              │                               若失效 → 重新计算:
              │                                   = baseHeight
              │                                   + (kfPanelVisible ? kfPanelHeight : 0)
              │                                   + expandedRows * rowHeight
              │                                   → 写入缓存，标记有效
              │           trackItem->setPos(0, trackY)
              │           trackY += h
              │
              └── Scene::update()  ← 一次性重绘

总结：invalidate() O(1) + height() 按需计算 + 统一重排 = O(N) 而非 O(N²)
```

**ClipItem 布局脏标记的协作流程**：

```
数据变化（Clip 增删、帧位置变化）
    │
    ├── IFFTimeline → onClipChanged(mode)
    │
    ├── FGraphicsTimelineTrackItem::clipChangedEvent()
    │       setClipsNeedLayout()
    │           → 所有 ClipItem::m_pLayoutCache->needReLayout = true
    │
    └── 下一帧 ClipItem::paint() 时检测：
            if (m_pLayoutCache->needReLayout) {
                QRectF newRect = calcBoundingRect();
                setPos(scene->frameToScenePos(clip->startFrame()), trackY);
                m_pLayoutCache->setCacheRect(newRect);  // 清除脏标记
            }
```

### 3.4 未来优化空间

1. **差量重排**：当某条轨道高度变化时，只重排该轨道**上方**的所有轨道 Y 坐标（下方轨道不受影响，`bottomToTop` 布局下上方轨道才需要平移）。当变化轨道接近底部时，可减少 80% 的重排计算。
2. **批量高度变化合并**：结合 `FF_DECLARE_PAUSE_EVENT_COMMANDER`，加载工程时所有轨道高度初始化完成后统一触发一次 `updateTrackHeight()`，而非每条轨道逐一触发。
3. **轨道虚拟化（Virtual Scroll）**：当轨道数量 > 50 时（如复杂多轨工程），只创建视口内可见的 TrackItem，超出视口的轨道不在 Scene 中创建图形项，按需创建/销毁，将内存和计算量从 O(N) 降至 O(可见轨道数)。

## 四、FF_DECLARE_PAUSE_EVENT_COMMANDER — 批量更新合并

### 4.1 背景

时间线中的批量操作（全选删除、批量移动、粘贴多个 Clip）会循环调用多次 VBL 层数据接口。每次调用都会触发完整的 Observer 回调链：

```
removeClip(clip1) → VBL 内部更新
    → Observer::onClipChanged(REMOVE)
        → FGraphicsTimelineTrackItem::clipChangedEvent(REMOVE)
            → removeClipItem(clip1)     // 从 Scene 删除图形项
            → Scene::update()           // 触发重绘
            → updateTrackHeight()       // 轨道重排

removeClip(clip2) → （重复以上完整流程）
...
removeClip(clip10) → （重复第 10 次）
```

**量化问题**（删除 10 个 Clip，低配机 i5）：
- 10 次 `updateTrackHeight()` 各约 5ms → 50ms
- 10 次 `Scene::update()` 各约 3ms → 30ms
- 合计约 80ms，界面有明显"逐个消失"的闪烁感

### 4.2 优化方案

利用 **RAII 宏** 在作用域开始时暂停 VBL 的事件广播，所有数据变更在 VBL 内部积压，作用域结束时（析构时）一次性发出 `BATCH` 事件，Scene 端只处理一次。

```cpp
// 实际使用示例（FTimelinePresenter 内部）
void FTimelinePresenter::deleteSelectedClips()
{
    // 宏 1：将作用域内所有操作合并为一条撤销记录
    FF_DECLARE_UNDO_COMMANDER(m_pUndoService, "Delete Selected Clips");

    // 宏 2：暂停 VBL 事件广播（RAII，析构时自动恢复）
    FF_DECLARE_PAUSE_EVENT_COMMANDER(m_pUndoService);

    for (auto clip : m_pClipSelector->selectedClips()) {
        m_pTimeline->removeClip(clip);  // 数据更新，不触发 Observer 回调
    }

    // ── 此处作用域结束 ──
    // FF_DECLARE_PAUSE_EVENT_COMMANDER 析构
    //   → VBL 恢复事件广播
    //   → 发出一次 FFDataChangedMode::BATCH 事件
    //   → Scene 收到 BATCH → 一次性增量同步所有变化
}
```

### 4.3 技术实现

**RAII 宏的实现原理**（简化等价代码）：

```cpp
// FF_DECLARE_PAUSE_EVENT_COMMANDER(service) 大致等价于：
struct FFPauseEventCommander {
    IFFUndoRedoService* m_service;

    explicit FFPauseEventCommander(IFFUndoRedoService* s) : m_service(s) {
        if (m_service) m_service->pauseEventBroadcast();   // 构造：暂停
    }
    ~FFPauseEventCommander() {
        if (m_service) m_service->resumeEventBroadcast();  // 析构：恢复
    }

    // 禁止拷贝，防止意外延长生命周期
    FFPauseEventCommander(const FFPauseEventCommander&) = delete;
    FFPauseEventCommander& operator=(const FFPauseEventCommander&) = delete;
};

#define FF_DECLARE_PAUSE_EVENT_COMMANDER(service) \
    FFPauseEventCommander _pauseEvtCmd_##__LINE__(service)
```

**BATCH 事件在 Scene 端的处理**：

```cpp
void FGraphicsTimelineTrackItem::clipChangedEvent(
    IFFClip* clip, FFDataChangedMode mode)
{
    switch (mode) {
    case ADD:
        addClipItem(clip);           // 精确：只新增这一个
        break;
    case REMOVE:
        removeClipItem(clip);        // 精确：只移除这一个
        break;
    case MOVE:
    case UPDATE:
        updateClipItemPos(clip);     // 精确：只更新这一个
        break;
    case BATCH:
        // 全量增量同步：对比数据层 vs 当前 ClipItem
        syncClipItems();             // 新增缺少的 + 移除多余的
        updateAllPositions();        // 一次性重新布局所有 ClipItem
        break;
    }

    // 所有模式最终触发一次重排和重绘
    updateTrackLayout();
    scene()->update();
}
```

**两个 RAII 宏的协作关系**：

```
FF_DECLARE_UNDO_COMMANDER     — 负责撤销/重做记录合并
FF_DECLARE_PAUSE_EVENT_COMMANDER — 负责 UI 回调合并

两者共享同一个 IFFUndoRedoService*，但功能独立：
- UNDO_COMMANDER：保证 N 次操作可以被一次 Ctrl+Z 撤销
- PAUSE_EVENT：保证 N 次操作只触发一次 UI 刷新

通常成对使用，单独使用其中一个也合法。
```

### 4.4 未来优化空间

1. **细粒度 BATCH 分类**：当前 BATCH 事件不携带变化列表，Scene 需要全量对比（O(N)）。可扩展为 `BATCH_ADD`、`BATCH_REMOVE`、`BATCH_MOVE` 并携带受影响的 Clip 列表，实现精确增量更新（O(变化数)）。
2. **UI 更新节流（Throttle）**：即使没有 PAUSE_EVENT，可在 Scene 侧加一个 16ms 节流定时器，将 16ms 内的多次 `update()` 请求合并为每帧只调用一次 `QGraphicsScene::update()`，彻底消除不必要的中间帧渲染。
3. **撤销合并粒度优化**：`FF_DECLARE_UNDO_COMMANDER` 合并粒度是单个函数调用栈。对于跨越多个函数的连续操作（如拖拽过程中的多次 trim），研究是否支持显式命名合并组，将相关操作聚合为一条语义明确的撤销记录。

## 五、缩略图异步加载

### 5.1 背景

打开一个含有大量视频 Clip 的工程时，每个 Clip 需要从视频文件解码若干帧作为时间线上的缩略图预览。

**耗时分析**（一个 5 秒 Clip，默认缩放级别约需 5 帧缩略图）：

| 操作 | 单次耗时 |
|------|---------|
| H.264 视频帧解码（1080p） | 20~60ms |
| HEVC/H.265 视频帧解码 | 40~100ms |
| 30 个 Clip × 5 帧 × 50ms | **7500ms ≈ 7.5 秒** |

若在主线程同步解码，打开中等规模工程会导致 UI 完全冻结 3~8 秒，用户体验极差。

### 5.2 优化方案

通过 `IFFClipVideoThumbnailService`（纯虚接口，VBL 层实现）将解码任务派发到线程池：

- **主线程**：绘制时查询缓存，若帧未就绪则绘制灰色占位块，同时发起异步请求
- **后台线程**：解码视频帧，结果存入内部缓存
- **回调**：解码完成后通过 `Qt::QueuedConnection` 信号通知主线程，主线程触发局部重绘

### 5.3 技术实现

**绘制时的异步查询**：

```cpp
// Src/FTimelineView/FGraphicsClipPainter.h (FGraphicsBaseClipPainter)
bool FGraphicsBaseClipPainter::drawThumbnail(
    FGraphicsClipItem* clipItem, QPainter* painter, const QRectF& boundingRect)
{
    // buildVideoThumbnailInfo 计算需要哪些帧，并从 ThumbnailService 查询已缓存的帧
    FThumbnailInfo info = buildVideoThumbnailInfo(clipItem, boundingRect);

    if (!info.isValid()) return false;

    for (int i = 0; i < info.frameList.size(); i++) {
        QRectF drawRect = calcFrameDrawRect(i, info);

        if (i < info.thumbnailList.size() && !info.thumbnailList[i].isNull()) {
            // ✓ 缓存命中：直接绘制缩略图
            painter->drawPixmap(drawRect, info.thumbnailList[i], sourceRect);
        } else {
            // ✗ 缓存未命中：绘制占位色块 + 发起异步请求
            painter->fillRect(drawRect, QColor(128, 128, 128, 80));
            m_pScene->clipVideoThumbnailService()
                    ->requestThumbnail(clipItem->clip(), info.frameList[i]);
        }
    }
    return true;
}
```

**FThumbnailInfo — 缩略图信息聚合结构体**：

```cpp
struct FThumbnailInfo {
    QRectF         visibleBoundingRect;    // Clip 当前可视区域
    QSize          originalThumbnailSize;  // 原始解码尺寸（避免重复查询）
    QSizeF         thumbnailSize;          // 当前缩放下的绘制尺寸
    FFLongLongRange range;                 // Clip 的帧范围 [trimIn, trimOut]
    qlonglong      interval;               // 相邻缩略图之间的帧间隔
    QList<qlonglong> frameList;            // 本次需要绘制的帧号列表
    QList<QPixmap>   thumbnailList;        // 对应的已缓存 QPixmap（null = 未就绪）
    QString        effectSlug;             // 特效 slug（滤镜预览时使用）

    bool isValid() const {
        // 帧号数量与缩略图数量一致时，结构体才有效
        return frameList.count() > 0 &&
               frameList.count() == thumbnailList.count();
    }
};
```

**异步完成回调 → 主线程更新**：

```cpp
// 在 FGraphicsTimelineScene 初始化时建立连接
connect(
    m_pClipVideoThumbnailService,
    &IFFClipVideoThumbnailService::sigThumbnailReady,
    this,
    [this](IFFClip* clip, qlonglong frameNo, QPixmap px) {
        // 此 lambda 在主线程执行（Qt::QueuedConnection 保证）
        FGraphicsClipItem* clipItem = findClipItem(clip);
        if (!clipItem) return;  // ClipItem 已被销毁，安全跳过

        // 通知 PaintCache 下次必须强制重绘（不能复用旧缓存）
        auto* cache = static_cast<FTimelinePaintCache*>(clipItem->paintCache());
        if (cache) cache->bNeedForceUpdate = true;

        // 触发局部重绘（只刷新该 ClipItem 的区域，非全局）
        clipItem->update();
    },
    Qt::QueuedConnection  // 跨线程信号，确保在主线程槽中执行
);
```

**多线程安全保障**：

| 风险 | 解决方案 |
|------|---------|
| ClipItem 在回调执行前被删除 | `findClipItem(clip)` 返回 nullptr 时直接 return |
| 多个解码线程同时写入缓存 | ThumbnailService 内部使用 `QMutex` 保护缓存 Map |
| 主线程读缓存 vs 后台线程写缓存 | 通过 `QueuedConnection` 让写操作序列化到主线程 |

### 5.4 未来优化空间

1. **优先级队列**：可视区域中心的 Clip 帧优先解码（高优先级），屏幕外 Clip 低优先级或暂停。用户滚动时动态调整解码优先级。
2. **磁盘持久化缓存**：以 `md5(filePath + fileModTime + frameNo + targetSize)` 为 key，将解码帧写入本地磁盘缓存。工程二次打开时直接读取，跳过解码，打开速度可提升 10 倍以上。
3. **降采样解码**：时间线缩略图最大尺寸约 `120×68`，解码时直接指定目标尺寸（`IFFDecoderHint::targetSize`），避免解码全分辨率帧再缩放，可降低解码耗时约 3~4 倍。
4. **音频波形预计算**：`IFFClipAudioThumbnailService` 同样异步生成波形 PCM 数据，可在素材导入时预计算并持久化为 `.waveform` 文件，大幅加速工程加载速度。

## 六、Substitute 拖拽 — 数据零污染

### 6.1 背景

若拖拽时直接修改 `IFFClip` 数据，每次鼠标移动事件（60fps 下约 16ms 一次）都会触发完整的数据→UI 回调链：

```
mouseMoveEvent
    │
    └── IFFTimeline::setClipStartFrame(clip, newFrame)
            │  VBL 内部数据更新
            ▼
        Observer::onClipChanged(MOVE)
            │
            ▼
        FGraphicsTimelineTrackItem::clipChangedEvent(MOVE)
            │
            ▼
        FGraphicsClipItem::setPos(frameToScenePos(newFrame), trackY)
            │
            ▼
        QGraphicsScene::update()  → 全部可见 Clip 重绘
```

**量化问题**（低配机 i5 + 核显，30 个可见 Clip）：

| 链路步骤 | 耗时 |
|---------|------|
| VBL 数据更新 + Observer 回调 | ~3ms |
| Scene 布局重排 | ~2ms |
| 全量 Clip 重绘（PaintCache 命中） | ~6ms |
| **合计** | **~11ms** |

11ms 接近 16.6ms 帧预算，加上其他 UI 开销后极易掉帧。更严重的问题是：无法支持"ESC 取消拖拽"功能（修改了数据就必须撤销，撤销逻辑复杂）。

### 6.2 优化方案

拖拽期间**完全绕过数据层**，只移动视觉副本（Ghost Item），数据保持不变。鼠标释放时一次性提交最终位置。详细实现参见主文档《第四章 Substitute 拖拽模式》。

核心类结构：

```
FGraphicsClipItemSubstitute        ← Ghost Item（跟随鼠标的视觉副本）
    m_oOriginalRange               ← 拖拽起始位置（用于 ESC 取消）
    m_oCurrentRange                ← 当前预览位置（实时更新）
    m_pSourceClipItem              ← 原 ClipItem 引用（同步外观）

FGraphicsItemSubstituteManager     ← 管理当前活跃的 Action
    startClipMove(clipItem, pos)   ← 启动移动替身
    startClipTrimIn(clipItem, pos) ← 启动裁剪入点替身
    ...（16+ 种操作）

FGraphicsClipItemSubstituteMoveHandler   ← 标准移动处理器
FGraphicsClipItemSubstituteMTrackSnapHandler ← 主轨磁性吸附处理器
FGraphicsClipItemSubstituteVTrackLinkHandler ← 垂直轨道联动处理器
```

### 6.3 性能对比

| 操作场景 | 直接修改数据（每帧） | Substitute 模式（每帧） | 提升 |
|---------|-------------------|----------------------|------|
| 单 Clip 拖拽（30 可见 Clip） | ~11ms | ~1.5ms | **86%** |
| 多 Clip 框选拖拽（10 个选中） | ~35ms | ~4ms | **89%** |
| 主轨磁性拖拽（触发让位重排） | ~45ms | ~5ms | **89%** |
| ESC 取消拖拽 | 需要撤销操作（复杂） | 直接销毁 Ghost（O(1)） | **质的提升** |

### 6.4 未来优化空间

1. **SubstituteItem 轻量化**：当前 Ghost Item 完整复制了原 Clip 的缩略图、文字、图标等所有外观。可简化为只绘制半透明蓝色边框轮廓，减少 Ghost 自身的绘制开销（约从 2ms 降至 0.3ms）。
2. **预测性提交**：当用户拖拽减速（连续 3 帧位移量 < 2px）时，预测即将松手，提前在后台线程计算目标位置（磁性吸附、冲突检测），降低松手瞬间的提交延迟感。

## 七、FClipPainterOptions — 绘制项按需裁剪

### 7.1 背景

不同高度的轨道和不同状态的 Clip，其所需绘制内容差异很大，但 `cpoDEFAULT` 模式会尝试绘制所有元素：

| Clip 状态 | 实际可见内容 | `cpoDEFAULT` 尝试绘制 | 浪费项 |
|---------|-----------|-------------------|------|
| 小高度轨道（~25px） | 背景色 + 名称 + 图标 | + 缩略图 + 音频波形 + 关键帧 | 缩略图/波形（查询即返回空） |
| Ghost SubstituteItem | 背景轮廓 | + 缩略图/波形/图标/关键帧 | 全部装饰元素 |
| 极短 Clip（< 20px 宽） | 背景色 + 边框 | + 名称（放不下） + 缩略图 | 文字测量 + 缩略图查询 |

### 7.2 优化方案

`FClipPainterOptions`（Qt 位标志枚举）精确控制每次 `paint()` 的绘制内容：

```cpp
// Src/FTimelineView/FGraphicsClipPainter.h
enum FClipPainterOption {
    cpoBackground    = 0x0001,   // 背景填充色
    cpoBorder        = 0x0002,   // 边框线
    cpoThumbnail     = 0x0004,   // 视频帧缩略图
    cpoWave          = 0x0008,   // 音频波形
    cpoIcon          = 0x0010,   // 媒体类型图标（视频/音频/图片标志）
    cpoText          = 0x0020,   // 片段名称文字
    cpoVideoKeyFrame = 0x0040,   // 视频关键帧菱形
    cpoMotionTrack   = 0x0080,   // 运动追踪标记点
    cpoFunctionIcon  = 0x0100,   // 功能图标（特效标志/关键帧存在标志）
    cpoAudioKeyFrame = 0x0200,   // 音频关键帧菱形
    cpoTaskQueue     = 0x0400,   // AI 任务进度圆圈

    // 预定义组合：
    cpoKeyFrame = cpoVideoKeyFrame | cpoAudioKeyFrame,

    // 小高度模式：省略缩略图、波形、运动追踪
    cpoSmall    = cpoBackground | cpoBorder | cpoText
                | cpoIcon | cpoFunctionIcon
                | cpoVideoKeyFrame | cpoTaskQueue,

    // 全量模式（默认）
    cpoDEFAULT  = cpoBackground | cpoBorder | cpoThumbnail | cpoWave
                | cpoIcon | cpoText | cpoKeyFrame
                | cpoMotionTrack | cpoFunctionIcon | cpoTaskQueue,
};
Q_DECLARE_FLAGS(FClipPainterOptions, FClipPainterOption)
```

**动态选择逻辑**（`FGraphicsBaseClipPainter::getPaintOptions()`）：

```cpp
FClipPainterOptions FGraphicsBaseClipPainter::getPaintOptions(
    FGraphicsClipItem* clipItem) const
{
    // 轨道高度级别：thlSmall / thlMiddle / thlNormal
    if (FGraphicsTrackHelper::inSmallHeightTrack(clipItem)) {
        return FClipPainterOption::cpoSmall;   // 小高度：跳过缩略图/波形
    }
    return m_oOptions;  // 正常高度：使用默认或调用者设置的选项
}
```

**在 `paintWithCache()` 中的使用**：

```cpp
void FGraphicsVideoClipPainter::paintWithCache(
    const FTimelinePaintOption& option, FTimelinePaintCache* cache)
{
    FClipPainterOptions opts = getPaintOptions(clipItem);

    if (opts & cpoBackground)    paintVideoBackground(...);
    if (opts & cpoBorder)        paintVideoBorder(...);
    if (opts & cpoThumbnail)     paintVideoThumbnail(...);  // 跳过 = 节省缩略图查询
    if (opts & cpoText)          paintVideoText(...);       // 跳过 = 节省 fontMetrics 计算
    if (opts & cpoVideoKeyFrame) paintVideoKeyFrames(...);
    // ...
}
```

### 7.3 各场景绘制选项对比

```
轨道高度    绘制选项       省略内容              节省耗时（估算）
──────────────────────────────────────────────────────────────
thlNormal   cpoDEFAULT    无（全量）             基准
thlMiddle   cpoDEFAULT    无（全量）             基准
thlSmall    cpoSmall      缩略图+波形+运动追踪   约 -60%

替身 Ghost  （可扩展）    全部装饰元素          约 -80%
极短 Clip   （可扩展）    缩略图+文字           约 -40%
```

### 7.4 未来优化空间

1. **宽度自适应裁剪**：增加 `cpoThumbnailVisible` 选项——仅当 Clip 宽度 > 单帧缩略图最小宽度（约 24px）时才绘制缩略图，否则自动退化为纯色背景。
2. **Ghost 专用选项**：为 SubstituteItem 定义 `cpoGhost = cpoBackground | cpoBorder`，Ghost 绘制只需背景+边框，可将 Ghost 的单次绘制耗时从 ~2ms 降至 ~0.2ms。
3. **条件化 Options 缓存**：将 `getPaintOptions()` 的返回值纳入 `FTimelinePaintCache` 的 `isSame()` 比较，避免选项变化时（如用户调整轨道高度后）仍使用旧 Options 绘制。
