### 什么是静态库 vs 动态库？
| 类型    | 静态库（Static Library） | 动态库（Dynamic Library）    |
| ----- | ------------------- | ----------------------- |
| 编译阶段  | 编译时合并进可执行文件         | 编译时记录引用，运行时加载           |
| 运行时依赖 | 不需要外部库              | 需要 `.dll/.dylib/.so` 存在 |
| 文件大小  | 较大（代码都打包进来了）        | 较小（共享代码）                |
| 适合场景  | 单个独立程序、无需部署额外文件     | 多程序共享库、更新方便             |

### 库文件的后缀和结构区别
| 平台      | 静态库         | 动态库                      | 是否需要导入库     |
| ------- | ----------- | ------------------------ | ----------- |
| Windows | `.lib`（静态库） | `.dll`（运行用）+ `.lib`（导入库） | ✅ 需要 `.lib` |
| macOS   | `.a`        | `.dylib`                 | ❌ 不需要导入库    |

**补充说明**：

- **Windows 下 `.lib` 分两种用途**：
  1. **静态库**：里面包含实际代码 → 链接后复制进 `.exe`
  2. **导入库**：仅供链接器识别 `DLL` 符号，不包含代码
- **macOS/Linux 下 `.a` 是静态库，`.dylib` 是动态库，本身都包含完整符号信息，无需 `.lib` 一样的导入库**

### 链接过程上的差异
| 阶段       | Windows                   | macOS                       |
| -------- | ------------------------- | --------------------------- |
| 编译时链接静态库 | `.lib` 直接打包进 `.exe`       | `.a` 打包进 `.exe`             |
| 编译时链接动态库 | `.lib` 提供导入信息 → 链接 `.dll` | `.dylib` 直接链接               |
| 运行时      | 加载 `.dll`                 | 加载 `.dylib`                 |
| 系统动态加载器  | `Windows Loader`          | `dyld`（Dynamic Link Editor） |

### 实际部署和运行差异
| 内容       | Windows                              | macOS                         |
| -------- | ------------------------------------ | ----------------------------- |
| 动态库查找路径  | 当前目录、系统目录、PATH 环境变量                  | `@rpath`、`@loader_path`、系统库目录 |
| 工具链      | MSVC / Clang with MS ABI             | Clang with Mach-O ABI         |
| 查看导入依赖工具 | `dumpbin /IMPORTS`、Dependency Walker | `otool -L`、`nm`               |
| 默认链接行为   | 倾向使用导入库 `.lib` + `.dll`              | 倾向直接链接 `.dylib`               |

### 举个例子比较

1. Windows：

```
target_link_libraries(MyApp PRIVATE MyLib.lib)
# 编译后运行依赖 MyLib.dll
```

- `.lib` 用来链接
- `.dll` 是运行时必须存在的二进制代码文件

2. macOS：

```
target_link_libraries(MyApp PRIVATE libMyLib.dylib)
# 编译器直接读取 .dylib 的符号表
# 运行时同样需要 libMyLib.dylib 存在
```



### 区别对照表

| 比较项           | Windows            | macOS             |
| ---------------- | ------------------ | ----------------- |
| 静态库后缀       | `.lib`             | `.a`              |
| 动态库后缀       | `.dll`             | `.dylib`          |
| 导入库           | `.lib`（动态库用） | ❌ 不需要          |
| 链接动态库方式   | 通过导入库         | 直接链接 `.dylib` |
| 运行时加载器     | Windows Loader     | dyld              |
| 是否强依赖导入库 | ✅ 是               | ❌ 否              |



### Win 下导入库的核心机制

**操作系统通过 `.exe` 文件中的“导入表（Import Table）” 来发现 `.dll` 依赖**。

在 Windows 下，一个 `.exe` 或 `.dll` 文件是 **PE 文件格式（Portable Executable）**，结构非常复杂，其中包含一个叫：

**Import Table（导入表）**，它记录了：

- 当前程序依赖的所有 DLL 名字（如 `SomeLib.dll`）
- 每个 DLL 中需要导入的函数或符号名字（如 `SomeFunction`）

**这个表是怎么进到 `.exe` 里的？**

当你在链接阶段使用导入库 `.lib` 时，链接器（`link.exe`）会：

1. 把 `.lib` 中的信息写进 `.exe` 的 Import Table
2. `.exe` 中就明确包含了一个“我要加载 `SomeLib.dll`，并调用它的 `SomeFunction`”

所以这个表是 编译时构建进去的，运行时不会变。

**程序运行时，系统动态链接器做什么？**

当你双击运行 `MyApp.exe` 时：

1. 操作系统加载 `.exe` 文件到内存
2. 发现它有 Import Table
3. 逐个加载对应的 `.dll` 文件（`SomeLib.dll`）
4. 找到 `.dll` 中导出的函数地址（比如 `SomeFunction`）
5. 把这些地址**写入 `.exe` 的 IAT（Import Address Table）中**
6. 程序里的调用就会跳转到 `.dll` 里的真正实现

**总结：**

| 项                    | 内容                                                      |
| --------------------- | --------------------------------------------------------- |
| `.lib` 作用           | 编译阶段，告诉链接器：程序依赖 `SomeLib.dll` 和其中的符号 |
| `.exe` 怎么记录依赖？ | 用导入表（Import Table），记录 DLL 和函数                 |
| 谁解析？              | 操作系统的动态链接器在程序启动时读取这个表                |
| 加载 `.dll` 规则      | 按固定的 DLL 搜索路径加载                                 |
| 最终实现跳转          | 系统会填充 IAT，让 `.exe` 中的函数调用跳转到 `.dll`       |



### 为什么mac下不需要链接导入库？

 macOS 使用的是“延迟绑定”的动态库机制（`dylib`），不需要 `.lib` 导入库来辅助链接。

macOS 的动态库 `.dylib` 本身就包含符号表！macOS 的 `.dylib`（或 Linux 下的 `.so`）格式本身是：

- 包含完整的导出符号表
- 支持链接器直接使用 `.dylib` 来解析符号

所以 macOS 下链接时：

`libMyLib.dylib` 中包含了足够的信息（比如导出函数名、符号）

链接器（如 `ld` 或 `clang`）直接使用 `.dylib` 文件完成链接

根本不需要一个类似 `.lib` 的“中间导入库”
