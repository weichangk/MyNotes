## 一、属性面板首次打开耗时优化


### 1.1 问题描述

第一次进入视频、文字、图片属性面板时耗时严重（Debug 模式下实测）：

| 属性面板类型 | 优化前耗时 | 优化后耗时 | 提升比例 |
|------------|-----------|-----------|---------|
| 视频属性面板 | 1.6s | 0.5s | **↓ 68.8%** |
| 图片属性面板 | 1.3s | 0.4s | **↓ 69.2%** |
| 文字属性面板 | 3.0s | 2.1s | **↓ 30.0%** |

### 1.2 耗时原因分析

#### 原因一：提前创建子控件
- 打开属性面板时会一次性创建所有子控件（包括折叠中的、不可见的）
- 大量控件的 `new` 操作和 Qt 布局计算集中在一个时间点执行
- 结论：大量控件同步初始化是主要耗时来源

#### 原因二：子线程与主线程同步阻塞
- 某些初始化工作在子线程中进行
- 主线程访问时，子线程尚未创建完成，需要等待
- 子线程中的 UI 事件屏蔽处理增加额外开销

#### 原因三：文字面板特殊问题
- 文字属性面板所有控件默认展开
- 无法使用懒加载方案
- 额外耗时来源：动画数据加载（每次进入都会同步加载）

### 1.3 优化方案分析

#### 方案一：提前创建属性面板（已排除）

```
❌ 不可行原因：
1. 提前创建的时机不好控制
2. 放在软件启动时会影响启动速度（启动已有 20+ 个 initXXX 方法）
3. 放在子线程中创建：
   - 需要处理线程同步（主线程可能在子线程未完成时访问）
   - Qt UI 控件不能在非主线程创建（Qt 限制）
```

#### 方案二：先显示主控件，再异步加载子控件（已排除）

```
❌ 不可行原因：
1. 主控件显示出来是空的，用户体验差
2. 如何处理从时间线直接跳转到指定功能属性子控件的情况？
3. 子控件未加载完成时访问的竞态问题难以解决
```

#### 方案三：延迟创建（懒加载）— 最终采用方案 ✅

核心思路：

```
控件树结构：
┌─────────────────────────────────────┐
│ 属性面板主控件（外壳）               │  ← 立即创建（轻量）
│                                     │
│ ┌─── 折叠控件 A ───┐               │
│ │  [内部子控件]     │  ← 展开时才创建 │
│ └───────────────────┘               │
│ ┌─── 折叠控件 B ───┐               │
│ │  [内部子控件]     │  ← 展开时才创建 │
│ └───────────────────┘               │
└─────────────────────────────────────┘
```

**实现要点：**
1. 将控件的外壳（标题栏/折叠头）和内部属性控件分离
2. 内部属性控件的创建推迟到折叠展开时
3. 内部属性创建前进行指针判空处理（防止早期访问崩溃）

### 1.4 具体实施细节

#### 视频 & 图片属性面板（延迟创建子控件）

```cpp
// 伪代码示意
class FPropertyVideoControl : public QWidget {
    FVideoPropertySubWidget* m_pSubWidget = nullptr;  // 初始为 nullptr

    void onExpand() {
        if (!m_pSubWidget) {
            // 展开时才创建内部控件（懒加载）
            m_pSubWidget = new FVideoPropertySubWidget(this);
            m_pSubWidget->init();
            layout()->addWidget(m_pSubWidget);
        }
        m_pSubWidget->setVisible(true);
    }

    void onCollapse() {
        if (m_pSubWidget) {
            m_pSubWidget->setVisible(false);
        }
    }
};
```

**效果：**
- 初次打开面板只创建折叠头控件（极轻量）
- 用户展开哪个面板，才创建对应内部控件
- 耗时从 1.6s → 0.5s（视频），1.3s → 0.4s（图片）

#### 文字属性面板（延迟加载动画数据）

文字面板所有控件默认展开（无法做控件延迟创建），但发现一个优化点：

```
原来：打开面板时立即加载所有动画预设数据（IO + 解析）
优化：延迟加载动画数据，等用户切换到动画 Tab 时再加载
```

**效果：** 耗时从 3s → 2.1s

### 1.5 遗留问题与未来优化方向

```
1. 已达到当前方案的优化极限（懒加载已覆盖主要场景）

2. 发现 Qt Layout 布局本身比较耗时（原因待查）
   - 猜想：大量嵌套 QVBoxLayout / QHBoxLayout 的测量计算
   - 可能优化方向：
     a. 减少布局嵌套层数
     b. 使用固定尺寸控件绕过布局计算
     c. 使用自定义布局替代 Qt 标准布局
```

### 2.2 批量编辑架构（性能相关）

属性面板支持多个 Clip 同时选中时的批量编辑，使用模板机制：

```cpp
// IFPropertyEditor.h

// 批量读取：统一读取多个 clip 的同一属性，检测是否存在"多值"
template<typename Target, typename Value>
struct FPropertyBatchGetter {
    static FMultipleValue<Value> get(
        const QList<Target>& targets,
        const std::function<Value(Target)>& readFunc
    );
};

// 批量写入：同时设置多个 clip 的属性（支持撤销/重做）
template<typename Target>
struct FPropertyBatchSetter {
    static bool set(
        IFFUndoRedoService* undoRedoService,
        const QList<Target>& targets,
        const std::function<void(Target)>& setFunc,
        const QString& debugDescription = ""
    );
};
```

**多值检测机制：**
```cpp
// 浮点数用 qFuzzyCompare 比较（避免精度误差）
if constexpr (std::is_floating_point<Value>::value) {
    if (!qFuzzyCompare(static_cast<float>(currentValue), static_cast<float>(value))) {
        return FMultipleValue<Value>(value, true);  // isMultiple = true
    }
}
```

**性能优化点：**
- 批量写入时使用 `FF_DECLARE_PAUSE_EVENT_COMMANDER` 暂停事件发送
- 避免 N 个 clip 的写入触发 N 次 UI 刷新，合并为 1 次
- 撤销/重做使用命令模式，O(1) 的撤销操作

### 2.3 Clip 功能特性检测

```cpp
// FClipsFeatureInspector.h
enum class FClipFeatureType {
    AIPotrait,      // AI 智能抠人像
    MotionTracking, // 运动追踪
    SmartCutout,    // AI 智能抠像
    Stabilization,  // 稳像
    VideoDenoise,   // 视频降噪
    PlanarTrack,    // 平面追踪
};

struct FClipsFeatureInspector {
    // 快速判断（仅通过 clipType）：O(n) 遍历
    bool supportsFeatureOnlyByClipType(
        FClipFeatureType featureType,
        const QList<FFVBLObjectRefPtr<IFFAbstractClip>>& clips
    );

    // 精确判断（完整检测）：更多条件检查
    static bool supportsFeature(
        FClipFeatureType featureType,
        const QList<FFVBLObjectRefPtr<IFFAbstractClip>>& clips,
        bool ignoreClipTypeCheck = false
    );
};
```

**性能策略：** 优先使用 `supportsFeatureOnlyByClipType`（快速路径），只有需要精确判断时才用 `supportsFeature`。

## 三、Qt Layout 性能问题（通用建议）

基于属性面板耗时分析，发现 Qt Layout 计算是性能瓶颈。总结规律如下：

### 3.1 Layout 耗时的常见来源

```
1. 嵌套 Layout 层数过多
   QVBoxLayout > QHBoxLayout > QGridLayout > ...
   每层都需要递归计算 sizeHint() 和 minimumSize()

2. 频繁调用 layout()->invalidate() 或 update()
   每次添加/删除子控件都会触发全量重新布局

3. 大量控件同时创建后立刻加入布局
   N 个 addWidget 调用会触发 N 次布局重算
```

### 3.2 优化策略

```cpp
// ❌ 低效：逐个 addWidget 触发多次布局
for (auto widget : widgets) {
    layout->addWidget(widget);   // 每次触发重布局
}

// ✅ 高效：批量操作，用 setUpdatesEnabled 暂停刷新
parentWidget->setUpdatesEnabled(false);
for (auto widget : widgets) {
    layout->addWidget(widget);
}
parentWidget->setUpdatesEnabled(true);  // 统一触发一次

// ✅ 高效：使用 QStackedWidget 替代动态 addWidget/removeWidget
// 所有子页面预先加入，切换时只改变 currentIndex（无需重新布局）
```

## 四、性能优化经验总结

| 优化场景 | 技术手段 | 效果 |
|---------|---------|------|
| 属性面板首次打开 | 懒加载（延迟创建子控件） | 耗时降低 60-70% |
| 属性面板数据动画加载 | 延迟加载数据（按需加载） | 耗时降低 30% |
| 批量属性编辑 | 暂停事件发送 + 合并刷新 | 避免 N 次 UI 刷新 |
| AI 功能可用性判断 | 快速路径（clipType 判断） | O(1) vs 全量检测 |
| Layout 刷新 | setUpdatesEnabled + 批量添加 | 避免多次重布局 |
