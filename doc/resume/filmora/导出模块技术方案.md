## 二、数据模型层

### 2.1 FExportDataDelegate（共享数据上下文）

`FExportDataDelegate` 是整个导出流程的**数据上下文对象**，在所有 Proxy、Panel、Presenter 间传递，类似 Android 的 Intent：

```cpp
// ExportDataModel/FExportDataDelegate.h
class FExportDataDelegate {
    // 导出上下文
    FFExportMode exportMode();           // emTimeline / emClips / emProjects
    FFLongLongRange exportRange();       // 导出时间范围（全片/选区）
    int formatType();                    // 导出格式索引

    // 数据对象
    IFFProject* project();              // 当前工程
    IFFTimeline* timeline();            // 当前时间线
    QList<IFFClip*> clips();            // 导出的 Clip 列表（指定 Clip 导出）
    QList<IFFTimeline*> timelines();    // 批量导出时的多时间线列表
    IFFMediaLibrary* mediaLibrary();    // 媒体库（子进程复用）
    IFFMediaPlayer* mediaPlayer();      // 媒体播放器（封面帧提取）

    // 授权数据
    QSharedPointer<FUnLicenseTimelineData> unLicenseData(); // 未授权资源信息

    // 扩展属性（KV 存储）
    QVariant property(const char* name);
    void setProperty(const char* name, const QVariant& value);

    // 设计师模板专用参数
    FDesignerExportParam getDesignerExportParam();
};
```

**三种导出模式（FFExportMode）：**

| 模式 | 说明 | 触发场景 |
|------|------|---------|
| `emTimeline` | 导出整个时间线 | 常规"导出"操作 |
| `emClips` | 导出指定 Clip 列表 | AutoReframe 批量导出 |
| `emProjects` | 批量导出多个工程 | 批量工程导出 |

### 2.2 FExportFormatInfo（格式信息数据库）

`FExportFormatInfo` 是导出格式数据的核心模型，从 `Format_H.dat` 文件（XML 格式）解析所有支持的导出格式：

```cpp
// ExportDataModel/FExportFormatInfo.h
class FExportFormatInfo : public QObject {
    // 格式元数据
    struct EncodeData {
        QString name;          // 格式名称（如 "MP4"、"MOV"）
        EncodeParam defaultParam;   // 默认编码参数
        EncodeParam currentParam;   // 当前用户选择的编码参数

        bool haveAudio = false; // 是否含音频轨

        QString frameRate;     // 帧率（30/25/23.976/…）

        QString videoCompressionMode;  // 压缩质量模式
```

## 三、Proxy 责任链（前置检查架构）

### 3.1 设计模式：责任链（Chain of Responsibility）

导出模块在实际编码开始之前，需要通过一系列检查门槛（License、素材授权、文件完整性等）。这些检查互相独立，但有先后顺序，且每个检查失败后有不同的处理方式（弹窗/购买引导/直接中止）。

**接口设计：**

```cpp
// ExportProxy/FExport.h

// 导出结果枚举
enum FExportResult {
    erSucceed,  // 通过，继续执行下一个
    erRetry,    // 需要重试（如购买后重新检查）
    erAbort,    // 中止导出
};

// 导出执行接口
class IFExport {
    virtual FExportResult execute() = 0;
};

// 抽象代理基类（包装下一个代理）
class FAbstractExportProxy : public IFExport {
    IFExport* m_pNextExportProxy;  // 持有下一个 Proxy

    FExportResult execute() override {
        FExportResult result = doExecute();  // 执行本代理逻辑
        if (result == erSucceed)
            return m_pNextExportProxy->execute();  // 通过则继续
        return result;  // 失败则直接返回
    }
    virtual FExportResult doExecute() = 0;  // 子类实现具体检查
};
```

### 3.2 各代理职责

| 代理类 | 检查内容 | 失败处理 |
|--------|---------|---------|
| `FAppLicenseExportProxy` | 程序授权状态（免费版/付费版/过期）| 弹购买/续费/水印提示对话框 |
| `FMYLicenseExportProxy` | 万兴账号 License（MY账号特殊授权）| 弹对应 License 对话框 |
| `FMediaItemLicenseExportProxy` | 时间线中是否有未授权付费素材 | 弹素材授权清单对话框（可选择水印导出） |
| `FCheckMissFilesProxy` | 时间线素材是否存在缺失/离线 | 弹素材缺失提示，可跳过继续 |
| `FCheckTailVideoFileProxy` | 片尾视频文件是否有效 | 弹错误提示 |
| `FFormatSelectExportProxy` | 显示格式选择面板（用户确认参数） | 用户取消 → erAbort |
| `FExportProgressProxy` | 启动实际导出进度 | 启动编码子进程 |

### 3.3 责任链构建示例

```cpp
// 构建责任链（倒序构建，最内层最先创建）
auto* pProgress = new FExportProgressProxy(nullptr);
auto* pFormat = new FFormatSelectExportProxy(pProgress);
auto* pTailVideo = new FCheckTailVideoFileProxy(pFormat);
auto* pMissFiles = new FCheckMissFilesProxy(pTailVideo);
auto* pMediaLicense = new FMediaItemLicenseExportProxy(pMissFiles);
auto* pMyLicense = new FMYLicenseExportProxy(pMediaLicense);
auto* pAppLicense = new FAppLicenseExportProxy(pMyLicense);

// 所有代理共享 DataDelegate
pAppLicense->setDataDelegate(pDataDelegate);
// …每个代理都设置同一个 DataDelegate

// 执行（从第一个代理开始）
FExportResult result = pAppLicense->execute();
```

## 五、多进程编码架构

### 5.1 为什么需要独立编码子进程

| 问题 | 解决方案 |
|------|---------|
| 编码是 CPU/GPU 密集型操作，占用主线程会导致 UI 卡死 | 独立子进程隔离，主进程 UI 保持流畅 |
| GPU 硬件编码器偶发崩溃会拖垮整个应用 | 子进程崩溃不影响主进程，可自动重试 |
| 编码占用大量内存（GPU 显存 + 帧缓冲），与主进程竞争 | 独立进程独立内存空间 |
| 需要不同的权限或 DLL 上下文（如 Dolby 编码器）| 子进程独立加载，不污染主进程 |

### 5.2 IFFEncoder 接口抽象

`FProcessEncoder` 和 `FEncoderServer` 都实现了 `IFFEncoder` 接口，这使得上层代码无需关心是否使用子进程：

```cpp
class IFFEncoder {
    virtual void setEncodeParam(const FFVBLMODEL::FFEncodeParam& param) = 0;
    virtual void setTimeline(IFFTimeline* timeline) = 0;
    virtual void setClips(QList<IFFClip*>& clips) = 0;
    virtual bool start() = 0;
    virtual bool pause() = 0;
    virtual bool resume() = 0;
    virtual bool stop() = 0;
    virtual bool retry() = 0;
    virtual double estimateDiskSpace() = 0;
    virtual void attach(IFFEncoderEventObserver* observer) = 0;
    virtual void detach(IFFEncoderEventObserver* observer) = 0;
};
```

两种编码器实现：

| 实现类 | 使用场景 | 编码执行位置 |
|--------|---------|------------|
| `FProcessEncoder` | 默认（`useProcessEncoder=true`） | 独立 `FExportExe.exe` 子进程 |
| `FEncoderServer`（直接模式） | 轻量导出/调试 | 主进程内（`useProcessEncoder=false`） |

### 5.3 FProcessEncoder 子进程通信详解

**启动序列：**

```
FProcessEncoder::start()
       │
       ▼
① initProcess()
  ├── 将 IFFTimeline 序列化为 JSON 临时文件（m_timelineFilePath）
  ├── 将 FFEncodeParam 序列化为 JSON 字符串（m_encodeParam）
  ├── 启动 FExportExe.exe（带参数：IPC 端口号 + timeline 文件路径）
  └── m_pProcressInitTimer->start(120000)  ← 120秒超时保护
       │
       ▼
② FExportExe.exe 启动（FExportApp::init()）
  ├── initEnv() / initVBL() / initMediaLibrary()
  ├── initDataModel() / initDecodeManager()
  └── FExportServerClient 与主进程建立 IPC 连接
       │
       ▼
③ FExportServerClient::sendInitFinishEvent(time)
  → 通过 IPC 发送 "InitFinish" 消息（含初始化耗时）
       │
       ▼
④ FProcessEncoder::processInitFinished(secTime)
  ├── m_pProcressInitTimer->stop()  ← 停止超时定时器
  ├── 通过 IPC 发送 "Start" 消息（含 encodeParam）
  └── m_pMonitorTimer->start(15000)  ← 启动 15s 卡死检测
```

**编码过程中消息类型：**

```cpp

void handleConnectedMsg(FFIPCEvent*);   // 连接建立
void handleDisconnectedMsg(FFIPCEvent*);// 连接断开
void handleStartMsg(FFIPCEvent*);       // 开始编码

void handleResumMsg(FFIPCEvent*);       // 恢复
void handleStopMsg(FFIPCEvent*);        // 停止
```

**编码回调（子进程 → 主进程）：**

```cpp

void onProgressChanged(double progress);               // 0.00~1.00 进度
void onTaskProgressChanged(int curTaskIdx, int totalTask); // 多任务批量进度
```

### 5.4 卡死检测与重试机制

`FProcessEncoder` 内置三个 `QTimer` 保障导出稳定性：

```
m_pProcressInitTimer（120s）
  作用：子进程启动后 120s 内必须发送 InitFinish 消息
  超时触发：判定子进程启动失败，回调 onStateChanged(esError)

m_pMonitorTimer（15s）
  作用：编码过程中每 15s 必须收到进度回调
  超时触发：判定子进程卡死
  处理：调用 reStartCloseGpu()（关闭 GPU 编码后重试）

m_pRetryTimer
  作用：重试时等待 IPC Server 关闭后再重新启动
  场景：GPU 卡死后 → 禁用 GPU → 切换软件编码重试
```

## 十二、面试常见问题与答案

### Q1：导出为什么要独立子进程？FExportExe 和主进程如何通信？

**A：** 导出是 CPU/GPU 密集操作，独立子进程有三个优势：
1. **UI 不卡顿**：子进程执行编码，主进程 UI 线程完全空闲；
2. **崩溃隔离**：GPU 硬编码偶发崩溃（尤其 NVIDIA 驱动问题），子进程崩溃不影响主进程，FExportExe 可自动重试；
3. **内存隔离**：4K 编码需大量 GPU 显存，独立进程不与主进程争资源。

通信使用项目内部的 `FIPC` 框架（`FFIPCClient` + `FFIPCServer`，基于命名管道/Socket）。主进程端 `FProcessEncoder` 实现 `IFFEncoder` 接口，通过 IPC 发送 Start/Pause/Resume/Stop 命令；子进程端 `FExportServerClient` 继承 `FFIPCClient`，回调编码进度和状态。

---

### Q2：Proxy 责任链的设计好处是什么？

**A：** 导出前有 7 种独立检查（License/素材授权/缺失文件/片尾视频/格式确认/磁盘空间等），若全写在一个函数里：
- 代码超过 500 行，职责混乱；
- 添加新检查需要修改主函数，违反开闭原则；
- 每种检查的"失败处理"各不相同（有的弹购买框、有的直接中止），难以统一。

责任链模式的优势：
1. 每个检查类独立，单一职责，易于单测；
2. 新增检查只需创建新 Proxy 类并插入链中，不修改其他代码；
3. `FExportResult` 枚举清晰表达 Succeed/Retry/Abort 三种结果；
4. 所有 Proxy 共享同一个 `FExportDataDelegate`，数据透传无需参数传递。

---

### Q3：FExportDataDelegate 的设计意图是什么？

**A：** 导出流程跨越 7+ 个 Proxy、多个 Panel、Presenter、进度对话框，如果用函数参数传递所有数据（timeline/encodeParam/project/mediaLibrary/水印状态/授权信息等），参数列表会非常臃肿。

`FExportDataDelegate` 是一个**上下文数据容器**（类似 Android `Intent` 或 Qt `QMimeData`），将所有导出相关数据封装为一个对象：
- 所有 Proxy 和 Presenter 共享同一个 `FExportDataDelegate*`，通过 getter/setter 读写数据；
- 支持 `property(name)/setProperty(name, value)` 动态扩展属性，不修改接口即可传递新数据；
- 生命周期与整个导出流程绑定，流程结束后销毁。

---

### Q4：FProcessEncoder 的超时和卡死检测如何实现？

**A：** 使用三个 `QTimer`：

1. **`m_pProcressInitTimer`（120s）**：子进程启动后 120s 内必须通过 IPC 发送 `InitFinish` 消息。超时说明子进程启动失败（可能进程已崩溃或 GPU 初始化卡死），触发错误回调通知 UI。

2. **`m_pMonitorTimer`（15s）**：编码过程中每 15s 必须收到 `onProgressChanged()` 回调。超时说明子进程卡死（常见于 GPU 驱动 Bug），触发 `reStartCloseGpu()` 策略：禁用 GPU 硬编码，切换为软件编码重试（`isDisableGPU = true`）。

3. **`m_pRetryTimer`**：重试前需等待旧的 IPC Server 完全关闭，避免端口冲突。Timer 触发时才真正重新启动子进程。

---

### Q5：FExportProgressPresenter 如何与 VBL 的进度数据绑定？

**A：** 进度 UI 通过 `IFFExportManager`（VBL 层接口）获取实时进度。`FExportProgressPresenter::init(view, exportManager)` 完成绑定：

- `IFFExportManager` 管理所有 `IFFExportPipeline`（每条导出管道）；
- 每个 `IFFExportPipeline` 管理若干 `IFFExportJob`（编码任务 or 上传任务）；
- `FExportProgressPipelineGroupPresenter` 监听 `IFFExportManager` 的回调，将进度分发到对应 `FExportProgressPipelinePresenter`；
- 最终 `FExportProgressLocalTaskItemPresenter` 更新 UI 的进度条。

这个三层 Presenter 结构（Group → Pipeline → TaskItem）与 VBL 的 Manager/Pipeline/Job 三层模型一一对应，职责清晰，支持同时展示多条并行导出管道的进度。

---

### Q6：导出格式数据（FExportFormatInfo）从哪里来？如何扩展新格式？

**A：** `FExportFormatInfo` 从 `Format_H.dat`（XML 格式的配置文件）解析所有格式：

- 每种格式包含：名称、是否有视频/音频/图片、网站索引（-1=本地，>=0=社媒平台）；
- 每种格式有默认编码参数（分辨率/帧率/比特率/编码器/色彩空间）；
- 用户自定义参数通过 `saveExportParam()` 写入 `FFConfig.ini`，下次打开时 `loadExportParam()` 恢复。

扩展新格式只需修改 `Format_H.dat`，无需改代码：
1. 添加新格式的 XML 节点，设定其默认参数；
2. 如果是社媒平台（需要登录上传），设置 `WebsiteIndex` 对应枚举值；
3. 如果需要特殊授权（Dolby），通过 CBS 开关动态隐藏（`removeDolbyFormatItems()`）。
