## 二、核心 Presenter 层设计

### 2.1 Presenter 层次结构

资源区的 Presenter 采用**三层嵌套**结构，与 UI 的三层嵌套结构一一对应：

```
FFMediaLibraryPresenter                        ← 顶层：协调所有 Tab
  │  持有 IFFMediaLibrary* m_pMediaLibrary
  │  持有 QHash<FFMediaCategoryType, FFMediaCategoryPresenter*> m_oCategoryPresenters
  │
  └── FFMediaCategoryPresenter × N            ← 中层：单个 Tab 的数据协调
        │  持有 IFFMediaManager* m_pMediaManager
        │  懒加载：m_oCategoryPresenters[type]
        │
        ├── FFFolderMediaManagerPresenter     ← 子层：目录树 Presenter
        │     持有 IFFFolderMediaManager*
        │     持有 IFFFolderMedia* m_pCurFolder（当前选中目录）
        │
        └── FFMediaItemManagerPresenter       ← 子层：素材列表 Presenter
              持有 IFFMediaItemManager*
              持有 IFFFolderMedia* m_pRootFolder（当前展示目录）
              持有 FFMediaItemDataFilterScene*（过滤场景）
```

**关键设计点：Presenter 懒加载**

`FFMediaLibraryPresenterPrivate` 中通过 `getOrCreateCategoryPresenter()` 实现懒加载：
每个 `FFMediaCategoryType`（媒体/特效/音频/…）第一次被访问时才创建对应的 `FFMediaCategoryPresenter`，之后缓存复用。这避免了启动时同时初始化十余个分类 Presenter 带来的开销。

```cpp
// FFMediaLibraryPresenter_P.h
FFMediaCategoryPresenter* getOrCreateCategoryPresenter(
    const FFVBLMODEL::FFMediaCategoryType categoryType)
{
    auto it = m_oCategoryPresenters.find(categoryType);
    if (it == m_oCategoryPresenters.end())
    {
        auto* p = createNewCategoryPresenter(categoryType);
        it = m_oCategoryPresenters.insert(categoryType, p);
    }
    return it.value();
}
```

### 2.2 Pimpl 惯用法（Private 数据类）

资源区的 Presenter 和 View 均使用 **Q_D/Q_DECLARE_PRIVATE** 模式将实现细节隐藏到 Private 类：

```
FFMediaLibraryPresenter（.h 只暴露接口）
  └── FFMediaLibraryPresenterPrivate（.cpp 内部持有所有数据）
        ├── m_pMediaLibrary: IFFMediaLibrary*
        ├── m_oCurrentCategoryType: FFMediaCategoryType
        └── m_oCategoryPresenters: QHash<CategoryType, CategoryPresenter*>

FMediaLibraryView（.h 只暴露接口）
  └── FMediaLibraryViewPrivate（.cpp 内部持有所有数据）
        ├── m_pLibraryPresenter: FFMediaLibraryPresenter*
        ├── m_pTabBar: FMediaTabBar*
        ├── m_mediaCategoryView: FMediaCategoryView*
        └── m_pEventHandler: FMediaLibraryViewEventHandler*
```

好处：
1. 减少头文件依赖，避免 FFMediaLibraryPresenter.h 污染大量包含它的文件
2. 提高编译速度（Private 成员变更不触发 FFMediaLibraryPresenter.h 的级联重编译）

### 2.3 观察者 + 多接口实现

`FMediaLibraryViewEventHandler` 同时实现 **5 个 Observer 接口**，集中处理来自不同子系统的事件：

```cpp
// FMediaLibraryView_P.h
class FMediaLibraryViewEventHandler :
    public IFAudioPlayerEventObserver,          // 音频播放回调
    public IFFBeatDetectionEventObserver,       // 拍点检测回调
    public IFFMediaProxyEventObserver,          // 代理文件创建回调
    public IFFProjectManagerEventObserver,      // 工程变更回调
    public IFFMediaFolderRedDotManagerEventObserver // 红点状态回调
{
public:
    void onAudioPlayerServiceEvent(const FFAudioPlayerEventParam& param) override;
    void onBeatDetectionProgressEvent(IFFAbstractMediaItem*, FFTaskCallbackParam) override;
    void onMediaProxyProgressEvent(IFFAbstractMediaItem*, FFTaskState, int progress) override;
    void onProjectClipChangedEvent() override;
    void onReloadEvent(int categoryType, const QString& categorySlug) override;
    void onHideRedDotChangedEvent(int categoryType, const QString& categorySlug) override;
};
```

**设计意图**：将 Observer 从 `FMediaLibraryViewPrivate` 中抽离为单独类，避免 Private 类的职责膨胀，同时隔离不同子系统的事件处理逻辑。

### 3.1 工厂方法模式

媒体库通过**工厂方法**创建，外部模块仅持有接口指针，不关心实现类：


## 四、过滤系统（Filter Architecture）

### 4.1 三层过滤设计

```
用户操作（搜索/筛选/分类切换）
        │
        ▼
FFMediaItemManagerPresenter::setFilter*(...)
        │
        ▼
FFMediaItemDataFilterScene（过滤场景协调器）
  ├── m_pMediaItemFilterConfig: IFFMediaItemFilterConfig*  ← 当前 Category 的过滤配置
  ├── m_baseMediaItemFilterList: MediaItemFilterList       ← 激活的过滤器链
  └── processByFilter(inData) → outData                   ← 对本地数据执行过滤
        │
        ▼
IFFMediaItemFilterConfig::checkFilters(folder, searchFilterType)
  判断 searchFilterType:
  ├── LocalFilter       → loadSwitchFolder()（VBL 本地切换，无网络请求）
  ├── OnlineFilter      → startOnlineRequest()（VBL 触发搜索 API）
  └── OnlineFilterSignal → startOnlineRequest() + emit sigLoadFolder()
```

### 4.2 过滤器接口体系

```cpp
// IFFMediaItemFilterConfig.h — 过滤配置接口（每个 Category 一套）
class IFFMediaItemFilterConfig {
    virtual void initFilters() = 0;
    virtual MediaItemFilterList findFilters(IFFFolderMedia*) = 0;   // 初始化时
    virtual MediaItemFilterList checkFilters(IFFFolderMedia*,
        FFMEDIALIBRARY::SearchFilterType&) = 0;                     // 每次加载时

```

**按分类实现的过滤配置：**

| 实现类 | 对应 Category |
|--------|-------------|
| `FFCommonMediaItemFilterConfig` | 本地媒体（mctMedia） |
| `FFEffectMediaItemFilterConfig` | 视频特效（mctEffect） |
| `FFStockMediaItemFilterConfig` | Filmstock 在线素材 |
| `FFAudioMediaItemFilterConfig` | 音频（bpm/时长/类型） |
| `FFTemplatesMediaItemFilterConfig` | 项目模板（比例/片段数） |
| `FFVCGMediaFolderFilterConfig` | VCG 版权素材 |

### 4.3 过滤执行流程

```cpp
// FFMediaItemDataFilterScene.cpp
void FFMediaItemDataFilterScene::loadResources(bool switchCategory)
{
    // 1. 根据当前 folder 查询激活的过滤器列表
    FFMEDIALIBRARY::SearchFilterType searchFilterType = FFMEDIALIBRARY::LocalFilter;
    updateMediaItemFilterList(
        m_pMediaItemFilterConfig->checkFilters(m_pCurrentFolder, searchFilterType),
        false);

    // 2. 将过滤参数注入每个过滤器
    for (auto pFilter : m_baseMediaItemFilterList)
    {
        pFilter->setFolder(m_pCurrentFolder);
        pFilter->filterParam(m_pCurrentFolder, m_searchParam);
    }

    // 3. 根据过滤类型决定走本地还是在线请求
    switch (searchFilterType)
    {
    case LocalFilter:
        loadSwitchFolder();     // IFFMediaItemManager::switchFolder() — 纯本地
        break;
    case OnlineFilter:
        startOnlineRequest();   // switchFolder(searchFolder) — 触发网络搜索
        break;
    case OnlineFilterSignal:
        startOnlineRequest();
        emit sigLoadFolder();   // 同时通知 UI 刷新当前帧
        break;
    }
}
```

## 七、缩略图缓存系统（FFThumbnailCache）

### 7.1 缓存架构

```
请求缩略图（getThumbnailBySize）
        │
        ▼
containsThumbnailWithSize() 检查内存缓存
  ├── 命中 m_thumbnailPixmapCache (QCache<String, QPixmap>)
  │      └── 直接返回 QImage（通过 toImage() 转换）
  │
  └── 未命中
        │
        ▼
loadThumbnailToCache(param)
  ├── 查 m_thumbnailCache (QCache<String, String>) 获取缩略图文件路径
  │     └── 未命中 → 调用 mediaItem->thumbnailFile() 从 VBL 获取路径
  │
  └── FFAsync::postReplyableResultTask([sz, thumbnailCacheId, filePath](){
            // ─── 后台线程 ───
            QImageReader reader;
            reader.setFileName(thumbnailFile);
            reader.setAutoTransform(true);
            QImage image = reader.read();
            // 2× 缩放（高 DPI 模式）
            QImage scaledImage = image.scaled(sz * 2.0, KeepAspectRatio, SmoothTransformation);
            return std::make_tuple(mediaId, filePath, cacheKey, QPixmap::fromImage(scaledImage));
        }).then([this](const KV& result){
            // ─── 主线程 ───
            m_thumbnailCache.insert(mediaId, filePath);
            m_thumbnailPixmapCache.insert(cacheKey, pixmap);
            // 调用 updateCallback 通知 View 刷新
        });
```

### 7.2 双层缓存设计

| 缓存层 | 类型 | Key | Value | 容量上限 |
|--------|------|-----|-------|---------|
| 路径缓存 | `QCache<QString, QString>` | mediaId | 缩略图文件路径 | 2000 条 |
| 像素图缓存 | `QCache<QString, QPixmap>` | `mediaId_WxH` | 缩放后的 QPixmap | 1500 条 |

**Key 设计**：像素图缓存的 Key 带尺寸后缀（`mediaId_120x90`），同一素材的不同显示尺寸分开缓存，避免重复缩放。

### 7.3 特殊素材 Key 生成

```cpp
QString FFThumbnailCache::getThumbnailCacheId(const IFFAbstractMediaItem* mediaItem) const
{
    // 场景检测片段：使用 mediaId + 场景序号（00, 01, 02…）
    if (auto* sceneItem = mediaItem->AsIFFVideoSceneMediaItem())
    {
        if (sceneItem->getSceneMediaItemType() == FFVBLMODEL::SceneDetect)
            return QString("%1%2").arg(mediaId).arg(sceneIndex, 2, 10, QChar('0'));
        // 智能短片段：使用 mediaId/type_start_end
        return QString("%1/%2_%3_%4")
            .arg(mediaId).arg(sceneMediaItemType).arg(sceneStart).arg(sceneEnd);
    }
    return mediaItem->mediaId();  // 普通素材：直接用 mediaId
}
```

### 7.4 迁移轨迹（含已注释代码的设计意图）

代码中可见已注释掉的旧实现方案，反映了缩略图缓存的两次重构：

| 版本 | 方案 | 已弃用原因 |
|------|------|-----------|
| v1 | `QFutureWatcher + QThreadPool` | 线程管理复杂，定时器刷新 UI 不精确 |
| v2 | `QTimer::timeout + QThreadPool::setMaxThreadCount(8)` | 同上，已注释 |
| v3（现行） | `FFAsync::postReplyableResultTask + QtPromise` | 统一异步框架，Promise 链清晰 |

```cpp
// FFVideoPreviewHelper.cpp — 单帧缩略图（用于首帧展示）
QtPromise::QPromise<QPixmap> FFVideoPreviewHelper::getThumbnail(
    const QString& videoPath, int width, int height, int nFrameIndex)
{
    // 1. 创建临时 VBL 时间线
    FFTimelineBuilder oBuilder(false);
    auto pTimeline = oBuilder.build();
    pTimeline->setConfigInfo({ art1_1, frt25 });

    // 2. 添加视频 Clip
    FFVideoClipBuilder clipBuilder(FFVideoClipBuilder::VideoClip, videoPath);
    auto pClip = clipBuilder.build();
    pTimeline->addClip(0, 0, pClip.data(), totOverride);

    // 3. 异步获取指定帧
    return FFAsync::postReplyableResultTask(
        [pTimeline, nFrameIndex, width, height]() {
            int nRealFrame = std::min(nFrameIndex, pTimeline->duration());
            return QPixmap::fromImage(
                pTimeline->getImage(nRealFrame, QSize(width, height)));
        });
}
```

```
   trackOnDownloadBegin(mediaItem)      ← 上报埋点
```

## 十六、面试常见问题与答案

### Q1：资源区的 MVP 架构是如何分工的？

**A：** 资源区采用三层嵌套 MVP 架构：

- **Model 层**：`IFFMediaLibrary`（VBL 接口），通过工厂方法创建，外部只依赖接口不依赖实现
- **View 层**：`FMediaLibraryView`（顶层容器）、`FMediaCategoryView`（Tab 内容）、`FMediaFolderView`（目录树）、`FMediaItemView`（素材列表）、`FMediaItemStackedView`（核心堆叠视图）
- **Presenter 层**：`FFMediaLibraryPresenter` → `FFMediaCategoryPresenter` → `FFFolderMediaManagerPresenter` + `FFMediaItemManagerPresenter`，与 UI 层一一对应

三层嵌套对应 UI 的三层嵌套，每层 Presenter 只负责对应 View 的数据逻辑，清晰隔离职责。

---

### Q2：缩略图异步加载如何实现？为什么不用 QFutureWatcher？

**A：** 现行方案使用 `FFAsync::postReplyableResultTask + QtPromise`，原因：

1. `QFutureWatcher` 需要手动管理 Watcher 生命周期（`watcher->deleteLater()`），且结果回调仍需手动跳回主线程；
2. `FFAsync::postReplyableResultTask` 是项目统一异步框架，底层已处理线程调度和异常；
3. `QtPromise::then()` 回调天然在主线程执行（Qt Promise 规范），代码更清晰；
4. 早期代码（已注释）就是用 `QFutureWatcher + QTimer`，实测有定时误差和线程安全问题。

缩略图缓存分两层：路径缓存（`QCache<String, String>`，2000 条）+ 像素图缓存（`QCache<String, QPixmap>`，1500 条），后者 Key 带尺寸后缀，支持同一素材多种显示尺寸分别缓存。

---

### Q3：过滤系统如何区分本地过滤和在线搜索？

**A：** 通过 `FFMEDIALIBRARY::SearchFilterType` 枚举决定：

- `LocalFilter`：调用 `IFFMediaItemManager::switchFolder()`，纯本地数据切换，无网络请求；
- `OnlineFilter`：调用 `switchFolder(searchFolder)` 触发 VBL 网络搜索 API；
- `OnlineFilterSignal`：同 `OnlineFilter`，额外 emit `sigLoadFolder()` 立即刷新当前帧数据。

判断逻辑在 `IFFMediaItemFilterConfig::checkFilters()` 中由各分类实现类决定。例如本地媒体文件夹切换为 `LocalFilter`，Filmstock 关键字搜索为 `OnlineFilter`。

---

### Q4：Delegate + Painter 分离设计的好处是什么？

**A：** Qt 标准的 `QStyledItemDelegate` 将事件处理和绘制混在一起。资源区将绘制抽离为 `IFFItemBasePainter` 体系：

- **Delegate（`FFCustomDelegate`）**：只处理 Mouse/Keyboard 事件，不含任何绘制代码；
- **Painter（`FBaseTableItemPainter` 及子类）**：纯绘制逻辑，不处理任何事件；
- 不同 Category 的格子使用不同的 Painter 实现（特效格子/VCG 格子/音频格子…），通过 Painter 工厂按需创建，不修改 Delegate；
- 方便 A/B 测试：替换 Painter 即可改变格子样式，不影响其他逻辑。

---

### Q5：FMediaItemStackedView 的状态管理如何设计？

**A：** `FMediaItemStackedView` 内部维护 `MediaItemViewState` 枚举（16 个状态），通过内部方法 `setViewState()` + `QStackedWidget` 切换：

- `mivsNormal`：正常显示素材格子（`FMediaItemIconView / FMediaItemTableView`）
- `mivs1THLoding`：第一页加载中（旋转动画 Widget）
- `mivsEmptyEffect/mivsImportMedia`：空状态（引导用户操作）
- `mivsStartAIMusic / mivsStartAiImage`：AI 功能入口页（非素材列表，是功能面板）

各状态对应不同的 `QStackedWidget` 页面，通过 `m_pContentStackedWdg->setCurrentIndex()` 切换，比 `show()/hide()` 多个 Widget 更高效。

---

### Q6：如何处理 Filmstock/VCG 等在线素材的分页加载？

**A：** 在线素材分页加载基于 `IFFFolderMedia::hasNextPage` 标记：

1. `FMediaItemStackedView::hasReachedEnd()` 判断列表是否滚动到底部
2. 触底后 emit `slotFetchMore()` → `FFMediaItemManagerPresenter::fetchMore()`
3. 调用 `IFFMediaItemManager::loadNextPage()`，VBL 请求下一页 API
4. 数据返回后 `onAfterSearchAllMediaChangedFinishEvent()` 回调 → `FFMediaItemModel::appendRows()`
5. View 自动渲染新增条目（Qt MVC 机制）

注意：收藏夹有特殊处理——连续两次 `switchFolder(favoriteFolder)` 且 VBL 返回"有下一页"时，应用层需自己管理状态防止混乱，代码注释中有明确说明。
