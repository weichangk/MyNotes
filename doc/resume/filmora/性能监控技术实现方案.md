## 1. 整体架构与进程模型

系统由三个独立进程构成，通过 **共享内存 + IPC** 通信：

```
Filmora 主进程
├── FFPerformanceClient（嵌入库）
│   ├── 写入 QSharedMemory（双缓冲 A/B）
│   └── IPC → Performance 进程（心跳、写满通知、初始化数据）
└── FEventTrackingClient（IPC 客户端）
    └── IPC → FEventTracking 子进程

Performance 进程（Performance.exe）
├── FFPerformanceServer
│   ├── 读取 QSharedMemory（双缓冲 A/B）
│   ├── PDH / Mac API 采样
│   ├── 卡顿检测 → MiniDumpWriteDump
│   └── 写 WESPerformance.json
└── FFPerformanceRetrieval
    └── AlibabaCloud OSS 上传

FEventTracking 子进程
├── FEventTrackingWorker（埋点上报）
└── FWESPerformanceTrack（上报 WESPerformance.json）
```

## 2. 共享内存协议：二进制布局与对齐

### 2.1 强制 1 字节对齐

所有跨进程共享的结构体使用 `#pragma pack(1)` 强制 1 字节对齐，消除编译器填充字节，保证两个进程（可能使用不同编译选项）读写同一块内存时字段偏移完全一致：

```cpp
// FFPerformanceDefine.h
#pragma pack(push)
#pragma pack(1)  // 设置 1 字节对齐

struct FPMemoryHeader
{
    int segmentCount;   // 缓冲区容量（段数）
    int writeIndex;     // 写入游标（预留）
    int readIndex;      // 读取游标（预留）
    int headerLength;   // header 自身字节数（= sizeof(FPMemoryHeader)）
    int segLength;      // 单条 FPSegmentData 字节数（= sizeof(FPSegmentData)）
    int writeCount;     // 当前已写入段数（实际使用的写计数）
    char nState;        // 状态标志位（STATE_NONE/READ/READDONE/ERROR）
};

struct FPSegmentData
{
    quintptr objectAddress; // Qt 对象指针（4B on x86, 8B on x64）
    int      eventType;     // 事件类型枚举
    quint64  eventTime;     // 时间戳（100ns 精度）
    int      level;         // 嵌套层级
    char     state;         // 附加状态
};

#pragma pack(pop)  // 恢复原有对齐
```

### 2.2 内存布局（x64 下的字节偏移）

**FPMemoryHeader**（共 29 字节，pack(1)）：

```
偏移  大小  字段
 0     4   segmentCount
 4     4   writeIndex
 8     4   readIndex
12     4   headerLength
16     4   segLength
20     4   writeCount
24     1   nState
        ──── 合计 25 字节（实际以 headerLength 字段值为准）
```

**FPSegmentData**（x64 下 8+4+8+4+1 = 25 字节，pack(1)）：

```
偏移  大小  字段
 0     8   objectAddress（x64 指针）
 8     4   eventType
12     8   eventTime（quint64）
20     4   level
24     1   state
        ──── 合计 25 字节
```

**共享内存总体布局**：

```
[  FPMemoryHeader  ][  FPSegmentData[0]  ][  FPSegmentData[1]  ] ... [  FPSegmentData[N-1]  ]
  headerLength 字节    segLength 字节         segLength 字节               segLength 字节

总大小 = sizeof(FPMemoryHeader) + segmentCount × sizeof(FPSegmentData)
```

数据段写入定位使用指针偏移运算：

```cpp
// FFPerfomanceMemory.cpp
void FFPerfomanceMemory::writeToIndex(FPMemoryHeader* header,
                                      const FPSegmentData& data,
                                      int index)
{
    char* segPos = (char*)header + header->headerLength + index * header->segLength;
    *reinterpret_cast<FPSegmentData*>(segPos) = data;
    header->writeCount = index + 1;
}
```

### 2.3 共享内存键与容量

```cpp
// FFPerfomanceMemory.cpp
static const QString g_memory_keyA = "{C0B29054-5FB9-4C58-BEB2-3E547E96164D}";
static const QString g_memory_keyB = "{CA0E0D5A-894D-4396-A6B8-FE2BF3E0AA9B}";
static int g_segmentSize     = 4096;  // 初始容量（段数）
static const int g_minSegmentSize = 1024;  // 容量下限
```

两个固定 GUID 作为共享内存名称（Windows 下映射为命名共享内存对象），确保跨进程能 attach 到同一块内存。

### 2.4 nState 状态标志

```cpp
// FFPerformanceDefine.h（相关宏）
#define STATE_NONE      0x00
#define STATE_WRITE     0x01   // （预留）
#define STATE_ENTER     0x02   // （预留）
#define STATE_READ      0x04   // 消费者正在读取
#define STATE_READDONE  0x08   // 消费者读取完毕
#define STATE_ERROR     0x10   // 发生错误，停止写入
```

状态流转：

```
[生产者] STATE_NONE
    │ writeSegmentData 写入各条目
    │ writeCount 达到 segmentCount → emit sigWriteFull()
    ▼
[消费者] 读取前：nState |= STATE_READ
    │ 逐条回调 callback(data[i], i)
    │ 清零 writeCount
    ▼
[消费者] 读完后：nState = STATE_READ | STATE_READDONE
    │ 切换当前 m_currMemory 到另一个缓冲
    ▼
[生产者] 切换到另一缓冲后：检查 STATE_READDONE 确认可用
    nState = STATE_NONE → 重新开始写入
```

## 3. 双缓冲共享内存的并发控制

### 3.1 双缓冲设计

使用两块共享内存（A、B），通过 `m_currMemory` 指针切换当前活跃缓冲：

```cpp
// FFPerfomanceMemory.h
QSharedMemory  m_memoryA;   // Buffer A
QSharedMemory  m_memoryB;   // Buffer B
QSharedMemory* m_currMemory = nullptr;  // 当前活跃缓冲（指向 A 或 B）
```

### 3.2 attach/create 流程（含重试降级）

```cpp
// FFPerfomanceMemory.cpp
bool FFPerfomanceMemory::attachOrCreateMemory(QSharedMemory* memory, int state)
{
    if (memory->isAttached())
        memory->detach();

    // 优先尝试 attach（另一个进程已经创建）
    if (memory->attach())
        return true;

    // attach 失败则尝试 create，容量不足时逐次减半重试
    while (g_segmentSize >= g_minSegmentSize)
    {
        int size = sizeof(FPMemoryHeader) + g_segmentSize * sizeof(FPSegmentData);
        if (memory->create(size))
        {
            // 直接在共享内存首部写入 header 初始值
            auto* header = reinterpret_cast<FPMemoryHeader*>(memory->data());
            header->segmentCount = g_segmentSize;
            header->writeCount   = 0;
            header->headerLength = sizeof(FPMemoryHeader);
            header->segLength    = sizeof(FPSegmentData);
            header->nState       = state;
            return true;
        }
        QT_LOG_WARN << "create memory failed A: " << m_memoryA.errorString()
                    << "size : " << size;
        g_segmentSize /= 2;  // 容量减半后重试（最低降至 1024 段）
    }
    return false;
}
```

**关键点**：`g_segmentSize` 是全局静态变量，一旦降级后对同进程内后续创建都生效。

### 3.3 生产者写入（writeSegmentData）完整实现

```cpp
// FFPerfomanceMemory.cpp
void FFPerfomanceMemory::writeSegmentData(const FPSegmentData& data)
{
    auto* header = getCurrentHeader();
    m_currMemory->lock();   // ← 跨进程互斥锁（Qt 映射到 OS 信号量/互斥）

    if (header->nState & STATE_ERROR)
    {
        // 缓冲已进入错误状态，拒绝写入，直接返回
        m_currMemory->unlock();
        return;
    }

    if ((header->nState & STATE_READ) || header->writeCount >= header->segmentCount)
    {
        // 情况1：消费者正在读取当前缓冲
        // 情况2：当前缓冲已写满
        // → 尝试切换到另一个缓冲
        m_currMemory->unlock();
        header = changeSharedMemory();  // 切换 m_currMemory 指向另一个缓冲
        m_currMemory->lock();

        if (!(header->nState & STATE_READDONE))
        {
            // 另一个缓冲尚未被消费者读完 → 无法安全写入，标记错误
            header->nState |= STATE_ERROR;
            m_currMemory->unlock();
            return;
        }
        // 另一缓冲可用：重置状态，从索引 0 开始写
        header->nState = STATE_NONE;
        writeToIndex(header, data, 0);
        m_currMemory->unlock();
        return;
    }

    // 当前缓冲有空间，正常追加写入
    writeToIndex(header, data, header->writeCount);
    m_currMemory->unlock();

    // 写满时通知消费者（Qt 信号，跨线程安全）
    if (header->writeCount == header->segmentCount)
    {
        emit sigWriteFull();
    }
}
```

### 3.4 消费者读取（readSegmentData）完整实现

```cpp
// FFPerfomanceMemory.cpp
void FFPerfomanceMemory::readSegmentData(const ReadSegmentDataCallBack& callback)
{
    auto* header = getCurrentHeader();
    m_currMemory->lock();

    if (header->writeCount == 0)
    {
        m_currMemory->unlock();
        return;  // 没有待读数据
    }

    // 先标记 STATE_READ，阻止生产者继续向此缓冲写入
    header->nState |= STATE_READ;   // 注意：保留原有 STATE_ERROR 位
    m_currMemory->unlock();

    // 在锁外做回调（避免长时间持锁阻塞生产者）
    // 直接通过指针偏移访问条目（零拷贝）
    FPSegmentData* data = reinterpret_cast<FPSegmentData*>(
        (char*)m_currMemory->data() + sizeof(FPMemoryHeader));
    for (int index = 0; index != header->writeCount; ++index)
    {
        callback(data[index], index);
    }
    header->writeCount = 0;  // 清空写计数（在锁外操作，依赖 STATE_READ 阻挡生产者）

    m_currMemory->lock();
    header->nState = STATE_READ | STATE_READDONE;  // 强制覆盖 STATE_ERROR，标记读取完成
    m_currMemory->unlock();

    // 消费者切换到另一个缓冲（下次读取从新缓冲取数据）
    m_currMemory = (m_currMemory == &m_memoryA) ? &m_memoryB : &m_memoryA;
}
```

**注意**：`header->writeCount = 0` 在 `lock` 之外执行。此时 `STATE_READ` 已设置，生产者检查到该标志后会切换自己的缓冲，不会再访问此 `writeCount`，因此在当前协议下是安全的。

### 3.5 强制读取（forceReadSegmentData）

用于主进程崩溃/断连后服务端强制回收最后数据：

```cpp
void FFPerfomanceMemory::forceReadSegmentData(const ReadSegmentDataCallBack& callback)
{
    auto* header = getCurrentHeader();
    if (header->writeCount == 0)
        return;
    // 不加锁、不修改状态，直接读取现有数据
    FPSegmentData* data = reinterpret_cast<FPSegmentData*>(
        (char*)m_currMemory->data() + sizeof(FPMemoryHeader));
    for (int index = 0; index != header->writeCount; ++index)
    {
        callback(data[index], index);
    }
}
```

## 4. 时间戳机制：steady_clock 与 system_clock 对齐

### 4.1 时间宏定义

```cpp
// FFPerformanceDefine.h
// 单位：100 纳秒（与 Windows FILETIME 兼容）

#define STEAAY_CLOCK_CURR_TIME \
    ((std::chrono::steady_clock::now().time_since_epoch().count()   \
      * std::chrono::steady_clock::period::num * 10000000)          \
     / std::chrono::steady_clock::period::den)

#define SYSTEM_CLOCK_CURR_TIME \
    ((std::chrono::system_clock::now().time_since_epoch().count()   \
      * std::chrono::system_clock::period::num * 10000000)          \
     / std::chrono::system_clock::period::den)
```

换算原理：`count() × (num/den)` 将 `duration` 转为秒，再乘以 `10^7` 得到 100ns 刻度值。

### 4.2 偏移对齐算法

```cpp
// FFPerformanceClient 构造函数
m_system2steady = SYSTEM_CLOCK_CURR_TIME - STEAAY_CLOCK_CURR_TIME;

// currentTime() 实现
int64_t FFPerformanceClient::currentTime()
{
    return STEAAY_CLOCK_CURR_TIME + m_system2steady;
}
```

| 时钟类型 | 特性 | 用途 |
|---------|------|------|
| `steady_clock` | 单调递增，不受系统时间修改影响 | 事件时间戳（保证时序不回拨） |
| `system_clock` | 与实际日历时间对齐，可能跳变 | 日志与上报中的绝对时间 |
| `m_system2steady` | 构造时计算一次的偏移常量 | 将 steady 时间戳对齐到系统时间 |

## 5. 主进程侧采集（FFPerformanceClient）

### 5.1 崩溃信号处理器注册

初始化时注册 `SIGSEGV` 处理函数，在主线程崩溃时向共享内存写入 `CrashThreadType` 条目：

```cpp
// FFPerformanceClient.cpp
void signalHandler(int sig)
{
    static bool bFirst = true;
    if (bFirst && g_client)
    {
        bFirst = false;
#ifdef Q_OS_WIN
        auto crash_thread_id = ::GetCurrentThreadId();
        if (g_main_thread_id == crash_thread_id)
        {
            g_client->writeCrashThreadId();  // 写入共享内存，让 Performance 进程感知崩溃
        }
#endif
    }
}
```

`static bool bFirst` 保证只处理第一次信号，避免崩溃处理器内部异常导致无限递归。

### 5.2 主线程心跳（100ms 定时器）

```cpp
// FFPerformanceClient.cpp
void FFPerformanceClient::onMainHeartBeat()
{
    // 1. 通过 IPC 向 Performance 进程发送带时间戳的心跳事件
    auto pProcess = F_IPC_SERVER->findProcess(Performance);
    if (pProcess)
    {
        auto event = FIPCSendMainHeartBeatEvent(currentTime());
        pProcess->sendEvent(&event);
    }

    // 2. 同时向共享内存写入 MainHeartBeatType 条目（双保险）
    FPSegmentData data;
    data.objectAddress = 0;
    data.eventType     = MainHeartBeatType;
    data.eventTime     = currentTime();
    data.isEnter       = true;
    m_memory->writeSegmentData(data);
}
```

### 5.3 Performance.exe 启动防竞态（QLockFile）

```cpp
// FFPerformanceClient.cpp
void FFPerformanceClient::startPerformanceExe()
{
    QLockFile lockFile(g_performance_exe_lock);
    if (!lockFile.tryLock())
    {
        // 另一个线程/进程正在启动 Performance.exe
        if (m_bConnectDone == false)
            QTimer::singleShot(1000, this, &FFPerformanceClient::startPerformanceExe);
```

## 6. Performance 服务进程（FFPerformanceServer）

### 6.1 Dump 生成（Windows）

```cpp
// FFPerformanceServer.cpp

// 底层 Dump 写入：调用 Win32 MiniDumpWriteDump API
int __GenerateDump(int processId, HANDLE processHandle, const std::string& path)
{
    HANDLE hFile = ::CreateFileA(
        path.c_str(),
        GENERIC_WRITE, 0, NULL,
        CREATE_ALWAYS, FILE_ATTRIBUTE_NORMAL, NULL);

    if (INVALID_HANDLE_VALUE != hFile)
    {
        ::MiniDumpWriteDump(
            processHandle,          // 目标进程句柄（通过 OpenProcess 获取）
            processId,              // 目标进程 PID
            hFile,                  // 输出文件句柄
            MINIDUMP_TYPE::MiniDumpNormal,  // Dump 类型
            NULL, nullptr, nullptr);
        ::CloseHandle(hFile);
    }
    return EXCEPTION_EXECUTE_HANDLER;
}

// 上层封装：检查文件是否已存在，避免重复生成
bool __Snapshot(int processId, HANDLE processHandle, const QString& path)
{
    if (QFile(path).exists())
        return false;  // 已存在则跳过
    __Snapshot2(processId, processHandle, path.toStdString());
    return true;
}
```

### 6.3 心跳卡顿检测算法

```cpp
// FFPerformanceServer.cpp — readMainHeartBeat()
void FFPerformanceServer::readMainHeartBeat(quint64 heartTime)
{
    quint64 orgHeartTime = m_lastHeartTime;
    quint64 time = heartTime - m_lastHeartTime;
    m_lastHeartTime = heartTime;

    // 排除系统挂起/唤醒（避免误判）
    if (time > m_config.heartExceptionMin * 10000
        && !qInBound(orgHeartTime, m_suspendTime, m_lastHeartTime))
    {
        // 在 [orgHeartTime, heartTime] 区间内找耗时最长的场景节点
        auto data = getBetweenHeartData(orgHeartTime, m_lastHeartTime);
        if (data.objectEvent.stackObj != 0)
        {
            QT_LOG_INFO << "Heart Stack:"    << m_infoMapper[data.objectEvent.stackObj]
                        << "HeartInterval:"  << (time / m_config.timePrecision)
                        << "EventType:"      << getEventName(data.objectEvent.eventType)
                        << "TotalTime:"      << (data.totalTime / m_config.timePrecision)
                        << "Count:"          << data.count
                        << "Total Count:"    << data.totalEventCount;

            QString shortName = getShortName(data.objectEvent.stackObj,
                                             data.objectEvent.eventType);
            sendHeartStuckEvent(shortName,
                                m_infoMapper[data.objectEvent.stackObj],
                                data.totalTime / 10000,
                                data.count);
        }
    }
}
```

**挂起补偿**：`m_suspendTime` 记录系统挂起时间窗口，`qInBound(orgHeartTime, m_suspendTime, m_lastHeartTime)` 排除落在挂起窗口内的"假卡顿"。

### 6.4 异常断连后的数据回收

当 IPC 断开（主进程崩溃或正常退出）时，服务端执行最终数据回收：

```cpp
// FFPerformanceServer.cpp — onClientQuitDown()
void FFPerformanceServer::onClientQuitDown(int error)
{
    // 使用 forceReadSegmentData 强制读取共享内存中的剩余条目
    // （不加锁、不修改状态，直接读取 —— 避免对端进程死亡后锁无法释放导致死锁）
    m_memory->forceReadSegmentData([=](const FPSegmentData& data, int index) {
        recordSceneNode(data, index);
    });

    // 汇总统计数据，写入 WESPerformance.json
    // 通过 QTimer::singleShot 延迟退出，确保 I/O 写入完成
    QTimer::singleShot(m_config.exitDelay, this, [=]() {
        // 上传日志 zip / 触发埋点上报
        QCoreApplication::quit();
    });
}
```

```cpp
pgmData.cpuUsage   = m_PDHCounter.GetCounterValue(processMetricsMap[pmProcessorTime],
                                                   PDH_FMT_LONG).longValue;
pgmData.workingSet = m_PDHCounter.GetCounterValue(processMetricsMap[pmWorkingSet],
                                                   PDH_FMT_DOUBLE).doubleValue / BYTES_IN_MB;

cptData.cpuUsage    = m_PDHCounter.GetCounterValue(computeMetricsMap[cmCpuUsage],
                                                    PDH_FMT_DOUBLE).doubleValue;

return { pgmData, cptData };
```

## 8. 卡顿检测与 MiniDump 生成

完整流程：

```
主进程 100ms 心跳 ──────────────────────────────────┐
                                                      │ IPC
Performance 进程接收 FIPCSendMainHeartBeatEvent        │
    │                                                  │
    ├── 记录 m_lastHeartTime = heartTime               │
    │                                                  ▼
    └── readMainHeartBeat(heartTime)           共享内存写 MainHeartBeatType

定时器（m_stuckTimer）触发 onTimeStuck()
    │
    ├── 计算 elapsed = currentTime - m_lastHeartTime
    │
    └── elapsed > 卡顿阈值（heartExceptionMin × 10000）
        AND 不在挂起窗口内（!qInBound(...)）
        │
        ├── [Windows] QtConcurrent::run → __Snapshot()
        │   ├── CreateFileA（创建 .dmp 文件）
        │   └── MiniDumpWriteDump(processHandle, pid, hFile, MiniDumpNormal, ...)
        │
        └── [macOS] sendGrabStuckStackEvent → 主进程抓调用栈
```

**OpenProcess 调用时机**：在收到 `sendInitDataEvent`（含主进程 PID）时，服务进程调用：

```cpp
m_processHandle = ::OpenProcess(PROCESS_ALL_ACCESS, FALSE, m_appData.processId);
```

持有此句柄直到主进程退出，以便在检测到卡顿时随时生成 Dump。

### 10.2 WESPerformance.json 上报（后台线程）

```cpp
// FWESPerformanceTrack.cpp
void FWESPerformanceTrack::run()
{
    const auto future = QtConcurrent::run([&]()
    {
        auto logPath = FF_GET_APP_INFO(IFFAppInfo::aitAppLogDir) + "/WESPerformance.json";
        QFileInfo fi(logPath);

        if (fi.exists())
        {
            QFile file(logPath);
            if (file.open(QIODevice::ReadOnly))
            {
                QByteArray buff = file.readAll();
                file.close();
                file.remove();  // 读完立即删除，防止重复上报

                QJsonDocument _doc = QJsonDocument::fromJson(buff);
                QJsonObject   obj  = _doc.object();

                if (obj.contains("Events"))
                {
                    QJsonArray  jsonArr   = obj.value("Events").toArray();
                    QStringList eventList;

                    for (auto value : jsonArr)
                    {
                        QJsonDocument oJsonDoc(value.toObject());
                        eventList.append(
                            QString(oJsonDoc.toJson(QJsonDocument::Compact)));
                    }
                    m_pClient->sendTrackEvent(eventList);  // 发送给埋点后端
                }
            }
        }
    });
    m_futures.addFuture(future);  // 追踪 future，析构时等待完成
}
```

**读完即删**的策略意味着：若 `sendTrackEvent` 失败，数据不可恢复（见第 12 节风险分析）。

## 12. 关键并发风险与工程设计决策

### 12.1 已知风险

| 风险 | 位置 | 说明 |
|------|------|------|
| **STATE_ERROR 无恢复** | `FFPerfomanceMemory::writeSegmentData` | 双缓冲同时满足"被读"且"另一缓冲未读完"时，缓冲被标记 `STATE_ERROR` 后永久停止写入，无重置机制 |
| **writeCount 锁外清零** | `readSegmentData` | `header->writeCount = 0` 在 `unlock()` 后执行，依赖 `STATE_READ` 阻挡生产者；若生产者绕过 `STATE_READ` 检查，存在竞态 |
| **std::localtime 非线程安全** | `FFSinks::serializeTimePoint` | 多线程并发写日志时时间格式化结果可能被覆盖 |
| **WESPerformance.json 读删** | `FWESPerformanceTrack::run` | 读取后立即删除文件，若后续 `sendTrackEvent` 失败则数据永久丢失 |
| **g_segmentSize 全局降级** | `attachOrCreateMemory` | `g_segmentSize /= 2` 修改全局静态变量，影响同进程后续所有创建操作 |
| **MDSF_FORCEEXIT 强制退出** | `FMiniDmpSender` | 上报完成后强制 `exit`，跳过 C++ 析构链，可能丢失未写入磁盘的数据 |

### 12.2 关键设计决策解读

**为什么选双缓冲而非环形队列？**

双缓冲实现更简单，并且与 `QSharedMemory` 的 `lock/unlock` 粒度匹配——整块缓冲区加锁，避免了环形队列需要的细粒度 CAS 操作（这在跨进程场景下 CAS 不直接可用）。代价是在两个缓冲都处于占用状态时发生数据丢失（`STATE_ERROR`），这在正常运行中被认为是可接受的。

**为什么用 `forceReadSegmentData`（不加锁）？**

主进程崩溃后其持有的 `QSharedMemory` 锁（系统信号量）理论上会由 OS 释放，但依赖此行为存在跨平台风险。`forceReadSegmentData` 以"直接读取、不修改状态"的方式规避死锁，但读到的数据可能处于写入中间状态（半写），服务端需对此做容错处理。

**为什么心跳同时走 IPC 和共享内存两个通道？**

IPC 通道（`FIPCSendMainHeartBeatEvent`）是主要的实时通知，延迟低但依赖 IPC 连接存活；共享内存通道（写 `MainHeartBeatType` 条目）是备份，在 IPC 连接断开后服务端仍能通过读取共享内存感知到最后一次心跳时间，用于判断崩溃还是卡死。

**为什么 SEH 回调中只做最小操作？**

崩溃处理路径中的栈可能已经损坏，执行复杂逻辑（内存分配、文件 I/O、Qt 对象操作）风险极高。`signalHandler` 中只调用 `writeCrashThreadId()`（向共享内存写一个固定大小结构体，无内存分配），是在崩溃路径安全操作范围内的合理设计。
