## 1. 整体架构

VBL 多线程并发设计涉及三个层次：

    任务管理层（BsBackgroundTaskManager）：任务队列 + 并发数限制 + condition_variable
    线程池层（StdThreadPool）：固定线程数 + packaged_task + future
    单任务层（BsBackgroundTask）：atomic 取消标志 + shared_ptr<thread>

## 2. BsBackgroundTask — 单个后台任务

文件：modules/BaseService/BackgroundTaskManager/BsBackgroundTask.hpp

    class BsBackgroundTask {
    public:
        using TaskFunc = std::function<void(BsBackgroundTask*)>;
        
        void Start() {
            // shared_ptr<thread> 管理线程生命周期（RAII）
            m_thread = std::make_shared<std::thread>([this]() {
                m_func(this);  // 执行任务，传入 this 供任务检查 cancel 状态
            });
        }
        
        // 请求取消（设置 atomic flag）
        void Cancel() {
            m_cancelled.store(true, std::memory_order_release);
        }
        bool IsCancelled() const {
            return m_cancelled.load(std::memory_order_acquire);
        }
        
    private:
        TaskFunc                       m_func;
        std::atomic<bool>              m_cancelled;  // 无锁取消标志
        std::shared_ptr<std::thread>   m_thread;     // RAII 线程管理
    };

## 3. BsBackgroundTaskManager — 任务队列 + 并发控制

文件：modules/BaseService/BackgroundTaskManager/BsBackgroundTaskManager.hpp

    class BsBackgroundTaskManager {
    private:
        std::deque<std::shared_ptr<BsBackgroundTask>> m_taskQueue;
        std::mutex                 m_mutex;
        std::condition_variable    m_cv;
        std::atomic<int>           m_activeCount{0};     // 当前运行数（原子）
        std::atomic<int>           m_maxConcurrency{4};  // 最大并发数（原子）
        std::thread                m_schedulerThread;
        bool                       m_stopped{false};
        
        void SchedulerLoop() {
            while (!m_stopped) {
                std::unique_lock<std::mutex> lock(m_mutex);
                // condition_variable + predicate 防止虚假唤醒
                m_cv.wait(lock, [this]() {
                    return m_stopped ||
                           (!m_taskQueue.empty() &&
                            m_activeCount.load() < m_maxConcurrency.load());
                });
                if (m_stopped) break;
                
                auto task = m_taskQueue.front();
                m_taskQueue.pop_front();
                lock.unlock();  // 释放锁后再启动任务（避免持锁期间执行任务）
                
                m_activeCount.fetch_add(1, std::memory_order_relaxed);
                task->Start();
            }
        }
    };

关键设计点：
- atomic<int> 控制并发数：无锁，比 mutex+int 更高效
- condition_variable predicate 防虚假唤醒：条件必须同时满足"有任务"且"未超并发"
- notify_all() 在任务完成回调中触发，重新调度等待中的任务

## 4. StdThreadPool — 线程池（BsPluginLoader 使用）

    class StdThreadPool {
    public:
        explicit StdThreadPool(size_t numThreads) {
            for (size_t i = 0; i < numThreads; ++i) {
                m_workers.emplace_back([this]() { WorkerLoop(); });
            }
        }
        
        // 提交任务，返回 future
        template<typename F>
        std::future<std::invoke_result_t<F>> Submit(F&& func) {
            using RetType = std::invoke_result_t<F>;
            auto task = std::make_shared<std::packaged_task<RetType()>>(
                std::forward<F>(func));
            std::future<RetType> fut = task->get_future();
            {
                std::unique_lock<std::mutex> lock(m_mutex);
                m_tasks.push_back([task](){ (*task)(); });
            }
            m_cv.notify_one();
            return fut;
        }
        
    private:
        void WorkerLoop() {
            while (true) {
                std::function<void()> task;
                {
                    std::unique_lock<std::mutex> lock(m_mutex);
                    m_cv.wait(lock, [this]() { return m_stop || !m_tasks.empty(); });
                    if (m_stop && m_tasks.empty()) return;
                    task = std::move(m_tasks.front());
                    m_tasks.pop_front();
                }
                task();  // 锁外执行任务
            }
        }
        
        std::vector<std::thread>          m_workers;
        std::deque<std::function<void()>> m_tasks;
        std::mutex                         m_mutex;
        std::condition_variable            m_cv;
        bool                               m_stop{false};
    };

## 5. 并发插件安装示例（future 的实际应用）

    // modules/BaseService/BsPluginManager/BsPluginLoader.cpp（示意）
    void BsPluginLoader::InstallPlugins(const std::vector<PluginInfo>& plugins) {
        StdThreadPool pool(std::thread::hardware_concurrency());
        
        std::vector<std::future<bool>> futures;
        for (const auto& plugin : plugins) {
            futures.push_back(pool.Submit([plugin]() -> bool {
                return InstallSinglePlugin(plugin);  // 并发安装
            }));
        }
        
        // 等待所有安装完成，收集结果
        int successCount = 0;
        for (auto& f : futures) {
            if (f.get()) ++successCount;  // get() 阻塞等待单个结果
        }
    }

## 6. 线程安全容器（safemap / safelist）

文件：Include/Common/safemap.h

    template<typename K, typename V>
    class safemap {
    public:
        void insert(const K& key, const V& value) {
            std::unique_lock<std::shared_mutex> lock(m_mutex);  // 写独占锁
            m_map[key] = value;
        }
        std::optional<V> get(const K& key) const {
            std::shared_lock<std::shared_mutex> lock(m_mutex);  // 读共享锁
            auto it = m_map.find(key);
            if (it != m_map.end()) return it->second;
            return std::nullopt;
        }
    private:
        std::unordered_map<K, V>  m_map;
        mutable std::shared_mutex m_mutex;  // 读写锁：多读单写
    };

shared_mutex（读写锁）优势：多个线程可以同时读，只有写操作需要独占锁，
适合读多写少场景（如 MediaInfo 缓存、对象池）。

## 7. 面试要点

1. **condition_variable 的标准用法**：BsBackgroundTaskManager 的 wait(lock, predicate)
   是防止虚假唤醒的标准写法。predicate 需要同时检查停止标志和业务条件。

2. **atomic 控制并发数是无锁设计**：m_activeCount.fetch_add/sub 是原子操作，
   控制并发任务数不需要加锁，比 mutex+int 更高效。

3. **packaged_task + future 是优雅的异步结果获取**：Submit() 返回 future，
   调用方可以先做其他工作再 get()，是 proactor 模式的 C++ 标准实现。

4. **shared_ptr<thread> 是线程的 RAII 管理方式**：直接持有 std::thread，
   析构时若未 join 会 std::terminate。shared_ptr<thread> 安全共享线程句柄。

5. **shared_mutex 适合读多写少场景**：safemap 用 std::shared_mutex，
   多线程同时读取不阻塞，写入时才需要独占。MediaInfo 缓存的典型应用。

## 8. 可能被追问的问题

**Q1：condition_variable 为什么需要 predicate？**
防止虚假唤醒（spurious wakeup）。wait(lock, predicate) 等价于：
while (!predicate()) wait(lock)，每次唤醒都重新检查条件，确保条件真正满足。

**Q2：std::future 和 std::async 的区别？**
std::async 自动决定是否创建新线程（实现定义），不精确控制线程来源。
packaged_task 手动决定在哪个线程/线程池执行，适合需要精确控制执行上下文的场景。

**Q3：atomic<bool> 和 mutex+bool 相比有什么优劣？**
atomic<bool>：无锁，适合简单的 flag（取消标志、停止标志），读写非常快。
mutex+bool：适合"检查-修改-操作"原子复合操作的场景。
取消标志只需要写一个值、读一个值，用 atomic 完全够用。

**Q4：条件变量 wait 期间锁是释放的吗？**
wait() 调用时原子地释放锁并阻塞（这是条件变量的关键设计），被唤醒时重新获锁。
notify_one/all 通常在锁释放后调用（先解锁再 notify），减少唤醒后竞争等待的开销。

**Q5：如何避免线程池任务的异常导致程序崩溃？**
WorkerLoop 中的 task() 调用用 try-catch 包裹，捕获异常后通过
promise::set_exception() 传递给 future，调用方在 get() 时重新抛出。
VBL 任务通常返回 VBL::Result 错误码而非抛出异常，减少了此类问题。
