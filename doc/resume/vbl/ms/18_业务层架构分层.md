## 1. 概念介绍

**分层架构**的核心思想：每一层只依赖其下层，不跨层调用，每一层通过接口对外提供服务。

VBL 整体三层架构：

```
UI 层 / 上层调用方（Filmora 主进程）
    ↓（通过 wrapper 层）
BusinessLayer（业务层）← 本节重点
    ↓
DataModel（数据模型层）
    ↓
Adapter（适配层）→ 底层引擎（WES/NLE）
```

**BusinessLayer 的职责**：
- 封装复杂业务逻辑（项目管理、编码导出、AI 处理）
- 协调 DataModel + BaseService + Adapter 三者协作
- 为 UI 层（通过 wrapper）提供高层 API
- 处理异步任务调度（后台任务、线程池）
- 管理 Undo/Redo 框架（与 BaseService::BsUndoManager 集成）

## 3. 模块间依赖关系

通过 CMakeLists.txt 中的 `target_link_libraries` 可以清晰看到依赖关系：

```
ProjectEditor（顶层协调者）

    ├── SourceManager     ← 资源管理

    ├── VisualEditor      ← 可视编辑
    ├── AIManager         ← AI 功能
    ├── DataModel         ← 数据模型（via Adapter）
    ├── BackgroundTaskManager ← 后台任务
    └── BsPluginManager   ← 插件管理

EncodeManager
    ├── DataModel         ← 获取要导出的 Timeline
    ├── BsEncodeManager   ← 底层编码实现
    ├── TimelineUX        ← Timeline 构建辅助
    ├── Adapter           ← 引擎适配
    └── BackgroundTaskManager

AIManager
    ├── DataModel
    ├── ProjectSerialize  ← AI 结果写入项目
    ├── BsEncodeManager   ← AI 预处理编码
    ├── BackgroundTaskManager ← 异步 AI 任务
    ├── BsPluginManager   ← AI 插件
    └── SourceManager

ProjectSerialize
    ├── DataModel
    ├── Adapter
    ├── BsUndoManager     ← 撤销记录
    ├── ListenerCenter    ← 事件通知
    └── SourceManager
```

**规律**：
- ProjectEditor 是最顶层的协调者，依赖最多模块
- DataModel 和 BackgroundTaskManager 是"被依赖最多"的基础服务
- 各子模块之间尽量通过 ListenerCenter（事件总线）间接通信，减少直接依赖

## 5. EncodeManager — Builder + Director 导出流程

EncodeManager 使用 Builder 模式构建用于编码的 Timeline：

```
UI 层发起导出
    ↓
EncodeManager::startExport(EncodeTaskParams)
    ↓
GenerateTimelineTask<TimelineBuilder>（模板，Builder 策略可替换）
    ↓
GenerateTimelineDirector::generateTimeline()
    ↓
TimelineDataBuilder::BuildBaseTimeline()    ← 构建基础 Timeline
    + BuildUnAuthResourceTimeline()         ← 处理未授权资源（水印替换）
    + BuildHideMotionTimeline()             ← 隐藏特定运镜
    + BuildVirtualClip()                    ← 插入虚拟片段
    + BuildWaterClipAndTailVideo()          ← 添加水印/尾帧
    ↓
IDmTimeline（导出用，与编辑用 Timeline 独立）
    ↓
BsEncodeManager::encode(timeline, profile)  ← 底层编码
```

```cpp
// modules/BusinessLayer/EncodeManager/base/GenerateTimelineData/GenerateTimelineTask.h
template<typename BuilderType>
class GenerateTimelineTask {
public:
    EncodeGenerateTimelineResult operator()() {
        BuilderType builder(m_TaskInfo);            // 创建具体 Builder
        GenerateTimelineDirector director(&builder); // Director 控制构建顺序
        SafePtr<IDmTimeline> pProjectTimeline(
            director.generateTimeline(this->m_TaskInfo)
        );
        auto result = std::make_tuple(director.getState(), pProjectTimeline);
        if (m_cb) { m_cb(result); }  // 通过回调返回结果（异步友好）
        return result;
    }
private:
    std::shared_ptr<EncodeTaskParamStruct> m_TaskInfo;
    std::function<void(EncodeGenerateTimelineResult)> m_cb;
};
```

**Builder 策略（可替换）**：

| Builder 类 | 场景 |
|------------|------|
| `TimelineBuilder` | 标准导出（完整 Timeline） |
| `ClipsTimelineBuilder` | 片段导出（只导出选中片段） |
| `TimelineDataNotCloneBuilder` | 不克隆 Timeline（特殊场景，性能优化） |

## 6. AIManager — 异步线程模型

AI 任务通过 `VbAIManagerThread` 管理异步线程：

```cpp
// modules/BusinessLayer/AIManager/VbAIManagerThread.hpp
class VbAIManagerThread {
public:
    // loopfunc 是业务逻辑，返回值含义：
    //   > 0: sleep N 毫秒后继续
    //   = 0: 立即继续
    //   < 0: 退出线程
    VbAIManagerThread(std::function<int(int)> loopfunc, int id);

    void start();           // 启动线程
    void quit(bool bWait);  // 请求退出（bWait=true 时等待线程结束）
    bool isRunning();

private:
    void thread_run();
    bool m_exit_thread = true;
    std::shared_ptr<std::thread> m_thread;  // RAII 线程管理
    std::function<int(int)> m_loopfunc;
};
```

```cpp
// modules/BusinessLayer/AIManager/VbAIManagerThread.cpp
void VbAIManagerThread::thread_run()
{
    VBL_LOG_INFO() << "ai start thread " << m_id;
    while (!m_exit_thread) {
        int time = m_loopfunc(m_id);   // 执行业务逻辑，获取等待时间
        if (time > 0) {
            // 等待 N 毫秒（轮询间隔，减少 CPU 占用）
            std::this_thread::sleep_for(std::chrono::milliseconds(time));
        }
        if (time < 0) {
            m_exit_thread = true;  // 业务逻辑要求退出
        }
    }
    VBL_LOG_INFO() << "ai exit thread " << m_id;
}
```

**使用示例**：

```cpp
// AI 语音转写任务
auto transcribeThread = std::make_unique<VbAIManagerThread>(
    [this](int id) -> int {
        if (m_transcribeQueue.empty()) return 50;  // 队列空：等 50ms
        auto task = m_transcribeQueue.front();
        auto result = callTranscribeAPI(task);      // 调用 AI API
        if (result.isDone()) {
            m_transcribeQueue.pop_front();
            postResultToUI(result);                 // 通过 EventBus 通知 UI
            return 0;                               // 立即继续
        }
        return 100;  // 未完成：100ms 后轮询
    },
    0  // thread id = 0
);
transcribeThread->start();
```

## 8. 面试要点

1. **BusinessLayer 是 VBL 的"服务层"，不直接操作数据，而是协调**：ProjectEditor 等高层模块不直接 new Clip 或调 Adapter，而是通过 DataModel 的工厂和接口，再配合 BaseService（Undo/日志/后台任务）完成操作，保持各层职责分离。

2. **GenerateTimelineTask 的模板参数是 Builder 策略**：`GenerateTimelineTask<TimelineBuilder>` vs `GenerateTimelineTask<ClipsTimelineBuilder>` 通过 C++ 模板在编译期选择 Builder 策略，零运行时开销。Director 控制构建步骤顺序，Builder 实现具体步骤，这是 Builder 模式的教科书实现。

3. **AIManager 的 loopfunc 设计用返回值控制频率**：返回值 >0 表示等待 N 毫秒、=0 立即继续、<0 退出。这个简洁的协议避免了复杂的条件变量/队列机制，适合 AI 任务这种"轮询结果"的异步场景（调用外部 AI API，等待结果返回）。

4. **子模块间通过 ListenerCenter 解耦**：ProjectEditor 完成项目加载后不直接调 TimelineEditor 刷新 UI，而是 `postMainThreadEvent("ProjectLoaded", data)`，所有订阅了这个事件的模块（TimelineUX/VisualEditor/SourceManager）都会在主线程收到通知并自行更新。

5. **每个子模块独立编译为库**：每个子目录是独立的 CMake target（通过 AppendModulesBasic 注册），可以按需 ON/OFF（`option(VBL_${subdir} "..." ON)`），方便裁剪：移动端可以关闭 AIManager、TemplateMatch 等大模块。

## 9. 可能被追问的问题

**Q1：ProjectEditor 依赖那么多子模块，是否违反了单一职责原则？**
> ProjectEditor 的职责是"项目生命周期协调者"，这本身是一个合理的职责。它依赖多个子模块，但通过接口（IVbSourceManager、IVbTimelineUX）而非具体类，是依赖倒置原则的正确应用。如果进一步拆分，可以按"加载流程"/"保存流程"/"恢复流程"划分，但对于当前规模的代码库，这会增加不必要的间接层。

**Q2：EncodeManager 为什么不直接用 IDmTimeline，而是要 Builder 重新构建一个 Timeline？**
> 编辑用的 Timeline 和导出用的 Timeline 需求不同：① 导出需要处理未授权资源（替换为水印）；② 需要插入片头/片尾/水印 Clip；③ 可能需要合并某些轨道或简化结构以提高编码性能；④ 多个任务并发导出时，每个任务需要独立的 Timeline 副本（不能共享，否则并发修改会互相干扰）。Builder 模式让这些"导出特有的变换"可以按步骤清晰组织。

**Q3：VbAIManagerThread 的轮询方式会不会有 CPU 占用问题？**
> `loopfunc` 返回 >0 时会 sleep N 毫秒（`this_thread::sleep_for`），CPU 在 sleep 期间完全让出（线程进入睡眠状态）。实际上大多数情况 AI 任务的返回值是 50-200ms 的等待，CPU 占用极低（几乎为零）。只有在有任务处理时才会密集运行，这是 AI 场景的合理权衡（比复杂的 condition_variable + 队列机制更易实现和调试）。

**Q4：ProjectSerialize 如何处理序列化后的向后兼容性（旧版本项目能被新版本打开）？**
> 通常通过版本号字段：项目 JSON 中包含 `version` 字段，反序列化时读取版本号，对旧版本格式执行升级逻辑（migration）。DataModel 层的接口设计（IDmClip/IDmEffect 等）注意向后兼容，新增字段采用有默认值的可选字段，不改变必填字段的含义。

**Q5：如果需要给一个全新的功能（如实时协作）添加到 BusinessLayer，应该怎么设计？**
> 新建一个子目录 `CollabManager`，通过 AppendModulesBasic 注册。依赖 ListenerCenter（订阅 Timeline 变更事件）、DataModel（获取当前状态快照）、BackgroundTaskManager（网络请求在后台线程）。与其他模块的交互通过 EventBus（发布"CollabChange"事件，UI 层订阅并更新）。这样新模块不需要修改任何现有模块，符合开闭原则。
