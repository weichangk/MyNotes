# 🎯 一、架构类问题（必问）

------

## ❓1. 为什么采用 MVC-ish，而不是标准 MVC？

### ✅ 标准回答

> 项目初期参考了经典 MVC 分层，但在实际开发中发现标准 MVC 存在一些问题，比如 Controller 容易承载过多业务逻辑，逐渐演变为“上帝类”。
>
> 因此我们在 MVC 基础上引入了 Service 层，将核心业务逻辑从 Controller 中剥离出来，使 Controller 仅负责 UI 事件编排和流程调度。
>
> 这种方式既保留了 MVC 的分层思想，又提升了系统的可维护性和可扩展性，因此我们采用了 MVC-ish 的工程化分层方案。

------

### 🎯 面试加分点

- 说出：**Controller 膨胀问题**
- 说出：**职责下沉到 Service**

------

## ❓2. MVC-ish 和 MVVM 的区别？为什么没用 MVVM？

### ✅ 标准回答

> MVVM 更适用于数据驱动的 UI 框架，例如 WPF 或前端框架，依赖数据绑定机制来实现 View 和 ViewModel 的自动同步。
>
> 但我们的项目基于 WinForms，其数据绑定能力较弱，如果强行引入 MVVM，会带来较高的实现成本和复杂度。
>
> 因此我们选择 MVC-ish 架构，通过 Service 层解耦业务逻辑，在保持开发效率的同时达到良好的分层效果。

------

### 🎯 加分点

👉 点出：

- WinForms 不适合 MVVM（很关键）
- 技术选型是“权衡”，不是“跟风”

------

## ❓3. Service 层具体做什么？和 Controller 怎么分工？

### ✅ 标准回答

> Service 层主要负责核心业务逻辑的实现，例如转码任务管理、参数处理、AI 调用调度等。
>
> Controller 主要负责 UI 层的事件响应和流程编排，例如接收用户操作、调用 Service 接口并更新 UI。
>
> 通过这种分层，我们避免了业务逻辑侵入 UI 层，也避免了 Controller 过度膨胀。

------

### 🎯 一句话总结（面试时可补）

> 👉 Controller 控流程，Service 管业务

# 🎯 二、底层引擎 / PInvoke（高频 + 高分）

------

## ❓4. 为什么用 P/Invoke？有没有考虑其他方案？

### ✅ 标准回答

> AgileTrans 转码引擎是以 C 接口形式提供的原生 DLL，因此我们采用 P/Invoke（DllImport）进行调用，这是 .NET 调用 native C 接口最直接且性能开销最小的方式。
>
> 相比 COM 或 C++/CLI，P/Invoke 更轻量，不依赖额外运行时，也更适合我们这种高频调用的性能敏感场景。

------

### 🎯 加分点

- 提到：
  - 性能
  - 轻量
  - 无额外依赖

------

## ❓5. P/Invoke 过程中遇到过什么问题？

👉 这个是**杀手级问题（决定深度）**

------

### ✅ 标准回答（可以挑你真实做过的说）

> 在 P/Invoke 过程中主要遇到过以下几个问题：
>
> 1. **参数封送问题（Marshalling）**
>     比如字符串编码（ANSI / UTF-8）和结构体对齐，需要和 C++ 侧保持一致，否则会导致数据异常。
> 2. **内存管理问题**
>     部分接口返回的是 native 分配的内存，需要明确释放策略，否则容易出现内存泄漏。
> 3. **异常隔离问题**
>     native 层异常无法直接传递到托管层，因此需要在接口层做错误码封装，统一转为托管异常。
> 4. **线程模型问题（如果你做过）**
>     某些接口需要在特定线程调用，我们通过调度层进行统一控制。

------

### 🎯 加分关键词

- struct layout
- GC / pinning
- 非托管内存释放

## ❓6. 本地 SDK + 云端 API 为什么要双路？

### ✅ 标准回答

> 主要是出于性能、效果和稳定性的综合考虑：
>
> - 本地 SDK：延迟低、无网络依赖，适合实时处理
> - 云端 API：模型能力更强，效果更好
>
> 因此我们通过策略模式封装两种能力，根据任务类型、设备性能以及网络情况动态选择执行路径，从而在性能和效果之间取得平衡。

------

------

## ❓7. 策略是怎么设计的？

### ✅ 标准回答（简单版）

> 我们抽象了一层统一的 AI 接口，例如 `IAIService`，分别实现本地和云端两种策略，然后通过策略选择器在运行时决定使用哪一种实现。
>
> 这样可以做到对上层透明，同时也方便后续扩展新的 AI 能力。

------

## ❓8. 80+ 子工程是怎么管理的？不会很乱吗？

### ✅ 标准回答

> 我们采用按功能域划分的模块化设计，例如转码、下载、AI、UI 组件等分别作为独立子系统，每个子系统内部再细分子模块。
>
> 同时通过接口抽象和依赖约束控制模块之间的依赖关系，避免循环依赖，从而保证整体结构清晰、可维护。

------

------

## ❓9. 模块化带来了什么好处？

### ✅ 标准回答

> 模块化设计主要带来三点好处：
>
> 1. 降低耦合，提高可维护性
> 2. 支持功能独立迭代
> 3. 提高团队协作效率（不同模块可并行开发）
