自动化测试（Automation Testing）简单说就是：**用代码或工具替代人工重复执行测试用例**，提升效率和稳定性。

------

## 一、自动化测试在做什么？

核心目标就三个：

- **回归测试自动化**（改代码后快速验证没坏）
- **重复测试自动化**（减少人工点点点）
- **持续集成（CI）配合**（每次提交自动跑测试）

------

## 二、常见类型

### 1. UI 自动化

模拟用户操作界面

- Web：Selenium / Playwright / Cypress
- App：Appium

👉 特点：覆盖真实用户流程，但**慢 + 易不稳定**

------

### 2. 接口自动化（最主流）

直接测 API

- Postman + Newman
- Python requests / Java RestAssured

👉 特点：**快、稳定、性价比最高**

------

### 3. 单元测试（开发写）

- C++：gtest
- Java：JUnit
- Python：pytest

👉 特点：粒度最小，覆盖代码逻辑

------

### 4. 性能/压力测试

- JMeter
- Locust
- LoadRunner

------

## 三、自动化测试的本质（面试常问）

一句话：

> 自动化测试 = 用代码验证代码

但关键不是“能不能自动化”，而是：

- 值不值得自动化（ROI）
- 稳不稳定（flaky test 问题）
- 是否易维护（用例结构设计）

------

## 四、常见面试追问点

如果你是面试准备，这些很高频：

### 1. 为什么接口测试比 UI 更适合自动化？

👉 更稳定、更快、更易定位问题

------

### 2. 如何设计自动化测试框架？

常见结构：

- 用例层（test cases）
- 业务层（API封装 / Page Object）
- 工具层（日志、配置、请求封装）
- 数据层（yaml / json / excel）

------

### 3. 什么是测试金字塔？

- 单元测试（最多）
- 接口测试（中间）
- UI测试（最少）

------

### 4. 自动化测试的痛点？

- 用例维护成本高
- UI频繁变更
- 测试数据依赖复杂
- 不稳定测试（flaky test）
