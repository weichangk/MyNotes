# 一、C++基础 & 对象模型（1-15）

**1. C++和C的区别？**
 C是面向过程，C++支持面向对象（封装/继承/多态），并引入STL、模板等。

**2. struct和class区别？**
 默认权限不同：struct public，class private。

**3. C++对象内存布局？**
 成员变量 + 对齐 + vptr（若有虚函数）。

**4. 空类大小是多少？为什么？**
 1字节，用于区分对象地址。

**5. this指针是什么？**
 指向当前对象的常量指针。

**6. static成员变量特点？**
 类共享，存储在全局区。

**7. static成员函数特点？**
 没有this指针，只能访问static成员。

**8. extern作用？**
 声明外部变量/函数。

**9. inline函数作用？**
 减少函数调用开销，建议编译器内联展开。

**10. 宏和inline区别？**
 宏是文本替换，inline有类型检查。

**11. 引用和指针区别？**
 引用必须初始化且不可为空，指针可变。

**12. const作用？**
 修饰变量不可修改，修饰函数保证不改成员。

**13. volatile作用？**
 防止编译器优化，常用于硬件/多线程。

volatile 的作用是告诉编译器该变量可能被外部因素改变，从而禁止对其进行优化，每次都必须从内存读取，但它**不提供线程安全和原子性保障**。

**14. mutable作用？**
 允许const对象修改该成员。

**15. 对齐规则？**
 按最大成员对齐。

# 三、内存管理（31-45）

**31. new/delete做了什么？**
 new分配内存+构造对象。

`new / delete` 是 C++ 里用于**动态内存分配与释放**的一对操作符，本质上它们不只是“分配内存”，而是做了一整套对象生命周期管理。

new = 分配内存 + 调用构造函数

delete = 调用析构函数 + 释放内存

new 在堆上分配内存并调用构造函数，delete 调用析构函数并释放内存；底层分别对应 operator new 和 operator delete。



**32. malloc/free和new/delete区别？**
 是否调用构造析构。

malloc/free 只是“内存管理”，申请/释放原始内存，不会调用构造函数

new/delete 是“对象级别操作”， 本质：new = malloc + 构造函数调用，delete = 析构函数 + free

为什么 C++ 还要 malloc？C 兼容，底层系统调用 / 内存池 / allocator 实现

一句话总结

> malloc/free 只负责“内存”，不关心对象；
>  new/delete 负责“内存 + 对象生命周期（构造/析构）”，是 C++ 面向对象语义的一部分



**33. operator new和new区别？**
 operator new只分配内存。

`operator new` 本质是一个**内存分配函数**

`new` 是 **C++ 关键字 / 表达式**，它做的是一整套流程：分配内存 + 构造对象



**34. delete this安全吗？**
 必须确保对象是new创建。

**35. 内存泄漏是什么？**
 分配未释放。

**内存泄漏（Memory Leak）\**指的是：程序在运行过程中\**动态分配了内存，但在不再使用后没有正确释放**，导致这部分内存无法被再次使用，并且随着程序运行持续累积。

如何检测内存泄漏？Visual Studio Diagnostics Tools

在 Visual Studio Diagnostics Tools 中，通过多次拍摄 Memory Snapshot，并进行对比分析，观察对象数量和大小是否持续增长，并结合 Retention Graph 查看引用链，可以定位内存泄漏的对象及其未释放原因。



**36. 野指针是什么？**
 指向非法内存。

指向“未知或无效内存地址”的指针。换句话说，就是这个指针“还在用”，但它指向的内存已经**不再属于合法、可控范围**了。

野指针是怎么产生的？1）未初始化的指针，2）释放后继续使用，3）指向局部变量的地址

如何避免野指针？1）初始化指针，delete 后置空，3）避免返回局部变量地址，4）使用智能指针



**37. 智能指针有哪些？**
 unique_ptr/shared_ptr/weak_ptr。

**38. shared_ptr原理？**
 引用计数控制块。

本质：**引用计数型智能指针（Reference Counting Smart Pointer）**，用于实现“多个指针共享同一块堆内存，并在最后一个引用释放时自动析构对象”。

一句话： shared_ptr = 指针 + 控制块（引用计数 + 管理信息）。它通常由两部分组成：1）指向对象的裸指针（T*）2）控制块（Control Block）

控制块结构

标准库实现类似：

```
struct ControlBlock {
    std::atomic<long> use_count;
    std::atomic<long> weak_count;
    void* ptr;
    Deleter deleter;
};
```

关键点：

> **use_count 是原子变量（线程安全）**

shared_ptr 的关键特性：1.线程安全（注意是“计数安全”，不是对象安全），2.拷贝是“浅拷贝”，3.删除时机：最后一个引用释放 use_count == 0 -> delete object

shared_ptr 的核心问题？ 1.循环引用，解决方案：weak_ptr，2.误用 this

shared_ptr 的代价？1.性能成本：原子操作（引用计数 ++ / --）2.内存成本：控制块额外占用



**39. weak_ptr作用？**
 解决循环引用。

weak_ptr 用来“观察 shared_ptr 管理的对象”，但不增加引用计数，从而打破循环引用并避免内存泄漏。

weak_ptr 的核心能力：1）观察对象是否还存在，2）安全访问对象（lock），3）检测生命周期

weak_ptr 的内部原理：只持有 control block 的 weak_count，不增加 use_count

weak_ptr 使用场景：1）打破循环引用，2）缓存设计，3）观察者模式（Observer），4）线程安全资源访问



**40. 循环引用问题？**
 shared_ptr互相引用导致无法释放。



**41. unique_ptr特点？**
 独占所有权。

`std::unique_ptr` 是 C++11 引入的**独占式智能指针**，核心目标是：**用最小开销实现 RAII 自动资源管理，并保证“唯一所有权”**。

核心特点：独占所有权，同一时刻只能有一个 `unique_ptr` 拥有某个对象，不能拷贝，只能**移动（move）**；轻量级（几乎无额外开销）；自动释放资源（RAII）

`unique_ptr` 本质是：“一个不可拷贝、可移动的 RAII 包装器 + 一个原始指针 + 可选删除器”

unique_ptr 是 C++ 的独占式智能指针，只允许单一所有权，禁止拷贝支持移动语义，利用 RAII 在作用域结束时自动释放资源，零引用计数开销，适用于明确单一生命周期的资源管理场景。



**42. RAII是什么？**
 资源绑定对象生命周期。

把资源的生命周期绑定到对象的生命周期上：对象创建时获取资源，对象销毁时自动释放资源。

RAII 的做法

把资源封装到对象里：

```
class MyInt {
public:
    MyInt() {
        p = new int(10);  // 获取资源
    }

    ~MyInt() {
        delete p;         // 释放资源
    }

private:
    int* p;
};
```

使用时：

```
void func() {
    MyInt obj;  // 进入作用域自动获取资源
}              // 离开作用域自动释放资源
```

👉 关键点：

- **构造函数 = 获取资源**
- **析构函数 = 释放资源**

现代 C++ 的 RAII（重点）：RAII 最典型的实现就是智能指针

std::unique_ptr

std::lock_guard

RAII 的核心价值（面试重点）：1.自动释放资源，2.异常安全，3.代码更清晰，4.避免资源泄漏

RAII 是 C++ 的资源管理思想，即资源的生命周期绑定对象生命周期。通过在构造函数中获取资源，在析构函数中释放资源，实现自动化资源管理，从而避免内存泄漏和异常安全问题。智能指针和 lock_guard 都是 RAII 的典型实现。



**43. 栈 vs 堆？**
 栈自动管理，堆需手动释放。

栈是一种由**编译器自动管理**的内存区域，遵循 **后进先出（LIFO）**。

特点：自动分配 / 自动释放；由编译器管理（函数调用时创建，返回时销毁）；速度非常快（只移动栈指针）；空间相对较小（一般 MB 级）；生命周期严格跟随作用域



堆是由**程序员手动管理**或智能指针管理的内存区域。

特点：手动分配 / 手动释放；使用 `new/delete` 或 `malloc/free`；生命周期由你控制；空间大（可用系统内存）；分配/释放较慢（涉及系统调用 + 内存管理算法）；容易产生内存泄漏



面试一句话：栈是编译器自动管理的快速局部内存，生命周期随作用域结束；堆是由程序员控制的动态内存，用于灵活分配，但需要手动管理，容易产生泄漏和碎片。



**44. 内存对齐原因？**
 提高CPU访问效率。

现代 CPU 读取内存不是按“字节”读，而是按 **字长（word）或缓存行（cache line）**读取，比如：

- 32位 CPU：常见按 4 字节
- 64位 CPU：常见按 8 字节或 16 字节
- Cache line：通常 64 字节

编译器为什么要“强制对齐”？提高访问效率；符合 CPU ABI 约定

总结一句话：为了让 CPU 能以最少的内存访问次数，最高效地按机器字长访问数据，同时避免跨 cache line 或跨总线访问带来的性能损耗甚至硬件异常。



**45. placement new是什么？**
 在指定内存构造对象。

# 四、STL（46-60）

**46. vector底层？**
 动态数组。

连续内存的动态数组（dynamic array），由 allocator 管理的一块可扩容的堆内存。

1. 内存结构（核心）

`vector` 内部通常维护三个指针（或等价结构）：

- `start`：指向已分配内存的起始
- `finish`：指向当前最后一个元素之后
- `end_of_storage`：指向整块容量的末尾

可以理解为：

```
[start ........ finish ........ end_of_storage]
 |---- size ----|------ capacity ------|
```

含义：

- `size = finish - start`（已有元素数量）
- `capacity = end_of_storage - start`（总容量）

2. 为什么说是“连续数组”

`vector` 的元素在内存中是**连续存储**的：

```
std::vector<int> v = {1,2,3};
```

内存类似：

```
[1][2][3][ ][ ][ ]
```

👉 这也是它支持：

- O(1) 随机访问 `v[i]`
- CPU cache 友好（性能好）

3. 扩容机制（重点面试点）

当 `size == capacity` 时，再 `push_back` 会触发：

### 步骤：

1. 申请一块更大的新内存（通常 **2倍扩容** 或 1.5倍）
2. 将旧数据 **拷贝/移动** 到新内存
3. 释放旧内存
4. 更新指针

```
旧： [1 2 3]
新： [1 2 3 _ _ _ _ _]
```

4. 扩容代价（为什么 push_back 有时慢）

单次 `push_back`：

- 平均 O(1)
- 但扩容时是 O(n)

👉 所以是 **均摊 O(1)**（amortized O(1)）

5. allocator（内存管理）

`vector` 不直接 new/delete，而是通过：

```
std::allocator<T>
```

作用：

- 分配原始内存（allocate）
- 构造对象（construct）
- 销毁对象（destroy）

👉 分离“内存”和“对象生命周期”



6. iterator 失效问题（高频面试）

由于扩容会整体搬家：



面试一句话总结：

vector 底层是基于连续内存的动态数组，通过 capacity 管理扩容，当空间不足时重新申请更大内存并搬移数据，因此支持高效随机访问，但在扩容和中间插入删除时存在性能开销，同时会导致迭代器失效。



### 会失效：

- 所有 `iterator`
- 所有 `pointer`
- 所有 `reference`

```
auto it = v.begin();
v.push_back(100); // 可能触发扩容
// it 失效
```

`allocator`（分配器）本质上是：

> **C++ 用来“管理内存分配与对象构造/销毁”的一层抽象接口。**

它是 STL 容器（`vector / list / map`）背后真正做内存工作的“底层工具”。



**47. vector扩容机制？**

**49. deque底层？**
 分段连续数组。

deque = 分段连续内存 + 中央控制表（map）

它不是一整块连续内存，而是：

- 多个“小连续数组”（buffer / block）
- 一个“指针数组”（map）管理这些块

为什么要这样设计？

1. 解决 vector 的问题

vector：

- ❌ 扩容要整体搬家（realloc）
- ❌ 头部插入 O(n)

deque：

- ✔ 头尾插入 O(1)
- ✔ 不需要整体搬迁

2. 为什么不用链表？

链表：

- ❌ cache 不友好（指针跳转）
- ❌ 内存碎片严重

deque：

- ✔ 每块是连续内存（cache 友好）
- ✔ 又能动态扩展

总结一句面试答案: deque 底层是由多个固定大小的连续内存块组成，并通过一个指针数组（map）进行管理，从而实现了头尾插入 O(1) 和随机访问 O(1)，但整体并不是连续内存。

# 五、C++11/14/17（61-75）

**61. auto是什么？**
 类型推导。

`auto` 是 C++11 引入的**类型自动推导关键字**，用于让编译器根据初始化表达式**自动推断变量类型**。

`auto` 在 C++ 里**不是运行时机制**，也不是一个“真实存在的类型”，它本质上是**编译期的类型推导占位符（type deduction placeholder）**。

换句话说：

> `auto` = 编译器帮你把“变量类型”用模板推导规则算出来，然后替换掉。



**62. decltype作用？**
 获取表达式类型。

`decltype` 是 C++11 引入的一个**类型推导关键字**，作用是：**在编译期获取表达式的类型**。

它和 `auto` 类似，但本质区别是：

- `auto`：通过初始化值“推导变量类型”
- `decltype`：直接“推导表达式的类型”（不一定要初始化）



**63. lambda是什么？**
 匿名函数对象。

在 C++ 里，**lambda（匿名函数）** 是一种“可以在代码中就地定义的临时函数对象”，本质上是编译器帮你生成的一个**匿名仿函数（functor）类**。

捕获外部变量：值捕获 `[=]`；引用捕获 `[&]`；指定捕获；

mutable 关键字：默认值捕获是只读的：加 mutable 才能改副本：

lambda 本质

编译器会把它转成类似这样：

```
class __Lambda {
public:
    int operator()(int a, int b) const {
        return a + b;
    }
};
```

👉 所以 lambda 本质是：

> **一个重载了 operator() 的匿名类对象**



面试高频总结

1️⃣ lambda 是什么？

👉 匿名函数对象（函数 + 状态）

2️⃣ 和函数指针区别？

- lambda 可以捕获变量
- 函数指针不能

3️⃣ capture 有哪几种？

- `[=]` 值捕获
- `[&]` 引用捕获
- `[a, &b]` 指定捕获

4️⃣ 本质是什么？

👉 编译器生成的 functor 类

一句话总结

👉 **lambda 是 C++ 中“可以携带上下文的匿名函数对象”，本质是语法糖 + 函数对象封装。**



**64. 捕获方式？**

**66. std::move做了什么？**

它到底“做了什么”

✔️ 1. 类型转换（唯一真正做的事）

把：

- 左值（lvalue）
- → 转换为右值引用（rvalue reference）

很多人误解：

三、真正“移动”的发生在哪里？

👉 **移动构造函数 / 移动赋值运算符**


实际调用的是：

```
string(string&& other); // move constructor
```

在这个函数里才会：

- 指针直接接管
- 原对象置空



常见面试总结答案（可以直接背）

**80. 死锁是什么？**

死锁（Deadlock）本质上是：

最终程序会“卡死”。

# 死锁的四个必要条件（面试高频）

操作系统经典理论：

1.互斥条件

资源一次只能一个线程使用。

例如 mutex。

2.请求与保持条件

线程已经拿到部分资源，
同时继续请求新资源。

例如：

```
thread1:
已经持有 A
继续申请 B
```

3.不可剥夺条件

资源不能强制抢走。

mutex 不能被别的线程强行释放。

4. 环路等待条件

形成循环等待链：

```
T1 等 T2
T2 等 T1
```

或者：

```
T1 -> T2 -> T3 -> T1
```

2️⃣ 编译（Compilation）

👉 输入：`.i`
 👉 输出：`.s`（汇编文件）

这一阶段做：

- 词法分析（token）
- 语法分析（AST）
- 语义分析（类型检查）
- 优化（部分编译器会做）
- 生成汇编代码

📌 本质：**把 C++ 变成汇编语言**

3️⃣ 汇编（Assembly）

👉 输入：`.s`
 👉 输出：`.o / .obj`（目标文件）

做的事情：

- 将汇编指令翻译成机器码
- 生成二进制目标文件（但还不能运行）

📌 本质：**变成“机器能读的半成品”**

4️⃣ 链接（Linking）

👉 输入：`.o + 库文件`
 👉 输出：可执行文件（如 `.exe / a.out`）

做的事情：

- 合并多个目标文件
- 解析函数调用（符号解析）
- 静态库/动态库链接
- 地址重定位

📌 本质：**把所有零件拼成一个完整程序**

🧠 一句话总结（面试常用）

> 预处理：展开代码
>  编译：翻译成汇编
>  汇编：生成目标机器码
>  链接：拼成可执行程序

**92. 静态库 vs 动态库？**
 编译进程序 vs 运行加载。

静态库是在**编译/链接阶段直接拷贝进可执行文件**的库。

优点：运行时不依赖库文件；运行速度略快

缺点：可执行文件变大；更新麻烦；内存浪费



动态库是在**运行时加载/链接的库**

## 工作方式

```
编译 → 生成可执行文件（只记录依赖）
运行 → 操作系统加载动态库
```

也就是：

> 程序运行时才“去找库”

优点：可执行文件更小；多程序共享一份库；库可以独立更新；插件化设计

缺点：运行依赖库文件；运行时有开销（动态加载，符号解析，重定位）；版本地狱问题（libA 依赖 libB v1；libC 依赖 libB v2）



动态库的“版本地狱”（dependency hell）本质是：**多个库依赖同一个库的不同版本，但系统只能加载一个版本或发生冲突**



动态库“版本地狱”解决方案：ABI稳定性 + 运行时路径隔离（rpath）

1.ABI 版本控制（最核心原则）

动态库解决版本问题的核心不是“技术”，而是：

> ❗ ABI 不能乱改

ABI 规则：

- 可以改（兼容）：
  - 新增函数
  - 新增结构体字段（在尾部）
- 不能改（破坏）：
  - 函数签名
  - 结构体布局
  - 删除函数



2.私有依赖隔离

每个程序绑定自己的库路径

> 栈溢出 = “函数调用层数太深 or 栈上数据太大，超过系统栈容量”



**98. ABI是什么？**
 二进制接口规范。

ABI 是编译后程序之间的“二进制契约”，规定函数调用方式、数据布局、符号规则等，保证不同模块/库在运行时可以正确互操作。



**99. 编译优化等级？**
 -O0 -O2 -O3。

1. `-O0`（默认，无优化）

特点：

- 不做优化
- 保留完整调试信息
- 代码结构与源码几乎一一对应

用途：

- 调试（debug）
- 断点、单步执行最准确

缺点：

- 性能最差
- 代码冗余多

2. `-O1`（基础优化）

特点：

- 做轻量优化
- 去掉部分无用代码
- 简单常量传播、死代码消除

目标：

- 在编译速度和运行速度之间折中

3. `-O2`（推荐生产级优化 ⭐）

特点：

- 开启大多数“安全优化”
- 不会显著增加代码体积
- 包括：
  - 循环优化
  - 指令重排
  - 内联（部分）
  - 寄存器优化
  - 常量传播增强

用途：

- 默认生产编译级别（Linux 系统、服务端常用）

👉 一句话：

> **性能/稳定性/编译时间最平衡的等级**

4. `-O3`（激进优化）

特点：

- 在 `-O2` 基础上更激进
- 强化：
  - 循环展开（loop unrolling）
  - 向量化（SIMD）
  - 更积极内联

优点：

- 可能更快

缺点：

- 代码体积变大
- 某些场景反而变慢（instruction cache miss）
- 调试更困难

5. `-Os`（优化体积）

特点：

- 类似 `-O2`
- 但优先减少**二进制大小**

适用：

- 嵌入式系统
- 手机 / IoT
- 对缓存敏感系统

6. `-Ofast`（极限性能模式 ⚠️）

特点：

- 包含 `-O3`
- 额外关闭部分严格标准：
  - 可能忽略 IEEE 浮点标准
  - 允许数学运算重排

优点：

- 极致性能

缺点：

- 数值结果可能不完全严格一致（有风险）



面试高频总结一句话

👉 `-O0` 用于调试
 👉 `-O2` 用于生产
 👉 `-O3` 用于极限性能
 👉 `-Ofast` 用于“可以接受不完全标准一致的极致性能”





**100. gdb作用？**
 调试工具。

GDB 是一个调试器，用于控制程序执行、设置断点、单步运行、查看变量和调用栈，从而定位程序逻辑错误和崩溃原因。
