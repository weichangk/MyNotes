# 项目背景

我在 Filmora 里负责 Timeline 时间线核心编辑区域开发。

Timeline 是视频编辑器最核心、也是最复杂的交互模块之一。

基于：

- Qt Graphics View
- 自定义轨道系统
- 多轨视频/音频/特效编辑

用户在：

- 拖拽
- 缩放
- 裁剪
- 移动素材

时，对流畅度要求非常高。

------

# 问题

当时间线素材数量增多后：

会出现：

- 拖动卡顿
- 缩放掉帧
- UI 不流畅
- 主线程占用过高

尤其复杂工程下：

Graphics Scene 中存在大量 Item。

最开始很多操作会触发：

- 全量刷新
- 大量 repaint
- 频繁数据同步

导致性能明显下降。

------

# 分析过程

我先通过：

- Qt Profiler
- 自定义埋点
- FPS统计

定位问题。

发现主要瓶颈有几个：

## 1. 全量刷新

Graphics View 默认会导致大量区域重绘。

------

## 2. 拖拽过程中频繁更新真实业务数据

每次 mouse move：

都会同步到底层模型。

导致：

- UI线程压力大
- 业务层频繁计算

------

## 3. 非可视区域 Item 也参与渲染

大量无意义绘制。

------

# 解决方案

我后面做了几块优化。

------

## 第一块：局部刷新 + 可视区域裁剪

基于 visibleRect：

只更新当前可视区域。

并对不可见 Item：

- 不参与 paint
- 不参与布局

降低绘制压力。

------

## 第二块：引入 Substitute 拖拽代理机制

这是我认为最核心的优化。

拖拽过程中：

不直接操作真实业务对象。

而是：

创建轻量级 Substitute Item。

拖动时：

只更新代理对象。

真正提交时：

再一次性同步业务层。

这样避免：

- 高频业务更新
- 频繁信号广播
- 大量 undo/redo 记录

主线程压力明显下降。

------

## 第三块：缓存机制

对部分静态图元：

增加缓存。

避免重复生成。

------

# 结果

优化后：

复杂时间线场景下：

- 拖拽明显更流畅
- FPS 提升
- 主线程占用下降
- 用户反馈明显改善

同时：

整体架构解耦性也更好。

1.添加UI性能埋点去拿日志分析耗时

	1.使用 QTimer 做监控时钟，QTimer是基于事件循环，UI 线程执行

## 1. 统一任务模型

支持：

- 串行
- 并行
- 独立线程

------

## 2. 优先级队列

缩略图优先加载可视区域。

# ③ 线上崩溃案例（这个特别重要）

业务负责人很喜欢听：

# “真实线上问题”

因为这代表成熟度。

------

# 推荐案例

## QObject 生命周期 + 异步回调崩溃

------

# 背景

某次线上：

用户反馈：

打开媒体资源后快速切换页面。

偶现崩溃。

问题难复现。

------

# 分析

通过：

- Crash Dump
- 调用栈
- 日志

发现：

崩溃发生在：

异步缩略图加载完成后。

回调 UI刷新阶段。

------

# 根因

页面关闭后：

UI对象已经析构。

但后台任务仍在执行。

完成后：

异步回调继续访问已释放 QObject。

导致：

# 野指针崩溃。

------

# 解决方案

## 1. 引入生命周期绑定

异步任务：

绑定页面 Context。

页面销毁：

自动 Cancel。

------

## 2. 弱引用保护

回调前：

检查对象有效性。

------

## 3. 统一线程切换

禁止后台线程直接更新 UI。

统一切回主线程。

------

# 结果

问题彻底解决。

同时：

后续异步模块统一接入框架。

避免类似问题重复发生。
